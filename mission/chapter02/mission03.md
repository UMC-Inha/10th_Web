1. useReducer

   -사용 시점
   1. 복잡한 상태 구조로 여러 하위 값들을 관리해야 할 때
   2. (핵심)새로운 상태 값이 이전 상태를 기반으로 결정될 때 => 이전 상태에 의존적인 복잡한 상태 변경은 setState의 개별적인 업데이트만으로 안전하게 처리하게 힘듦

   -useReducer 미사용 시 생길 수 있는 문제점
   1. 상태 간 동기화 지옥 -한 번의 사용자 액션이 여러 개의 상태 업데이트 일으킴 -상태 업데이트 순서가 잘못되면 UI가 깨짐 -상태들이 서로 강하게 얽혀있어, 하나를 빠뜨리면 버그가 생김 -왜 이러한 값이 생기는 지에 대한 질문에 답하려면 모든 이벤트 핸들러 코드를 찾아봐야 함
   2. 일관성 없는 상태 관리 -상태 업데이트 순서와 처리 방식이 다른데, 이러한 차이가 버그를 만들어 냄
   3. 복잡한 규칙 관리의 어려움 -빠른 금액 버튼 처리의 복잡함

   -해결법: dispatch로 "무엇을 할지"만 전달하고, 상태 변경의 "어떻게"는 한 곳에서 관리한다. -예: const [state, dispatch] = useReducer(numberRadReducer, initialState);

   -useReducer로 얻은 이점
   1. 예측 가능한 상태 변경
      -useState: 여러 곳의 함수를 뒤져야 원인 찾음
      -useReducer: 리듀서에서 해당 action만 보면 됨
   2. 완벽한 타입 안정성
   3. 테스트 용이성: Reducer는 순수 함수이므로 React 환경 없이도 테스트 가능하며, 모든 상태 변경이 한 곳에서 일어나므로 검증 간단
   4. 디버깅과 유지보수성 -명확한 액션 흐름: 어떤 액션이 언제 발생했는지 추적 가능 -상태 변경 이력: 이전 상태로 되돌리기 쉬움 -미들웨어 패턴: Redux처럼 로깅, 에러 리포팅 확장 가능

   -언제 useReducer 선택해야 할까?
   1. 판단 기준
      -useState 사용해도 충분한 경우
      (i) 독립적인 단순 상태
      (ii) 상태 간 연관성이 없거나 적은 경우
      (iii) 비즈니스 로직이 단순한 경우
      -useReducer를 고려해야 하는 경우
      (i) 하나의 액션이 여러 상태를 동시에 변경해야 할 때
      (ii) 상태 변경 로직이 복잡하고 조건이 많을 때
      (iii) 상태가 서로 긴밀하게 연결되어 있을 때
      (iv) 테스트 코드 작성이 중요한 경우
