    | **구분** | **useState** | **useReducer** |
    | --- | --- | --- |
    | **상태 구조** | 기본 타입(문자열, 숫자)이나 단순한 객체 | 복잡한 객체나 여러 층으로 이루어진 배열 |
    | **로직 위치** | 컴포넌트 내부 (이벤트 핸들러 안) | 컴포넌트 외부 (Reducer 함수 안) |
    | **가독성** | 상태가 적을 때 매우 직관적임 | 상태가 많고 로직이 복잡할 때 더 깔끔함 |
    | **테스트** | UI와 결합되어 있어 테스트가 까다로움 | 순수 함수인 Reducer만 따로 테스트 가능 |
    | **추천 상황** | 독립적인 1~2개의 상태 관리 | 여러 상태가 서로 의존적이거나 복잡할 때 |
    
    1️⃣ useState 사용
    
    `useState`는 상태 업데이트 로직이 간단하고, 컴포넌트 내에서 직관적으로 코드를 읽고 싶을 때 사용한다.
    
    - **장점:** 코드가 짧고 이해하기 쉽다.  초기 개발 속도가 빠르다.
    - **단점:** 상태 업데이트 로직이 여러 곳에 흩어지면, 나중에 추적하기 힘들다.
    
    2️⃣useReducer 사용
    
    `useReducer`는 Action과 Reducer를 분리한다.
    
    - **장점:**
        1. **관심사 분리:** 상태 업데이트 로직이 컴포넌트 밖으로 나가서 UI 코드가 깨끗해진다.
        2. **안정성:** 이전 상태를 기반으로 다음 상태를 만들 때 로직을 한곳에서 관리하므로 버그가 줄어든다
        3. **재사용성:** 같은 Reducer 로직을 다른 컴포넌트에서도 재사용할 수 있다.
