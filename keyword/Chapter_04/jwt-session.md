# 세션 방식과 토큰 방식의 가장 큰 차이
가장 큰 차이는 인증 상태를 어디에 저장하느냐

## 세션 방식

- 서버가 로그인 상태를 저장함
- 클라이언트는 보통 세션 ID만 가지고 있음
- 요청이 오면 서버가 세션 저장소를 조회해 사용자를 식별함

## 토큰 방식

- 서버가 별도의 로그인 상태를 저장하지 않아도 됨
- 클라이언트가 토큰 자체를 저장하고 요청마다 함께 보냄
- 서버는 토큰의 서명과 내용을 검증해 사용자를 식별함

즉, 세션은 서버 중심 / 토큰은 클라이언트가 인증 정보를 들고 다니는 방식

# 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징
## Sticky Session

한 사용자의 요청을 항상 같은 서버로 보내는 방식

### 특징

- 구현이 비교적 단순함
- 각 서버가 자기 세션만 관리하면 됨
- 추가 저장소 없이도 동작 가능

### 단점

- 특정 서버에 부하가 몰릴 수 있음
- 사용자가 붙어 있던 서버에 장애가 생기면 세션이 사라질 수 있음
- 서버 확장성과 장애 대응 측면에서 한계가 있음

## Session Replication

각 서버가 세션 정보를 서로 복제하는 방식

### 특징

- 어느 서버로 요청이 가더라도 세션을 확인할 수 있음
- Sticky Session보다 유연함
- 장애가 발생해도 다른 서버에 세션 정보가 남아 있을 수 있음

### 단점

- 서버 간 세션 동기화 비용이 발생함세셔
- 서버 수가 많아질수록 네트워크 부담과 복잡도가 커짐
- 성능 저하 가능성이 있음

## Centralized Session Store

Redis 같은 중앙 저장소에 세션을 모아서 관리하는 방식

### 특징

- 모든 서버가 공통 저장소를 조회하므로 분산 환경에 적합함
- 서버가 늘어나도 세션 관리가 비교적 일관됨
- 현대적인 분산 시스템에서 많이 사용됨

### 단점

- 중앙 저장소에 장애가 생기면 전체 인증 처리에 영향을 줄 수 있음
- 저장소 자체의 고가용성 구성이 필요함
- 별도의 인프라 관리가 필요함

# JWT(Json Web Token)의 장점과 단점
## 장점

- 서버가 상태를 별도로 저장하지 않아도 됨
- 토큰만 검증하면 되므로 확장성이 좋음
- 여러 서버가 있는 환경이나 MSA 환경에 유리함
- 토큰 안에 사용자 ID, 권한, 만료 시간 등의 정보를 담을 수 있음
- 서버 간 인증 정보를 공유하기 쉬워짐

## 단점

- 한 번 발급된 토큰은 만료 전까지 유효함
- 따라서 즉시 로그아웃, 강제 무효화, 권한 회수가 어려움
- 토큰 길이가 세션 ID보다 길어질 수 있어 네트워크 부담이 생길 수 있음
- 토큰이 탈취되면 만료 전까지 악용될 가능성이 있음
- 민감한 정보를 토큰 안에 직접 넣는 것은 주의해야 함

# JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법
## 즉시 무효화 문제의 원인

JWT는 기본적으로 stateless 방식임

즉, 서버가 토큰의 상태를 별도로 저장하지 않고, 요청이 들어오면 서명과 만료 시간만 검증해서 유효성을 판단함

그래서 이미 발급된 JWT는 서버 입장에서 따로 추적하지 않으면 로그아웃을 하더라도 만료 시점 전까지 계속 유효하게 됨

## 해결 방법

### 1. 블랙리스트 방식

- 로그아웃된 토큰이나 특정 JWT ID를 저장해둠
- 요청이 올 때마다 블랙리스트에 있는지 확인함
- 즉시 무효화가 가능하지만 결국 서버 저장소 조회가 필요해짐

### 2. 짧은 만료 시간 사용

- Access Token의 유효 시간을 짧게 잡음
- 탈취되더라도 악용 가능한 시간을 줄일 수 있음
- 대신 사용자는 더 자주 재발급을 받아야 함

### 3. Refresh Token과 함께 운영

- Access Token은 짧게, Refresh Token은 상대적으로 길게 둠
- Access Token이 만료되면 Refresh Token으로 재발급받음
- 보안성과 사용자 편의성을 어느 정도 균형 있게 가져갈 수 있음

### 4. 저장소 기반 메타데이터 검증 추가

- 토큰 자체만 믿지 않고, 별도 저장소에 토큰 상태를 함께 관리함
- 필요 시 특정 토큰을 무효화시킬 수 있음
- 하이브리드 방식

# 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리
JWT의 확장성과 세션 방식의 제어 가능성을 함께 가져가려는 방식

## 동작 원리

1. 사용자가 로그인하면 서버가 JWT를 발급
2. 이때 JWT 안에 `jti` 같은 고유 식별자를 넣음
3. 서버는 이 토큰의 상태나 메타데이터를 Redis 같은 저장소에 함께 저장할 수 있음
4. 사용자가 요청을 보내면
    - 먼저 JWT의 서명을 검증하고
    - 저장소에서 해당 토큰이 유효한지 / 블랙리스트인지 / 폐기되었는지를 확인
5. 로그아웃 시에는 해당 `jti` 또는 토큰 정보를 저장소에 기록해 즉시 차단함

## 핵심 의미

- JWT만 사용하면 무효화가 어려움
- 세션만 사용하면 서버 확장 시 관리 부담이 커짐
- 그래서 하이브리드 방식은
    - 인증은 JWT로 처리하고
    - 통제는 저장소 메타데이터로 보완하는 구조
