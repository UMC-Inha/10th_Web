#  CSRF
    
    ## CSRF란?
    
    CSRF(Cross-Site Request Forgery)는 로그인된 사용자의 권한을 악용해서 사용자가 원하지 않는 요청을 보내게 만드는 공격
    
    핵심은 피해자가 정상 사이트에 로그인해 받아둔 세션 쿠키가 브라우저에 저장되어 있고, 공격자 사이트가 정상 사이트로 요청을 유도하면 브라우저가 그 쿠키를 자동으로 함께 전송하기 때문에 서버는 이 요청을 “로그인한 사용자의 정상 요청”으로 오해할 수 있음
    
    예시
    
    1. 피해자가 은행 같은 정상 사이트에 로그인한다.
    2. 공격자 사이트를 방문한다.
    3. 공격 페이지의 `img` 태그나 자동 제출 폼이 정상 사이트로 요청을 보낸다.
    4. 브라우저가 세션 쿠키를 자동 첨부한다.
    5. 서버는 인증된 요청으로 착각해 송금, 비밀번호 변경 같은 작업을 수행할 수 있다.
# CSRF 방어 전략
    
    ## `SameSite` 쿠키 사용
    
    쿠키에 `SameSite` 옵션을 설정하는 것
    
    `SameSite` 는 쿠키가 어떤 상황에서 전송될지 제한하는 속성이라서 크로스 사이트 요청에서 세션 쿠키가 자동으로 붙는 문제를 줄일 수 있음
    
    특히 `SameSite=Strict`를 쓰면 같은 사이트 요청에만 쿠키가 붙어서 공격자 사이트에서 정상 사이트로 보낸 요청에는 세션 쿠키가 실리지 않게 됨
    
    - `Strict` : 같은 사이트에서만 쿠키 전송
    - `Lax` : 일부 크로스 사이트 요청 허용
    - `None` : 모든 요청에 쿠키 전송, 대신 `Secure` 필요
    
    ## CSRF 토큰 사용
    
    `SameSite` 토큰 사용보다 더 강력한 방법
    
    1. 서버가 랜덤한 토큰을 만든다.
    2. 그 토큰을 세션에 저장한다.
    3. 서버가 내려주는 HTML 폼의 hidden input이나 AJAX 헤더에 그 토큰을 넣는다.
    4. 요청이 들어오면 서버가 세션에 저장된 토큰과 요청에 담겨 온 토큰을 비교한다.
    5. 둘이 다르면 요청을 거부한다.
    
    이 방식의 핵심은 정상 사이트가 직접 내려준 페이지에만 토큰이 들어있다는 점임
    
    공격자 사이트는 피해자의 세션 쿠키는 브라우저가 자동으로 붙일 수 있어도, 정상 사이트 세션에 연결된 정확한 CSRF 토큰까지는 마음대로 넣기 어려움
    
    ## `SameSite` + CSRF 토큰 함께 사용
    
    `SameSite` 만으로 부족할 때 전통적이면서 강력한 방법으로 CSRF 토큰을 사용
    
    하고, 또 CSRF 토큰의 장점으로 `SameSite`와 함께 쓰면 더 견고한 보안이 됨
    
    둘 중 하나만 고르기보다 브라우저 차원의 차단(`SameSite` ) + 서버 차원의 검증(CSRF 토큰)을 같이 가져가는 방식이 좋음
    
# CSRF 토큰의 장점과 단점
    
    ## 장점
    
    - 강력하고 유연한 방어수단
    - 레거시 브라우저에서도 동작할 수 있음
    - `SameSite`와 함께 쓰면 **이중 방어**가 가능
    - 특정 액션이나 폼마다 **서로 다른 토큰을 적용하는 식의 세밀한 제어**가 가능
    - 오래 사용돼 온 방식이라 **검증된 방법**으로 소개됨
    
    ## 단점
    
    - **구현이 복잡함** 토큰 생성, 저장, 검증 로직이 모두 필요
    - 모든 폼과 AJAX 요청에 토큰을 붙여야 해서 **개발 오버헤드가 있음**
    - 세션별 토큰 저장과 만료 처리가 필요해 **관리 부담**이 생김
    - 글에서는 **XSS가 있으면 토큰도 유출될 수 있음** 즉, CSRF 토큰이 강력하더라도 XSS까지 막아 주는 것은 아님
# CAPTCHA
    
    CAPTCHA는 **사람과 컴퓨터를 구별하기 위한 자동화된 테스트** 
    
    정식 명칭을 *Completely Automated Public Turing test to tell Computers and Humans Apart* 이고, 쉽게는 “너 로봇 아니지?”를 확인하는 장치임
    
    예시
    
    - 신호등이 있는 사진을 모두 고르기
    - 횡단보도가 보이는 칸을 클릭하기
    - 흐릿한 글자를 읽어 입력하기
    
    글에 따르면 CAPTCHA는 **자동화된 봇 공격을 막는 데 효과적**이며, Google reCAPTCHA 같은 서비스로 구현할 수 있음 다만 사용자가 불편함을 느낄 수 있어서, 보안과 사용성 사이의 균형을 고려해 적용해야 함

# 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건
    
    쿠키 방식의 핵심은 **브라우저가 조건만 맞으면 인증 정보를 자동으로 보내준다는 점**
    
    서버가 `Set-Cookie`로 쿠키를 내려주면, 이후 브라우저는 **같은 도메인으로 요청할 때** 그 쿠키를 `Cookie` 헤더에 자동으로 포함함
    
    그래서 프론트엔드가 매번 직접 인증 정보를 붙이지 않아도 됨
    
    다만 이 “자동 전송”은 **항상 되는 것**이 아니라, **같은 도메인/같은 오리진일 때는 자연스럽게 되고**, 프론트엔드와 백엔드가 서로 다른 도메인이나 포트를 쓰는 **크로스 오리진 환경에서는 기본적으로 막힘** 
    
    예를 들어`app.example.com`과 `api.example.com`도 서로 다른 오리진임
    
     오리진은 **프로토콜, 도메인(서브도메인 포함), 포트**가 모두 같아야 같음
    
    ### 한 줄 정리
    
    - **같은 오리진 요청**: 쿠키 자동 전송
    - **다른 오리진 요청**: 기본적으로 자동 전송되지 않음, 추가 설정 필요
# 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까? 
    
    크로스 도메인에서 쿠키를 보내려면 **서버와 클라이언트가 둘 다 설정을 맞춰야 함**
    
    둘 중 하나라도 빠지면 쿠키가 안 붙음
    
    ### 서버 측 설정
    
    서버는 CORS 설정에서 **credentials 허용**을 켜야 함
    
    `allowCredentials = true`가 필요하고, 이 경우 `Access-Control-Allow-Origin`에 `*`를 쓸 수 없고, **허용할 정확한 origin을 명시해야 함**
    
    ### 클라이언트 측 설정
    
    클라이언트는 브라우저에게 **“이 요청에는 쿠키도 같이 보내라”**고 알려줘야 함
    
    - `fetch` 사용 시: `credentials: 'include'`
    - `axios` 사용 시: `withCredentials: true`
    
    ### 한 번에 정리
    
    - **서버**: credentials 허용 + 허용 origin 명시
    - **클라이언트**: `include` 또는 `withCredentials` 설정
# 쿠키 기반 인증에서 CSRF 공격이 발생할 수 있는 원리
    
    쿠키 기반 인증에서 CSRF가 생기는 이유는 **브라우저가 인증 쿠키를 자동으로 붙여 보내기 때문**
    
    즉, 사용자가 어떤 사이트에 로그인해 세션 쿠키를 가지고 있는 상태에서 악성 사이트를 방문하면, 그 악성 사이트가 정상 사이트로 요청을 유도할 수 있음
    
    이때 브라우저가 쿠키를 같이 보내면 서버는 그것을 **로그인한 사용자의 정상 요청**으로 착각할 수 있음
    
    예시
    
    `bank.com`에 로그인한 상태에서 `bad.com`에 들어갔을 때
    
    만약 크로스 사이트 요청에도 인증 쿠키가 자동으로 붙는다면, `bad.com`이 `bank.com`으로 송금 요청을 만들고 브라우저가 쿠키까지 함께 보내서 실제 송금이 일어날 수 있다는 것이다.
    
    ### 쉽게 말하면
    
    CSRF는 **“비밀번호를 훔치는 공격”** 이라기보다,
    
    **“이미 로그인된 브라우저를 몰래 이용해서 요청을 보내게 하는 공격”** 이라고 이해하면 쉬움 이 설명은 글의 은행/악성 사이트 예시를 풀어 쓴 것
    
# 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)
    
    헤더 방식은 보통 `Authorization: Bearer <token>`처럼 **개발자가 직접 토큰을 헤더에 넣어서 보내는 방식**
    
    브라우저가 자동으로 관리해주지 않기 때문에 조금 번거롭지만, 오히려 장점이 될 수 있음
    
    ### 1) CSRF 방어에 유리함
    
    헤더 방식의 가장 큰 장점은 **CSRF를 원천적으로 막기 쉽다는 점**
    
    악성 사이트가 요청을 보내더라도 브라우저는 `Authorization` 헤더에 토큰을 자동으로 넣어주지 않기 때문에, 공격자는 토큰을 모르면 인증된 요청을 만들기 어려움
    
    ### 2) 필요한 요청에만 선택적으로 전송 가능
    
    쿠키는 조건만 맞으면 자동으로 계속 붙지만, 헤더 방식은 **개발자가 필요한 요청에만 토큰을 붙일 수 있음**
    
    그래서 인증이 필요 없는 요청에는 토큰을 안 보내도 되고, **네트워크 효율성 향상이 됨**
    
    ### 3) CORS 설정이 상대적으로 단순함
    
    글에서는 헤더 방식이 쿠키 방식보다 **CORS 설정이 단순한 편**
    
    쿠키처럼 `credentials` 설정이 필요 없어서, 와일드카드 origin 사용 등에서 더 유연할 수 있음
    
    ### 한 줄 정리
    
    헤더 방식은
    
    - **CSRF에 강하고**
    - **필요한 요청에만 인증 정보를 보낼 수 있고**
    - **쿠키 방식보다 CORS 구성이 단순한 편**
# 토큰을 클라이언트에 저장할 때 LocalStorage, SessionStorage, 메모리 저장의 장단점을 비교
    
    헤더 방식을 쓸 때 **토큰을 브라우저 어디에 저장할지가 중요함**
    
    또한  `LocalStorage`, `SessionStorage`, `메모리 저장` 각각에 보안·편의성 트레이드오프가 있음
    
    ### 1) LocalStorage
    
    LocalStorage는 가장 편함
    
    브라우저를 꺼도 값이 남아 있어서 로그인 유지 구현이 쉽기 때문
    
    하지만 **XSS에 완전히 노출됨**
    
    악성 스크립트가 실행되면 `localStorage.getItem('token')`처럼 토큰을 읽어갈 수 있음
    
    **장점**
    
    - 로그인 유지가 쉬움
    - 브라우저 종료 후에도 남음
    
    **단점**
    
    - XSS에 매우 취약함
    
    ### 2) SessionStorage
    
    SessionStorage는 탭 단위 저장소라서 **탭을 닫으면 자동으로 사라짐**
    
    LocalStorage보다는 조금 안전하지만, **여전히 XSS에는 취약**하다고 설명함 그리고 새 탭을 열면 다시 로그인해야 할 수 있다는 불편함도 있음
    
    **장점**
    
    - 탭 종료 시 자동 삭제
    - LocalStorage보다 잔존 위험이 덜함
    
    **단점**
    
    - 여전히 XSS에 취약함
    - 탭이 바뀌면 불편할 수 있음
    
    ### 3) 메모리 저장
    
    메모리 저장은 JavaScript 변수나 React state 등에 토큰을 두는 방식
    
    **가장 안전한 방법**이지만, 페이지를 새로고침하면 토큰이 사라진다는 단점이 있고, 이 문제는 **Refresh Token을 HttpOnly 쿠키에 저장하는 방식**으로 보완할 수 있음
    
    **장점**
    
    - 상대적으로 가장 안전함
    
    **단점**
    
    - 새로고침하면 사라짐
    
    ### 표로 정리
    
    | 저장 위치 | 장점 | 단점 |
    | --- | --- | --- |
    | LocalStorage | 오래 유지됨, 구현 편함 | XSS에 매우 취약 |
    | SessionStorage | 탭 닫으면 삭제됨 | 여전히 XSS 취약, 탭 간 불편 |
    | 메모리 | 상대적으로 가장 안전 | 새로고침 시 사라짐 |
# Single Page Application, Mobile Application, Server Side Rendering Application 에서 적합한 인증 전략
    
    **환경(**각 앱 구조)마다 인증 전략이 달라질 수 있다고 생각함
    
    ## SPA (Single Page Application)
    
    ### 방법 1) HttpOnly 쿠키 방식
    
    - XSS로부터 토큰을 보호하는 데 유리함
    - 대신 CORS 설정이 복잡하고
    - 클라이언트가 토큰 만료 시간 같은 정보를 직접 보기 어려움
    
    ### 방법 2) 메모리 + Bearer 헤더 방식
    
    - 토큰 만료 확인, 선택적 인증 등에서 더 유연함
    - 대신 토큰 관리 로직을 직접 구현해야 함
    - 새로고침 시 로그인이 풀리는 문제는 Refresh Token을 HttpOnly 쿠키에 저장해 보완할 수 있음
    
    ### 내 생각
    
    SPA에서는 **Access Token은 메모리에 두고, Refresh Token은 HttpOnly 쿠키에 두는 방식**이 가장 좋을 것 같음 보안과 사용성을 같이 챙기기 쉽기 때문
    
    ---
    
    ## Mobile Application
    
    **헤더 방식이 표준적**
    
    브라우저의 쿠키 메커니즘을 그대로 기대하기 어렵고, 대신 iOS Keychain, Android 보안 저장소 같은 **플랫폼 보안 저장소**를 활용할 수 있기 때문
    
    React Native에서는 관련 라이브러리도 있음
    
    ### 내 생각
    
    **보안 저장소에 토큰 저장 + 모든 API 요청에 Bearer 헤더 추가**
    
    ---
    
    ## SSR (Server Side Rendering) Application
    
    **쿠키 방식이 표준적**
    
    이유는 서버가 요청 시점에 쿠키를 읽어서 초기 렌더링을 할 수 있고, 이후 클라이언트에서도 같은 쿠키 흐름을 이어가기 쉽기 때문
    
    ### 내 생각
    
    **HttpOnly 쿠키 기반 인증**
    
    서버 렌더링과 궁합이 좋고, 초기 요청 단계에서 인증 상태를 처리하기 편하기 때문

    ## 최종 정리
    
    ### 환경별 추천 전략
    
    - **SPA**
        - Access Token: 메모리 저장
        - Refresh Token: HttpOnly 쿠키
        - API 요청: Bearer 헤더
    - **Mobile App**
        - 토큰: 플랫폼 보안 저장소
        - API 요청: Bearer 헤더
    - **SSR App**
        - 인증: HttpOnly 쿠키 기반 인증
