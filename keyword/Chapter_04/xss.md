# XSS 공격
## XSS 공격이란

XSS (Cross-Site Scripting)은 공격자가 웹 애플리케이션에 악성 스크립트를 넣고 그 스크립트가 다른 사용자의 브라우저에서 실행되게 만드는 공격

신뢰할 수 없는 입력이 신뢰받는 사이트의 코드처럼 실행되는 것

## 예시

사용자가 입력창에 `<script>...</script>` 같은 코드를 넣었을 때 서버는 이 값을 그대로 저장하거나 그대로 HTML에 출력하면 브라우저는 그것을 단순한 글자가 아니라 실제 JS로 해석해 실행할 수 있음

## 위험한 이유

세션 쿠키 탈취, 가짜 로그인 폼을 통한 피싱, 키 입력 감시, 악성 사이트로 리다이렉트, 개인정보 유출로까지 이어질 수 있기 때문

## 유형

- Stored XSS
    
    DB에 저장된 악성 스크립트가 여러 사용자에게 반복 실행되는 경우
    
- Reflected XSS
    
    URL 파라미터 등에 담긴 스크립트가 즉시 반사되어 실행되는 경우
    
- Dom-based XSS
    
    서버 응답이 아니라 프론트엔드 JS가 DOM을 잘못 조작하면서 발생하는 경우

# XSS 방어 전략
## 1. 사용자 입력을 그대로 HTML에 넣지 않기

가장 기본적인 원칙은 사용자 입력을 신뢰하지 않는 것

입력값은 아무 처리 없이 렌더링하면 브라우저가 악성 스크립트까지 정상 코드처럼 실행할 수 있음

## 2. HTML 이스케이프 처리

사용자가 입력한 값을 텍스트로만 보여주고 싶을 때는 HTML 이스케이프를 해야 함

예를 들어 `<` 를 `&lt;`, `>`를 `&gt;`로 바꾸면 브라우저는 태그가 아니라 그냥 문자로 보기 때문에 스크립트가 실행되지 않음

- 주요 이스케이프 문자
    
    `&`, `<` , `>`, `"`, `'`
    
    특히 `&`는 HTML 엔티티의 시작 문자이기에 가장 먼저 치환해야 이중 이스케이프 문제를 피할 수 있음
    

즉

- 태그처럼 보이는 입력을 실제 태그가 아니라 문자열로 출력하고 싶을 때
- 댓글, 상품명, 닉네임처럼 HTML이 필요없는 입력값일 때

이스케이프를 사용함

## 3. HTML Sanitization

사용자 입력에 일부 HTML은 허용하고 싶지만, 태그나 스크립트는 제거하고 싶을 때 사용함

DOMPurify 같은 라이브러리를 사용해 `<script>` 같은 위험한 요소는 제거함

이스케이프는 HTML 전체를 문자로 바꿔버리는 방식이고, 새니타이즈는 허용 가능한 태그만 남기고 위험한 태그는 제거하는 방식이라는 차이가 있음

따라서 리치 텍스트, 에디터 입력값처럼 일부 마크업을 살려야 하는 경우에 더 적합함

## 4. CSP (Content Security Policy) 설정

추가 방어 수단으로 CSP(Content Security Policy)도 있음

CSP는 HTTP 헤더를 통해 어떤 출처의 스크립트만 실행할지 브라우저에 알려주는 정책

예를 들어 `Content-Security-Policy: default-src 'self'` 같은 설정으로 이를 통해 인라인 스크립트 실행을 제한할 수 있음

다만 CSP는 XSS를 완전히 없애는 방법보다 XSS가 발생했을 때 피해를 줄여주는 추가 방어막에 가깝기에 보통 이스케이프나 새니타이즈와 함께 씀
