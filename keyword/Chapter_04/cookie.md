# HTTP는 왜 무상태(Stateless)로 설계되었나?
    
    HTTP는 각 요청을 서로 완전히 독립적으로 처리하도록 설계됨
    
    즉, 서버는 이전 요청의 내용을 기본적으로 기억하지 않음 이렇게 만든 이유는 초창기 웹이 복잡한 애플리케이션보다는 문서를 단순하게 주고받는 것이 중심이었고, 서버가 사용자별 상태를 계속 들고 있지 않으면 구조가 단순해지고 확장하기 쉬워지기 때문
    
    쉽게 말하면, “HTTP는 매 요청을 처음 보는 요청처럼 처리하자”는 철학에 가까움 그래서 서버를 여러 대로 늘려도 특정 사용자의 이전 상태를 맞춰 들고 있는 필요가 적음 대신 로그인 상태 유지나 장바구니 같은 기능은 HTTP 자체가 아니라 쿠키 같은 추가 장치로 해결하게 됨
    
# HTTP의 무상태성이 주는 장점과 단점
    
    ## 장점
    
    - 확장성이 좋음
        
        어떤 서버든 요청을 다 처리할 수 있어서 수평 확장이 쉬움
        
    - 장애 대응이 쉬움
        
        특정 서버에서 죽어도 다른 서버가 대신 처리해주기 쉬움
        
    - 구현이 단숨한
        
        서버가 사용자별 상태를 오래 관리하지 않아도 되기 때문
        
    
    ## 단점
    
    - 로그인 상태 유지가 안 됨
        
        로그인에 성공해도 다음 요청에서 서버는 그 사실을 기본적으로 모르기 때문
        
    - 연속성 있는 기능 구현이 어려움
        
        장바구니, 다단계 회원가입, 사용자 맞춤 페이지 같은 기능은 별도 상태 관리 수단이 필요함
        
# 쿠키의 Domain 디렉티브
    
    `Domain`은 이 쿠키를 어느 도메인까지 보낼지를 정하는 옵션
    
    기본적으로 쿠키는 쿠키를 설정한 현재 호스트에만 돌아가는 host-only 쿠키가 되는데, 이 경우에는 서브 도메인에 자동으로 공유되지 않음
    
    eg. `login.example.com` 이 쿠키를 만들었는데 `Domain`을 따로 지정하지 않으면, 그 쿠키는 기본적으로 `login.example.com`에만 붙음, 하지만 `Domain=example.com`으로 저장하면 `example.com`과 그 하위 서브 도메인들에서도 보낼 수 있음 그래서 로그인 서브도메인과 메인 서비스 도메인 사이에서 인증상태를 공유할 때 자주 활용됨
    
    정리하면
    
    - `Domain` 생략
        - 현재 호스트에만 적용
        - 서브 도메인에는 공유되지 않음
    - `Domain=example.com`
        - `example.com`과 그 서브도메인들에 전송 가능
    - 아무 도메인이나 지정할 수는 없음
        - 현재 서버와 관련 없는 외부 도메인은 거부됨
# 쿠키의 Path 디렉티브
    
    `Path`는 사이트 안에서 어떤 URL 경로에 요청할 때 쿠키를 보낼지를 정하는 옵션
    
    eg. `Path=/private`로 쿠키를 설정하면, `/private` 및 그 하위 경로 요청에는 쿠키가 붙고, `/public` 같은 다른 경로에는 붙지 않음
    
    `Path`를 생략하면 기본값은 쿠키를 설정한 요청 URL의 경로 부분이 됨
    
    정리하면
    
    - `Path=/private`
        - `/private`, `/private/...` 에는 전송
        - `/public`에는 전송되지 않음
    - `Path`는 전송 범위를 줄이는 용도이지,
        - 보안 경계 자체는 아님
        - 다른 경로에서 쿠키를 읽지 못하게 완전히 막는 장치로 보면 안 됨
- 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠
    
    ## 세션 쿠키
    
    세션 쿠키는 브라우저 세션이 끝날 때 제거되는 쿠키
    
    보통 `Expires`이나, `Max-Age`를 지정하지 않으면 세션 쿠키가 되어, 로그인 세션처럼 브라우저를 닫으면 끝나도 되는 상태에 자주 쓰임
    
    ## 영속 쿠키
    
    영속 쿠키는 특정 시점까지 남아 있는 쿠키
    
    `Expires` 또는 `Max-Age`를 지정해서 브라우저를 닫아도 유지되게 만들 수 있음
    
    로그인 상태 유지, 오늘 하루 보지 않기 같은 기능에 자주 사용됨
    
    ### 정리
    
    - 세션 쿠키
        - `Expires`, `Max-Age` 없음
        - 브라우저 세션 종료 시 제거
    - 영속 쿠키
        - `Expires`, `Max-Age` 있음
        - 지정한 시간까지 유지
# 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나?
    
    ## HttpOnly
    
    `HttpOnly`는 자바스크립트에서 `document.cookie`로 쿠키를 읽지 못하게 만듦
    
    그래서 악성 스크립트가 쿠키를 훔쳐가는 XXS 기반 쿠키 탈취를 줄이는 게 중요함 다만 HttpOnly 쿠키도 브라우저가 요청에 자동으로 붙여 보내는 것 자체는 가능함
    
    ## Secure
    
    `Secure`은 쿠키를 HTTPS 요청에서만 전송하게 만듦
    
    그래서 HTTP 평문 전송 과정에서 쿠키가 노출되는 위험을 줄여 중간자 공격(MITM)에 더 강해짐 다만 `Secure`만으로 자바스크립트 접근이나 로컬 탈취까지 모두 막는 것은 아님
    
    ## SameSite
    
    `SameSite` 은 크러스 사이트 요청에 쿠키를 얼마나 실을지 제어하는 속성
    
    핵심 목적은 CSRF 공격 방어
    
    `Strict`는 같은 사이트 요청에만, `Lax`는 일부 안전한 top-level navigation에만, `None`은 크로스 사이트 요청에도 보내되 반드기 `Secure`가 함께 있어야 함
    
# 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)
    
    ## 1. 용량 제한
    
    쿠키는 작은 데이터만 저장하도록 설계되었기에 (약 4KB) 많은 사용자 정보나 복잡한 상태를 담기엔 부적합함
    
    ## 2. 보안 한계
    
    쿠키는 클라이언트 쪽에 저장되므로 사용자가 개발자 도구 등으로 값을 바꾸려 시도할 수 있음
    
    민감한 권한 정보 자체를 쿠키에 그대로 넣으면 위험함
    
    ## 3. 네트워크 오버헤드
    
    쿠키는 해당 조건에 맞는 요청마다 자동으로 전송됨
    
    이미지, CSS, API 요청 등에도 계속 붙으면서 불필요한 트래픽이 생길 수 있음
    
    ## 4. 도메인 제약
    
    쿠키는 설정된 도메인 범위 안에서만 전송되고 접근됨
    
    이는 보안을 위한 장점이지만, 여러 도메인에 서비스가 흩어진 구조나 SSO처럼 인증을 공유해야 하는 경우엔 제약이 됨
    
# 쿠키만으로 상태 관리를 해결할 수 없는 이유
    
    쿠키는 어디까지나 브라우저에 작은 값을 저장하고 다음 요청에 자동으로 다시 보내게 해주는 수단일 뿐임
    
    상태를 담을 수는 있어도, 그 상태가 진짜인지, 안전한지, 언제까지 유효한지, 서버가 어떻게 통제할지까지 해결해주지는 못함
    
    ### 이유
    
    - 사용자 조작 가능성
        - 쿠키 값 자체를 클라이언트가 바꾸려 시도할 수 있으므로, 서버가 그대로 믿으면 안 됨
    - 저장 용량이 작음
        - 복잡한 상태 전체를 담기 어려움
    - 인증/권한 통제 기능이 부족함
        - 로그인 유지, 즉시 로그아웃, 권한 변경 반영 같은 문제는 보통 세션 저장소나 토큰 검증 로직이 따로 필요함
        - 따라서 실무에서는 쿠키 단독보다 세션 ID를 담거나, 토큰을 담거나, 둘을 조합하는 방식으로 주로 사용함
    
    쿠키는 상태 관리 전체의 해답이 되지 못하기에 실제 인증 시스템에서는
    
    쿠키 + 세션, 쿠키 + JWT 같은 조합이 많이 쓰임
