# 세션 하이재킹이란?
## 세션 하이재킹이란?

새션 하이재킹(Session Hijacking)은 공격자가 사용자의 세션 ID를 탈취한 뒤, 그 사용자로 가장해서 서버에 요청을 보내는 공격

HTTP는 무상태이기 때문에 로그인 상태를 유지하려고 서버가 세션을 만들고 브라우저는 보통 쿠키에 세션 ID를 담아 보내는데, 이 세션 ID를 빼앗기면 공격자는 피해자인 척 할 수 있음

### XSS를 이용한 쿠키 탈취

공격자가 악성 스크립트를 주입하고, 피해자가 그 페이지를 열면 그 스크립트가 `document.cookie`로 세션 쿠키를 읽어 공격자 서버로 보내고, 공격자는 그 세션 ID로 피해자 계정에 접근

쉽게 말하면 피해자의 신분증(세션 쿠키)를 훔쳐 피해자인 척 하는 것

# XSS 공격을 차단하는 방법
## 1. 입력값 이스케이프 처리

XSS 취약점 자체를 없애는 것

사용자가 입력한 값을 그대로 HTML에 넣지 말고, `<`, `>`, `&` 같은 특수문자를 이스케이프해서 브라우저가 태그나 스크립트가 아니라 문자열로 처리하게 하는 것

예를 들어 사용자가 `<script>...</script>` 를 넣어도 이를 `&lt;script&gt;...&lt;/script&gt;`처럼 바꾸면 브라우저는 실행 가능한 코드로 해석하지 못함

그래서 댓글, 입력폼, 게시글 제목 같은 사용자 입력 영역에는 이스케이프로 기본 방어를 함

## 2. HttpOnly 쿠키 사용

XSS 자체를 없애는 방법은 아니지만, XSS가 발생했을 때 자바스크립트가 `document.cookie`로 세션 쿠키를 읽지 못하게 막아 쿠키 탈취 피해를 줄이는 핵심 방어책

## 3. WAF, CSP 같은 추가 방어 수단

입력값 처리만 믿지 않고, 브라우저 정책과 보안 장비로 한 번 더 막는 추가 방어책

# 쿠키의 다양한 옵션들

아래는 실무에서 자주 보는 `Set-Cookie` 옵션

## `HttpOnly`

- 자바스크립트에서 쿠키 접근을 막는 옵션
- `document.cookie`로 읽을 수 없게 해서 XSS를 통한 쿠키 탈취를 줄임
- 브라우저는 HTTP 요청에 이 쿠키를 자동으로 포함

## `Secure`

- 쿠키를 HTTPS 연결에서만 전송하게 만듦
- HTTP 구간에서 쿠키가 노출되는 것을 줄여 네트워크 스니핑에 대한 방어에 도움을 줌
- 프로덕션 환경에서는 필수로 권장

## `SameSite`

- 크로스 사이트 요청에 쿠키를 얼마나 보낼지 정하는 옵션
- CSRF 방어용
    - `Strict` : 같은 사이트 요청에만 보냄
    - `Lax`: 일부 안전한 요청에 보냄
    - `None`: 크로스 사이트 요청에도 보내지만 `Secure`가 필요함

`SameSite`는 CSRF를 어느 정도로 줄여주는 옵션이지 완전한 방어책은 아니기에, 진짜 CSRF 방어는 토큰이나 커스텀 헤더 검증이 추가로 필요할 수 있음

## `Path`

- 쿠키를 어떤 경로의 요청에 보낼지 정함
- 예를 들어 `Path=/`면 사이트 전역, `Path=/admin`이면 그 경로 아래 요청에만 보낼 수 있음
- 현재 서버와 관련 없는 외부 도메인으로 설정할 수 없음 (브라우저가 거부함)

## `Max-Age`

- 쿠키가 몇 초 동안 유효한지 정함
- 예를 들어 `Max-Age=3600`이면 1시간 뒤 만료됨
- 세션 타임아웃을 줘서 탈취된 쿠키의 유효시간을 줄이는 방법이 있음

## `Expires`

- 쿠키의 정확한 만료 시각을 날짜로 지정함
- `Max-Age`와 비슷하게 영속 쿠키를 만들 때 쓰이며, 둘 다 있을 경우 일반적으로 `Max-Age`가 우선함

---

### 세션 쿠키 vs 영속 쿠키

- `Expires` 나 `Max-Age`가 없으면 보통 세션 쿠키
- `Expires` 또는 `Max-Age`가 있으면 영속 쿠키
