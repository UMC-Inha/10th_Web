- `pushState`, `popstate` 이벤트, 전체 리로드와의 차이 🍠
    
    **`pushState(state, title, url)`**: 브라우저 주소창의 URL을 변경하고, 브라우저 방문 기록(History)에 새로운 줄을 추가합니다. 가장 중요한 특징은 주소만 바뀔 뿐, 서버에 새로운 페이지를 요청(전체 리로드)하지 않는다는 것입니다.
    
    **`popstate`** 이벤트: 사용자가 브라우저의 '뒤로 가기'나 '앞으로 가기' 버튼을 누를 때 발생하는 이벤트입니다. 주소가 바뀌었으니 화면을 다시 그리라고 리액트에게 알려주는 알람시계 역할을 합니다.
    
- 전체 리로드 방식과 SPA 라우팅 방식의 가장 큰 차이는 무엇일까? 🍠
    - MPA는 링크를 누를 때마다 브라우저가 새로고침 됩니다. 이 과정에서 자바스크립트 메모리가 싹 다 초기화됩니다. 예를 들어, 사용자가 폼(Form)에 입력하던 글자나, 재생 중이던 유튜브 영상, `useState`로 저장해 둔 데이터들이 페이지를 이동하면 전부 날아갑니다.
    - SPA는  최초 접속 시 하나의 큰 HTML/JS 파일을 받아옵니다. 그 이후로는 `pushState`를 써서 주소만 바꾸고, 자바스크립트가 알아서 화면의 필요한 '컴포넌트'만 레고 블록 갈아 끼우듯 교체합니다. 새로고침이 없으므로 리액트의 상태(State)가 그대로 유지됩니다. 음악을 재생해 둔 채로 다른 페이지(컴포넌트)로 이동해도 음악이 끊기지 않는 부드러운 앱(App) 같은 경험을 제공합니다.
- `preventDefault()`와 `stopPropagation()`의 차이와 역할은 무엇인가? 🍠
    
    **`e.preventDefault()`** (기본 동작 막기):  브라우저가 해당 태그에 대해 원래 기본적으로 하려고 했던 행동을 취소합니다. `<a>` 태그를 눌렀을 때 다른 페이지로 튕겨 나가는(전체 리로드) 현상을 막고 싶을 때. `<form>`에서 엔터를 치거나 `submit` 버튼을 눌렀을 때 페이지가 새로고침 되는 것을 막고 싶을 때.
    
    `e.stopPropagation()` (이벤트 전파/버블링 막기): 클릭 같은 이벤트가 부모 태그로 타고 올라가는(Bubbling) 현상을 차단합니다.  카드(부모 `div`) 전체에 '상세 페이지 이동' 클릭 이벤트가 있고, 그 카드 안에 쪼그만 '좋아요 버튼(자식 `button`)'이 있다고 가정해 봅시다. 좋아요 버튼을 눌렀는데, 부모 카드까지 클릭된 걸로 인식해서 상세 페이지로 튕겨 나가는 대참사가 발생합니다. 이때 좋아요 버튼에 `e.stopPropagation()`을 걸어주면, "나만 클릭된 걸로 치고 부모한테는 클릭됐다고 소문내지 마!" 하고 막아줍니다.
    
- 선언적 라우팅(`Route`, `Routes`) 구조가 가지는 장점은 무엇일까? 🍠
    
    태그 형태로 주소와 컴포넌트를 매칭하는 방식을 선언적(Declarative) 라우팅이라고 합니다. 선언적 라우팅을 하지 않으면 `if-else` 구조로 코드를 짜야합니다. 
    
    선언적 라우팅의 장점은 
    
    1. 직관적인 가독성
    2. 컴포넌트의 일관성 : 라우트 설정 로직마저도 리액트의 컴포넌트 형태를 띄고 있기 때문에 리엑트의 문법과 생태계를 거스르지 않고 자연스럽게 섞여 들어 갑니다.
    3. 유지보수와 중첩의 편리함: 페이지 내에서 복잡한 UI구조를 짤때 레고 조립하듯 매우 쉽게 구현할 수 있습니다.1


    - 영상을 보고 학습한 내용을 정리해주세요!
    
    `useEffect`  화면을 그리는 작업 외에 발생하는 모든 부수효과를 전담해서 처리하는 리엑트 훅입니다. `useState`의 비동기적 특성 때문에 바로 최신값을 조회할 수 없는 문제를 해결합니다. `useEffect`는 화면 렌더링이 완전히 끝난 직후에 실행되므로 언제나 업데이트된 최신 상태 값을 보장받고 조작할 수 있습니다.
    
    **Dependenct Array**: `useEffect`의 인자인 [] 안에 무엇을 넣느냐에 따라 실행 횟수가 완전히 달라집니다. 빈배열일 경우에는 마운트일때 딱 한번만 실행됩니다. 특정 변수를 넣을때는 지정해든 특정변수가 값이 바뀔때마다 코드가 다시 실행됩니다.
    
    **Clean Up Function:** `useEffect`가 컴포넌트가 사라지기 직전이나 다음 효과가 실행되기 직전에 메모리에 남았는 찌거기를 제거해줘야합니다. 
    
    **주의 사항!**
    
    `useEffect` 안에서 상태를 변경하는 함수 를 실행하면서, 동시에 의존성 배열에 그 상태를 넣으면 **절대 안 됩니다.**  
    
     상태가 바뀜 ➡️ 의존성 배열이 감지하고 `useEffect` 재실행 ➡️ 또 상태가 바뀜 ➡️ 또 `useEffect` 재실행... 이라는 **무한 렌더링 지옥**에 빠지게 됩니다. 만약 안에 서버 요청 코드가 있었다면 서버에 무한 디도스(DDoS) 공격을 가하는 셈이 되므로 극도로 주의해야 합니다.

     - **`fetch`** 정리
    
    별도의 설치 없이 최신 브라우저에 기본으로 내장된 함수입니다.
    
    - **장점:**
        - **설치 불필요:** 외부 라이브러리에 의존하지 않아 프로젝트가 가볍습니다.
        - **순정 기술:** 자바스크립트 표준(ES6+)이므로 유행을 타지 않고 어디서든 작동합니다.
    - **단점:**
        - **수동 처리:** JSON 변환, 에러 핸들링 등을 개발자가 직접 일일이 코드로 짜야 합니다.
        - **기능 부족:** 타임아웃(Timeout) 설정이나 요청 취소 같은 고도화된 기능을 구현하기 복잡합니다.
- **`axios`** 정리
    
    Node.js와 브라우저 모두에서 사용할 수 있는 가장 인기 있는 HTTP 클라이언트 라이브러리입니다.
    
    - **장점:**
        - **자동화:** JSON 변환, 에러 처리 등을 알아서 해줘서 코드가 간결해집니다.
        - **풍부한 기능:** 요청/응답 가로채기(Interceptors), 요청 취소, 타임아웃 설정 등 실무에 필요한 기능이 다 들어있습니다.
        - **호환성:** 구형 브라우저(IE 등) 지원 범위가 더 넓습니다.
    - **단점:**
        - **설치**
- **`fetch`**와 **`axios`**의 차이
    
    #### 인터셉터(Interceptors)의 존재
    
    **axios**는 요청을 보내기 직전에 모든 요청에 로그인 토큰을 붙이는 전역 설정을 할 수 있습니다. **fetch**는 매번 함수를 호출할 때마다 헤더에 토큰을 넣어주는 코드를 반복해서 적어야 합니다.
    
    #### JSON 데이터 처리 방식의 차이
    
    **fetch:** 서버에서 데이터를 가져온 후, 이게 JSON이라는 것을 알려주기 위해 **`res.json()`이라는 함수를 한 번 더 호출**하고 기다려야 합니다. (2단계)
    
    **axios:** 서버가 준 데이터를 알아서 JSON으로 변환해서 **`response.data`** 안에 쏙 넣어줍니다. 우리는 그냥 꺼내 쓰기만 하면 됩니다. (1단계)

| **비교 항목** | **fetch (네이티브)** | **axios (라이브러리)** |
| ---         | ---                 | ---                   |
| **설치 여부** | **불필요** (브라우저 내장) | **필요** (`npm` 또는 `yarn`) |
| **JSON 처리** | **수동** (`.json()` 호출 필수) | **자동** (응답 객체의 `data`에 바로 담김) |
| **에러 처리** | **404, 500을 에러로 안 봄** | **400, 500번대 모두 에러로 처리** |
| **타임아웃** | 지원 안 함 (별도 구현 필요) | 설정 한 줄로 가능 (`timeout: 5000`) |
