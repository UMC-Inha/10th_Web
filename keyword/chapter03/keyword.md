- `pushState`, `popstate` 이벤트, 전체 리로드와의 차이 🍠
    
    **전체 리로드:** 사용자가 링크를 클릭하면 브라우저가 서버에 새 페이지를 요청하여 새로고침 됩니다. 이때 메모리가 초기화 되어 이전에 사용하던 State와 같은 데이터가 사라지게 됩니다.
    
    **pushState**:  서버로 새로운 요청을 보내지 않으면서 브라우저의 주소창 URL만 살짝 바꾸어 페이지를 새로고침 하지 않고 자바스크립트가 바뀐 Url에 따라서 내용물만 갈아 끼웁니다.
    
    popState는 뒤로 가기/앞으로 가기 버튼을 눌렀을 때 발생하는 이벤트로 스택에서 이전 주소를 꺼낸 후 발생하여 리엑트에게 바뀐 주소에 따라 컴포넌트를 다시 렌더링 하라고 알려줍니다
    
- 전체 리로드 방식과 SPA 라우팅 방식의 가장 큰 차이는 무엇일까? 🍠
    
    전체 리로드 방식과 SPA 라우팅 방식의 가장 큰 차이는 서버와의 요청 차이입니다
    전체 리로드는 매번 '전체 HTML/CSS/JS'를 다시 받지만, SPA는 최초에 한 번 로드한 뒤에는 JSON 데이터만  주고받으며 자바스크립트가 특정 부분만 갈아 끼우기 서버 요청면에서 훨씬 가볍습니다
    
    또한 전체 리로드는 새로고침으로 메모리가 소실되기에 그렇지 않은 SPA 방식에 비해 사용자 경험 측면에서도 이전 행동들을 반영하지 못한다는 큰 차이가 있습니다
    
- `preventDefault()`와 `stopPropagation()`의 차이와 역할은 무엇인가? 🍠
    
    **`pevent.preventDefault()`**:는 a태그의 페이지 이동 form의 submit 기능 같이 브라우저의 기본기능을 막습니다
    
    `stopPropagation()` 는 이벤트가 부모로 전파되는 것을 막는 역할 입니다. 예를 들면 특정 컴포넌트에서 버튼을 눌렀는데 부모 영역의 클릭 이벤트까지 같이 동작하여 의도하지 않은 이벤트가 일어나는 것을 방지하는 역할을 합니다
    
- 선언적 라우팅(`Route`, `Routes`) 구조가 가지는 장점은 무엇일까? 🍠
    
    명령형 방식은 `if (url === '/admin')` 같은 조건문이 코드 곳곳에 흩어질 수 있지만, 선언적 방식은 한곳에서 모든 경로를 관리하기 때문에 직관적인 구조를 가져 유지보수가 용의하며,
    새로운 페이지를 추가할 때 새로운 루트의 path 한 줄만 추가하면 되므로 확장성에도 유리합니다.
    
    또한 특정 페이지 안에 다른 탭이 있는 복잡한 UI를 구성할 때,
    특정 레이아웃은 고정시킨 채 내부의 내용만 갈아 끼우는 구조를 표현하기 쉽습니다.
    이는 코드의 재사용성을 높여주고, 불필요한 전체 화면 렌더링을 방지합니다.
    
    주소가 변하면 선언된 규칙에 따라 컴포넌트를 렌더링하기에 직접 `pushState`나 `popstate`를 다룰 때 발생할 수 있는 주소는 바뀌었는데 화면은 안 바뀌는 것을 방지합니다
    
    `:id`와 같은 파라미터를 사용하여 `/user/1`, `/user/2` 같은 수많은 페이지를 하나의 `<Route>`로 처리할 수 있습니다.

- 위의 영상을 보고 학습한 내용을 정리해주세요!
    
    `useEffect`  화면을 그리는 작업 외에  부수효과를 처리하는 리엑트 훅입니다. `useState`와는 다르게
    `useEffect`는 화면 렌더링이 완전히 끝난 직후에 실행되므로 언제나 업데이트된 최신 상태 값을 보장받고 조작할 수 있습니다.
    
    useEffect의 인자인 [] 안에 무엇을 넣느냐에 따라 실행 횟수가 달라집니다. 빈배열일 경우에는 마운트일 때 딱 한번 실행되지만, 특정 변수를 넣을때는 지정힌 변수의 값이 바뀔때마다 재실행됩니다.
    
    useEffect`가 컴포넌트가 사라지기 직전이나 다음 효과가 실행되기 직전에 메모리에 남아 있는 쓰레기들 제거해줘야 메모리 누수가 일어나지 않습니다.
    
    useEffect에서 상태를 변경하는 함수를 실행하면서, 의존성 배열에 그 상태를 넣으면 상태 변화를 시키는 함수가 해당 상태 변화를 감지하여 실행되기에  무한 루프에 빠지게 됩니다. 이때 만약 서버 요청 코드가 있다거나 한다면 서버에 디도스 공격을 가하는 셈이 되므로 조심해야 합니다.


1. fetch 

특징:

브라우저에서 바로 사용 가능한 내장형 인터페이스로 라이브러리 설치가 필요 없어 프로젝트 용량이 가벼워집니다.
기능이 단순하여 세부적인 설정은 개발자가 직접 구현해야 할 때가 많습니다.

주의점:

두 단계 응답: 데이터를 받을 때 .json() 과정을 거쳐야 실제 데이터에 접근할 수 있습니다.
404나 500 에러를 실패가 아닌 통신은 성공했으므로 성공으로 처리합니다. 따라서 response.ok 상태를 수동으로 체크해야 합니다.
구형 브라우저에서는 작동하지 않을 수 있습니다

2. axios

특징:

자동 JSON 변환: 응답받은 데이터를 .json() 없이도 알아서 객체로 바꿔줍니다 .
폭넓은 브라우저 호환성: 구형 브라우저에서도 안정적으로 동작합니다.
강력한 기능: 요청 취소(Cancel), 타임아웃 설정, 인터셉터(Interceptors) 등 편리한 기능을 기본 제공합니다.

주의점:

npm이나 yarn을 통해 별도로 설치해야 하며 아주 미세하지만 프로젝트 번들 크기가 커질 수 있습니다.

3. fetch와 axios의 차이

라이브러리 독립성:

fetch와 axios는 각각 내장형 인터페이스와 라이브러리로 fetch는 별도의 설치의 필요없이 가벼운 프로젝트를 유지할 수 있지만
axios는 외부 라이브러리 이기에 npm등으로 설치를 해야합니다

Json:

fetch는 응답받은 데이터를 .json()을 통해 객체로 전환하여 사용해야 하지만 axios는 자동으로 데이터를 변환해 즉시 사용 할 수 있습니다

에러처리:

fetch는 404, 500같은 에러를 통신이 성공 했기에 성공으로 처리하여 response.ok로 검수해야하지만 axios는 즉시 에러로 던져줍니다

부가기능:

fetch는 타임아웃, 요청 취소등의 추가 기능 구현이 번거롭지만 axois는 옵션으로 간편하게 추가 가능합니다
