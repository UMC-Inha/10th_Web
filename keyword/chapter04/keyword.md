# 사용자 인증부터 폼 유효성 검사까지, 서비스의 기본기 다지기 (1)

 <br>
 <br>
 
# 세션 방식과 토큰 방식의 가장 큰 차이는 무엇인가요? 🍠
> **세션 = 서버가 상태를 저장하는 Stateful 방식
토큰 = 클라이언트가 상태를 들고 다니는 Stateless 방식**
>

---

| 구분      | 세션(Session)      | 토큰(Token, JWT 등)         |
| --------- | ------------------ | --------------------------- |
| 상태 저장 | 서버               | 클라이언트                  |
| 인증 방식 | 세션ID로 서버 조회 | 토큰 자체 검증              |
| 구조      | Stateful           | Stateless                   |
| 서버 부담 | 높음 (저장 필요)   | 낮음 (저장 없음 or 적음)    |
| 확장성    | 낮음               | 높음                        |
| 보안      | 상대적으로 안전    | 토큰 탈취 위험              |
| 로그아웃  | 쉬움 (세션 삭제)   | 어려움 (토큰 만료까지 유효) |
| 저장 위치 | 쿠키 (세션ID)      | 헤더 / 쿠키 / storage       |

---

## 동작 흐름 비교

### 세션 방식

👉 **서버가 기억한다.**

```jsx
로그인 성공
→ 서버가 세션 생성
→ 세션ID를 쿠키로 전달

이후 요청
→ 쿠키 자동 전송
→ 서버가 세션 조회
→ 사용자 확인
```

---

### 토큰 방식

👉 **클라이언트가 증명한다.**

```jsx
로그인 성공
→ 서버가 토큰 발급

이후 요청
→ 토큰을 헤더에 담아 전송
→ 서버가 토큰 검증
→ 사용자 정보 추출
```

---

## 로그아웃 처리

### 세션

👉 매우 쉽다.

```jsx
서버에서 세션 삭제 → 끝
```

- 서버가 상태를 들고 있어 세션을 지우면 즉시 무효가 된다.

✔ **특징**

- 강제 로그아웃 가능
- 관리자 차단 가능
- 즉시 반영

---

### 토큰

👉 어렵다.

```jsx
이미 발급된 토큰은 만료 전까지 유효
```

- 서버가 토큰을 기억하지 않아 삭제할 수 없다.

✔ **해결 방법**

- 블랙리스트
- 짧은 만료 시간
- refresh token 구조

---

## 서버 부담

### 세션

👉 부담이 크다.

```jsx
서버가 모든 사용자 상태 저장
```

- 유저가 많아지면 메모리가 증가하여 세션 저장소가 필요하다.

---

### 토큰

👉 부담이 적다.

```jsx
서버가 상태 저장 안 함
```

- 요청마다 검증만 하면 되고 DB 조회 없이 처리 가능하다.

---

## 확장성

### 세션

👉 확장이 어렵다.

```jsx
서버 여러 대 → 세션 공유 문제 발생
```

예:

- A 서버 로그인
- B 서버 요청 → 로그인 모름 ❌

✔ **해결 방법**

- Redis
- Sticky Session

---

### 토큰

👉 확장이 매우쉽다.

```jsx
어느 서버든 토큰만 검증하면 됨
```

- 서버 간 상태 공유가 필요 없어 마이크로서비스 구조에 유리하다.

---

# 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징을 정리해주세요. 🍠

> 서버가 여러 대일 때 <br>
> 👉 “세션을 어떻게 **공유**를 할 것인가?!”

---

## 1️⃣ Sticky Session (세션 고정)

👉 **한 사용자를 항상 같은 서버로 보내는 방식**

### 구조

```jsx
사용자 → 로드밸런서 → 항상 같은 서버(A)
```

### 동작 방식

- 로드밸런서가 사용자 요청을 기억
- 항상 같은 서버로 라우팅

### ✔ 장점

- 구현이 매우 간단
- 기존 세션 방식 그대로 사용 가능
- 별도 저장소 필요 ❌

### ✔ 단점

- 특정 서버에 트래픽이 몰린다.
- 서버 장애 시 세션이 날아간다.
- 확장성 떨어짐

---

## 2️⃣ Session Replication (세션 복제)

👉 **모든 서버가 서로 세션 데이터를 복사해서 공유**

### 구조

```jsx
A 서버 ↔ B 서버 ↔ C 서버
(세션 서로 복제)
```

### 동작 방식

- 한 서버에 세션 생성
- 다른 서버들에 복제

### ✔ 장점

- 어느 서버로 가도 로그인 유지 가능
- Sticky Session 필요 ❌

### ✔ 단점

- 네트워크 트래픽 증가
- 서버 수가 많아질수록 비효율
- 동기화 문제 발생 가능

---

## 3️⃣ Centralized Session Store (중앙 저장소)

👉 세션을 별도의 중앙 저장소(Redis 등)에 저장

### 구조

```jsx
사용자 → 서버 A/B/C
        ↓
     Redis (세션 저장)
```

### 동작 방식

- 모든 서버가 같은 저장소 사용
- 세션은 Redis/DB에 저장
- 서버는 세션ID로 조회

### ✔ 장점

- 확장성 매우 좋음
- 서버 간 상태 공유 문제 ❌
- 장애 대응에 유리

### ✔ 단점

- Redis 등 인프라가 필요
- 네트워크 비용 발생
- 저장소 장애 시 영향 큼

---

## 전체 비교

| 전략                      | 개념               | 장점           | 단점             | 추천 상황  |
| ------------------------- | ------------------ | -------------- | ---------------- | ---------- |
| **Sticky Session**        | 같은 서버로만 보냄 | 간단           | 장애/확장성 문제 | 소규모     |
| **Session Replication**   | 서버끼리 복제      | 어느 서버든 OK | 트래픽/비효율    | 거의 안 씀 |
| **Central Store (Redis)** | 중앙 저장소 사용   | 확장성 최고    | 인프라 필요      | 실무 표준  |

# JWT(Json Web Token)의 장점과 단점을 각각 설명해주세요. 🍠

## 장점

### 1️⃣ Stateless (서버 부담 ↓)

> **서버가 상태를 저장하지 않아도 된다.**

- 토큰 안에 사용자 정보가 있어서
- 서버는 검증만 하면 된다.

✔ **효과**

- 서버 메모리 사용 ↓
- 세션 저장소 필요 ❌

---

### 2️⃣ 확장성이 좋음

> **어느 서버든 동일하게 처리 가능하다.**

- 서버 간 세션 공유가 필요 없어
- 마이크로 서비스 / 클라우드 환경에 적합하다.

✔ **예**

- 서버 A 로그인 → 서버 B 요청 → 문제 없음

---

### 3️⃣ 다양한 클라이언트 지원

> **웹, 모바일, API 모두 동일하게 사용 가능하다.**

- React / 앱 / 외부 서비스 연동 쉬움
- Authorization 헤더로 통일 가능

---

### 4️⃣ 빠른 인증 처리

> **DB 조회 없이 처리 가능**

- payload에 정보를 포함하여
- 서명 검증만 하면 끝

---

### 5️⃣ 정보 전달 가능

> **토큰 자체에 데이터 포함 가능**

```jsx
{
  "userId": 1,
  "role": "ADMIN"
}
```

- 인증 + 간단한 인가 처리 가능

---

## 단점

### 1️⃣ 로그아웃 / 강제 만료가 어렵다

> **이미 발급된 토큰은 서버가 직접 끊기 어렵다**

- 만료 전까지 계속 사용 가능

✔ **해결**

- 블랙리스트
- 짧은 만료시간
- refresh token

---

### 2️⃣ 토큰 탈취 위험

> **토큰 = 인증 수단 자체**

- 탈취되면 바로 사용자처럼 행동이 가능하다.

---

### 3️⃣ payload 노출 ( 암호화 ❌ )

> **누구나 디코딩이 가능**

- `base64` 인코딩일 뿐
- 민감 정보를 넣으면 안된다!!

---

### 4️⃣ 토큰의 크기가 크다.

> **매 요청마다 전송된다.**

- 세션ID보다 무겁고 네트워크 비용이 증가한다.

---

### 5️⃣ 정보 변경 반영이 어렵다

> **토큰 안에 있는 내용은 발급 시점 기준이다.**

예:

- ADMIN → USER 변경
- 기존 토큰은 계속 ADMIN

✔ **해결**

- 짧은 만료시간
- DB 재검증하기

---

# JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법은 무엇인가요? 🍠

## JWT는 왜 “즉시 무효화”가 어려울까?

👉 JWT는 **Stateless** 구조이기 때문!

---

## 동작 과정

JWT는 로그인 시 이렇게 동작한다:

1. 서버가 토큰 발급
2. 클라이언트가 토큰을 저장
3. 이후 요청마다 토큰 전송
4. 서버는 토큰 검증만 하고 통과

<aside>
💡

**서버는 토큰을 저장하지 않는다!!!!**

</aside>

---

## 생기는 문제

예: 로그아웃 상황

1. 사용자가 로그인 → JWT 발급
2. 로그아웃 클릭
3. 이때 서버는

**아무것도 하지 못한다!!**

- 토큰은 이미 클라이언트에 있고
- 서버는 그 토큰을 기억 하지 않기 때문이다.

**결과**

👉 **토큰이 만료(exp)될 때까지 계속 사용 가능하다.**

- 로그아웃 후에도 API 요청 가능
- 계정 정지해도 계속 접근 가능
- 권한 변경 반영 안됨
- 토큰 탈취 시 즉시 차단 불가

---

## 해결 방법

### 1️⃣ Access Token의 수명을 짧게

15분 / 30분 등 짧게 설정하기

- 탈취 위험이 줄어든다.
- 재발급이 필요하다.

---

### 2️⃣ Refresh Token 도입

> **짧은 Access + 긴 Refresh**

```jsx
Access Token → 짧게
Refresh Token → 길게
```

- Access 만료 → Refresh로 재발급

---

### 3️⃣ Refresh Token 서버 저장

> Refresh는 Stateful로 관리하기

- DB/Redis 저장
- 로그아웃 시 삭제 가능
- 즉시 차단 가능

---

### 4️⃣ 블랙리스트 방식

> **“금지 토큰” 목록 관리**

- 로그아웃 시 토큰 저장
- 요청마다 조회
- 특정 토큰 차단 가능
- 서버 상태 관리 필요 (비효율)

---

### 5️⃣ Token Version 방식

> **사용자 기준으로 전체 토큰 무효화**

```jsx
{
  "userId": 1,
  "tokenVersion": 3
}
```

- DB version ≠ 토큰 version → 차단

---

### 6️⃣ 중요한 요청일 경우에는 DB 재검증

예:

- 관리자 기능
- 결제

👉 DB에서 권한 다시 확인

---

# 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리를 간단히 설명해주세요. 🍠

## 하이브리드 방식이란?

👉 **토큰 방식의 장점**과 **세션 방식의 장점**을 함께 가져가려는 인증 구조

- 클라이언트는 **JWT 같은 토큰**을 들고 다닌다.
- 서버는 그 토큰과 관련된 **메타데이터를 저장소(DB, Redis 등**)에 관리한다.

<aside>
💡

**인증 자체는 JWT처럼 처리하되,
제어와 무효화는 세션처럼 서버가 일부 개입하는 구조!**

</aside>

---

## 저장소 메타데이터란?

👉 토큰 자체와 별개로 서버가 저장하는 **부가 관리 정보**

```json
{
  "userId": 1,
  "tokenVersion": 3,
  "status": "ACTIVE",
  "currentRefreshTokenId": "rt_abc123",
  "lastLoginAt": "2026-04-08T12:00:00Z",
  "deviceId": "ios-001"
}
```

이런 데이터는 JWT안에 직접 넣기보다
DB나 Redis에서 관리하는 편이 더 유리할 수 있다.

✔ **이유**

- 자주 바뀔 수 있고
- 즉시 반영이 필요하고
- 보안 정책과 연결되며
- 전체 토큰 제어에 중요하기 때문

---

## 기본 구조

### 클라이언트

- Access Token (JWT)
- Refresh Token (또는 세션성 식별자)

### 서버

- JWT 검증 로직
- DB / Redis 등 저장소
- 사용자 상태 / 토큰 메타데이터 관리

### 저장소에 보관할 수 있는 것

- Refresh Token 해시
- Token Version
- Blacklist
- Device/Session ID
- 사용자 활성 상태
- 강제 로그아웃 여부
- 마지막 사용 시간

---

## 대표적인 하이브리드 구조

> **Access Token은 JWT로 짧게 사용하고
> Refresh Token은 서버 저장소에서 관리한다.**

- Access Token = Stateless
- Refresh Token = Stateful

---

### Access Token

Access Token은 모든 API 요청에 자주 붙는다.

예:

- `/api/me`
- `/api/post`
- `/api/comments`
- `/api/orders`

이 요청마다 매번 DB나 Redis를 크게 조회하면 성능상 부담이 생길 수 있다.

그래서 Access Token은 보통:

- 짧게 발급
- JWT로 서명하여
- 빠르게 검증만 하도록 만든다

<aside>
💡

Access Token은 **빠른 인증 처리**에 초점을 둔다.

</aside>

---

### Refresh Token

Refresh Token은 모든 요청마다 쓰는 것이 아니라 **재발급 시점에만** 사용된다.

즉, Access Token보다 요청 빈도가 낮아서
서버 저장소에 저장하고 관리해도 부담이 상대적으로 적다.

✔ **장점**

- 로그아웃 시 삭제 가능
- 탈취 의심 시 차단 가능
- 특정 디바이스만 종료 가능
- 토큰 재발급 제한 가능
- 회전(rotation) 정책 적용 가능

<aside>
💡

Refresh Token은 **제어와 보안**에 초점을 둔다.

</aside>

---

## 저장소에 넣는 대표적인 메타데이터

### 1️⃣ Refresh Token 자체 또는 해시값

재발급용 토큰 유효성 확인에 사용한다.

보통 원문 그대로 저장하기보다 **해시로 저장**하는 경우가 많다.

---

### 2️⃣ Token Version

사용자 전체 토큰 무효화에 사용

```json
{
  "userId": 1,
  "tokenVersion": 5
}
```

JWT 안에도 `tokenVersion` 을 넣고, DB의 현재 값과 비교해 다르면 차단 가능

---

### 3️⃣ Blacklist

이미 로그아웃되거나 탈취 의심되는 토큰을 차단할 때 사용

예:

- access token jti
- 특정 refresh token id

---

### 4️⃣ JTI (JWT ID)

JWT마다 고유 ID를 넣는 방식

```json
{
  "sub": "1",
  "jti": "jwt-abc-123"
}
```

- 블랙리스트 관리
- 중복 사용 탐지
- 감사 로그 추적

등이 가능

---

### 5️⃣ Device ID / Session ID

기기별 로그인 관리를 위해 사용

예:

- iPhone 로그인
- Chrome 로그인
- iPad 로그인

이런 세션을 각각 구분해서:

- 특정 기기만 로그아웃
- 새 기기 로그인 탐지
- 보안 알림 발송
  등이 가능하다.

---

### 6️⃣ 사용자 상태

예:

- ACTIVE
- SUSPENDED
- DELETED
- LOCKED

JWT가 유효하더라도 사용자 상태가 정지(SUSPENDED)면 차단할 수 있다.

---

### 7️⃣ 마지막 사용 시간

토큰/세션 활동 추적에 사용한다.

예:

- 장시간 미사용 세션 정리
- 이상 활동 탐지
- 관리 콘솔 표시

---

## 하이브리드 방식에서 자주 사용하는 기법들

### 1️⃣ Refresh Token Rotation

👉 재발급할 때마다 refresh token을 새로 바꾸는 방식

✔ **흐름**:

1. 기존 refresh token으로 요청
2. 서버가 검증
3. 새 refresh token 발급
4. 기존 refresh token 폐기

✔ **장점**

- 토큰 재사용 공격 탐지에 유리하다
- 탈취된 오래된 refresh token 악용을 줄일 수 있다

---

### 2️⃣ Token Versioning

👉 사용자 단위 전체 무효화 전략

예:

- 비밀번호 변경
- 관리자 강제 로그아웃
- 계정 보안 문제 발생

이때 DB의 `tokenVersion`을 증가시키면 이전 버전 토큰은 전부 무효 처리할 수 있다.

---

### 3️⃣ JTI + Blacklist

👉 특정 access token이나 refresh token만 정밀하게 차단하는 방식

예:

- 특정 장치의 토큰만 무효화
- 탈취 의심 토큰만 차단
- 로그아웃된 토큰 차단

---

### 4️⃣ Sliding Session 개념 접목

👉 세션처럼 사용 중이면 유효 시간을 연장하는 정책을 일부 접목할 수도 있다.

예:

- 최근 15분 내 활동 있으면 refresh 연장
- 장시간 미사용 시 만료

---

## 하이브리드 방식의 장점

### 1️⃣ JWT의 확장성을 어느 정도 유지할 수 있다

Access Token 검증은 비교적 가볍게 처리하고, 재발급/정책 관리는 저장소에서 처리하므로

분산 환경에서도 유연하게 운영할 수 있다.

---

### 2️⃣ 즉시 무효화 문제를 완화할 수 있다

JWT 단독 구조보다 훨씬 제어가 쉬워진다.

예:

- 로그아웃 시 refresh token 제거
- 블랙리스트로 access token 차단
- tokenVersion 증가로 전체 토큰 폐기

---

### 3️⃣ 기기별 관리가 가능하다

메타데이터에 device/session 정보를 저장하면 사용자별 로그인 목록 관리가 가능하다.

예:

- "내 계정에 로그인된 기기 보기"
- "이 기기에서 로그아웃"
- "새 로그인 알림"

이런 기능을 구현하기 쉽다.

---

### 4️⃣ 권한 변경/계정 상태 반영이 쉬워진다

JWT만 믿지 않고 DB/Redis 메타데이터를 함께 보면 사용자 상태 변경을 더 잘 반영할 수 있다.

예:

- 계정 정지 즉시 차단
- 관리자 권한 변경 반영
- 비밀번호 변경 후 전체 토큰 종료

---

### 5️⃣ 보안 사고 대응력이 높아진다

토큰 탈취, 의심 로그인, 재사용 공격 등에서 JWT 단독 방식보다 대응이 좋다.

---

## 하이브리드 방식의 단점

### 1️⃣ 구조가 복잡해진다

JWT 단독보다 훨씬 고려할 것이 많다.

예:

- 어떤 메타데이터를 저장할지
- DB vs Redis 선택
- 블랙리스트 유지 시간
- refresh rotation 정책
- version 전략
- 디바이스 식별 정책

👉 설계 복잡도가 올라간다.

---

### 2️⃣ 완전한 Stateless 장점은 줄어든다

저장소 조회가 다시 들어가므로 JWT의 "서버 저장소가 거의 필요 없음" 장점은 일부 줄어든다.

---

### 3️⃣ 저장소 장애에 대한 고려가 필요하다

Redis/DB가 핵심 인증 흐름에 들어오면 그 저장소의 장애가 인증 시스템에 큰 영향을 줄 수 있다.

---

### 4️⃣ 운영 비용이 증가한다

- 저장소 운영
- 만료 데이터 정리
- 보안 로그 관리
- 블랙리스트 TTL 관리
- 토큰 회전 정책 운영

등으로 비용이 늘어날 수 있다.

---

# HTTP는 왜 무상태(Stateless)로 설계되었나요? 🍠

> HTTP는 각 요청을 독립적으로 처리하기 위해 무상태로 설계되었다.

✔ **핵심 이유**

1️⃣ **확장성**

- 서버가 다른 상태를 젖아하지 않음
- 어떤 서버든 요청 처리 가능
- 로드밸런싱 쉬움

2️⃣ **단순성**

- 서버가 상태 관리 ❌
- 구조가 단순
- 유지보수 쉬움

3️⃣ **장애 대응**

- 특정 서버 죽어도 다른 서버로 처리 가능!
- 상태 의존성 ❌

4️⃣ **캐싱 효율**

- 동일 요청 → 동일 응답
- 캐싱 활용 가능

---

<br><br>

# HTTP의 무상태성이 주는 장점과 단점을 각각 정리해주세요. 🍠

## **장점**

| 항목      | 설명                |
| --------- | ------------------- |
| 확장성    | 서버 수평 확장 쉬움 |
| 단순성    | 상태 관리 없음      |
| 장애 대응 | 서버 교체 쉬움      |
| 캐싱      | 효율적인 캐싱 가능  |

## **단점**

| 항목                  | 설명                    |
| --------------------- | ----------------------- |
| 상태 유지 불가        | 로그인 상태 유지 어려움 |
| 매 요청마다 인증 필요 | 토큰/쿠키 필요          |
| 사용자 경험 저하      | 계속 인증해야 함        |

# 쿠키의 Domain 디렉티브에 대해 정리해주세요. 🍠

> 쿠키를 어떤 도메인에서 전송할지 결정하는 속성

✔ **예시**

```json
Set-Cookie: sessionId=abc; Domain=example.com
```

👉 적용 범위:

- example.com
- sub.example.com

✔ **특징**

- 명시하지 않으면 → 현재 도메인만 적용
- 지정하면 → 하위 도메인까지 포함

<br>

> 💡 너무 넓게 설정하면 보안 위험이 증가한다!

# 쿠키의 Path 디렉티브에 대해 정리해주세요. 🍠

> 쿠키를 **어떤 경로(URL)**에서 전송할지 결정하는 것

✔ **예시**

```json
Set-Cookie: sessionId=abc; Path=/admin
```

👉 적용 범위:

- /admin
- /admin/users

❌ 적용 안되는 범위:

- /home 등

# 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠

## **세션 쿠키**

- 브라우저 종료 시 삭제
- 서버 세션과 함께 사용

```json
Set-Cookie: sessionId=abc
```

## **영속 쿠키**

- 만료 시간 존재
- 브라우저 종료 후에도 유지

```json
Set-Cookie: userId=123; Max-Age=3600
```

## ✔ **비교**

| 구분      | 세션 쿠키         | 영속 쿠키              |
| --------- | ----------------- | ---------------------- |
| 저장 기간 | 브라우저 종료까지 | 만료 시간까지          |
| 사용 목적 | 로그인 상태       | 자동 로그인, 설정 저장 |

# 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나요? 🍠

## **HttpOnly**

```json
Set-Cookie: token=abc; HttpOnly
```

### **역할**

- JavaScript 접근 차단

### **방어**

- XSS 공격

---

## **Secure**

```json
Set-Cookie: token=abc; Secure
```

### **역할**

- HTTPS에서만 전송

### **방어**

- 네트워크 도청 (MITM 공격)

---

## **SameSite**

```json
Set-Cookie: token=abc; SameSite=Strict
```

### **역할**

- 다른 사이트 요청에서 쿠키 전송 제한

### **방어**

- CSRF 공격

---

## **정리**

| 속성     | 역할                | 방어 |
| -------- | ------------------- | ---- |
| HttpOnly | JS 접근 차단        | XSS  |
| Secure   | HTTPS만 전송        | 도청 |
| SameSite | 타 사이트 요청 제한 | CSRF |

---

<br><br>

# 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)을 정리해주세요. 🍠

## 1️⃣ **용량 제한**

- 보통 4KB 제한
- 많은 데이터 저장 불가

## 2️⃣ **보안 문제**

- 클라이언트에 저장됨
- 탈취 위험 존재
- 조작 가능성 있음

## 3️⃣ **네트워크 부담**

- 매 요청마다 자동 전송
- 트래픽 증가

## 4️⃣ **도메인 제한**

- 다른 도메인과 공유 불가
- same-origin 정책 영향

---

<br><br>

# 쿠키만으로 상태 관리를 해결할 수 없는 이유는 무엇인가요? 🍠

> **쿠키는 단순 저장소일 뿐, 인증/보안을 보장하지 않는다**

## **문제점**

### 1️⃣ **위변조 가능**

- 클라이언트에서 수정 가능

### 2️⃣ **민감 정보 저장 위험**

- 노출 가능
- 탈취 위험

### 3️⃣ **서버 제어 불가**

- 쿠키 자체로는 무효화 어려움

### 4️⃣ **인증 신뢰 부족**

- "이 사용자가 진짜 맞는지" 보장 못함

# XSS(Cross-SIte Scripting) 정리해보기! (XSS공격과 방어 전략)🍠

## XSS란?

**👉 웹 페이지에 악성 스크립트를 삽입해서사용자의 브라우저에서 실행되도록 만드는 공격**

- 공격자는 **JS 코드 삽입**
- 피해자는 **그 코드를 실행**
- 실행 위치 = **사용자의 브라우저**

✔ **예시**

```html
<script>
  alert("해킹");
</script>
```

- 이게 페이지에 들어가면 사용자가 접속할 때 실행됨

---

## XSS의 위험성

### 공격자가 할 수 있는 것

- 쿠키 탈취
- JWT 탈취
- 사용자 계정 탈취
- 요청 위조 (API 호출)
- 피싱 UI 생성
- 관리자 권한 악용

## XSS 공격 종류

### 1️⃣ Stored XSS (저장형)

**👉 서버(DB)에 저장된 후 실행**

#### 흐름

```
공격자 → 게시글에 스크립트 저장
→ 서버 DB 저장
→ 사용자 조회
→ 스크립트 실행
```

#### ✔ 특징

- 가장 위험
- 많은 사용자에게 영향

---

### 2️⃣ Reflected XSS (반사형)

**👉 요청 → 응답에 바로 반사**

#### 예

```
https://site.com?q=<script>alert(1)</script>
```

#### 흐름

```
URL → 서버 → 그대로 응답 → 실행
```

#### ✔ 특징

- URL 기반 공격
- 클릭 유도 필요

---

### 3️⃣ DOM-based XSS

**👉 서버가 아니라 JS에서 발생**

#### 예

```jsx
document.innerHTML = location.search;
```

- 입력값 그대로 DOM에 삽입

---

## XSS 방어 전략

### 1️⃣ 입력값 필터링 (Input Validation)

**👉 위험한 입력 차단**

#### 예

- `<script>`
- `onerror=`
- `javascript:`

#### 한계

- 완벽하게 막기 어려움

---

### 2️⃣ 출력 시 이스케이프 (Output Escaping)

**👉 HTML로 해석되지 않게 변환**

#### 예

```html
<script>
  alert(1);
</script>
```

변환하면:

```html
&lt;script&gt;alert(1)&lt;/script&gt;
```

#### ✔ 방법

- `textContent` 사용
- `innerHTML` 사용 금지
- 템플릿 엔진 escape 사용

---

### 3️⃣ DOM 조작 시 안전하게

❌ 위험

```jsx
element.innerHTML = userInput;
```

✅ 안전

```jsx
element.textContent = userInput;
```

---

### 4️⃣ CSP (Content Security Policy)

**👉 허용된 스크립트만 실행**

#### 예

```
Content-Security-Policy: script-src 'self'
```

#### ✔ 효과

- 외부 스크립트 차단
- inline script 차단

---

### 5️⃣ HttpOnly 쿠키

**👉 JS에서 쿠키 접근 차단**

```
Set-Cookie: token=abc; HttpOnly
```

#### ✔ 효과

- XSS로 쿠키 탈취 방지

---

### 6️⃣ Secure 쿠키

👉 HTTPS에서만 전송

---

### 7️⃣ SameSite 쿠키

👉 CSRF 방어 (XSS 보조)

---

### 8️⃣ 라이브러리 / 프레임워크 사용

React, Vue 등은 기본적으로:

- 자동 escaping 제공
- XSS 위험 줄여줌

---

# 세션 하이재킹은 무엇인가요? 🍠

👉 **사용자의 세션 정보를 탈취해서 해당 사용자로 가장해 서버에 접근하는 공격**

### ✔ **흐름**

```json
사용자 로그인 → 세션ID 발급
↓
공격자가 세션ID 탈취
↓
공격자가 해당 세션으로 요청
↓
서버: 정상 사용자로 인식 ❌
```

---

### 공격 방법

- XSS → 쿠키 탈취
- 네트워크 스니핑 (HTTP)
- 세션 ID 예측
- CSRF와 결합 공격

---

### 결과

- 계정 탈취
- 개인정보 유출
- 관리자 권한 악용

---

### 방어 방법

#### 1️⃣ HTTPS 사용

- 네트워크 탈취 방지

<aside>
💡

**HTTPS만으로는 부족하다**

- 네트워크 스니핑은 막음
- XSS 공격은 막지 못함
- HttpOnly + Secure + SameSite 조합 필수
</aside>

---

#### 2️⃣ HttpOnly 쿠키

- JS에서 세션ID 접근 차단
- XSS 공격으로도 쿠키 탈취 불가
- 가장 기본적이고 필수적인 방어법

---

#### 3️⃣ Secure 쿠키

- HTTPS에서만 전송
- 네트워크 스니핑 공격 방어
- 프로덕션 환경에서 필수

---

#### 4️⃣ 세션 만료 시간 설정

- 오래 유지되지 않도록 탈취된 쿠키의 유효 시간 제한

---

#### 5️⃣ 로그인 시 세션 재발급 (Session Fixation 방지)

# CSRF 공격 정리해보기 (CSRF 공격과 방어 전략) 🍠

## CSRF란?

👉 사용자가 의도하지 않은 요청을 다른 사이트를 통해 서버에 보내도록 유도하는 공격

---

### ✔ **예시**

상황:

- 사용자가 은행 로그인 상태
- 공격자가 만든 사이트 방문

공격 코드:

```html
<img src="https://bank.com/transfer?amount=1000&to=hacker" />
```

흐름:

```html
사용자 로그인 상태 ↓ 악성 사이트 방문 ↓ 자동 요청 발생 ↓ 브라우저가 쿠키 자동
포함 ↓ 서버: 정상 요청으로 착각
```

<aside>
💡

**핵심 원인: “브라우저의 쿠키 자동 전송”**

</aside>

---

## CSRF 방어 전략

### 1️⃣ CSRF Token

👉 **서버가 생성한 랜덤 토큰을 요청에 포함시켜 전송**

```html
1. 서버 → 토큰 발급 2. 클라이언트 → 요청에 포함 3. 서버 → 토큰 검증
```

✔ **특징**

- 공격자는 토큰을 알 수 없음
- 위조 요청 차단 가능

✔ **장점**

| 항목           | 설명                      |
| -------------- | ------------------------- |
| 강력한 보안    | 위조 요청 거의 차단       |
| 서버 제어 가능 | 검증 로직 확실            |
| 범용성         | 대부분 환경에서 사용 가능 |

✔ **단점**

| 항목             | 설명                |
| ---------------- | ------------------- |
| 구현 복잡        | 토큰 생성/검증 필요 |
| 상태 관리 필요   | 서버 저장 필요      |
| SPA에서 번거로움 | API 구조에서 불편   |
| 토큰 노출 가능성 | XSS에 취약          |

---

### 2️⃣ SameSite 쿠키

👉 **다른 사이트 요청에서 쿠키 전송 제한**

✔ **옵션**

| 값     | 설명               |
| ------ | ------------------ |
| Strict | 완전 차단          |
| Lax    | 일부 허용          |
| None   | 허용 (Secure 필요) |

---

### 3️⃣ Referer / Origin 검증

👉 **요청이 어디서 왔는지 확인**

```html
Origin: https://example.com
```

✔ **특징**

- 서버에서 도메인 체크
- 위조 요청 차단 가능

---

### 4️⃣ 쿠키 대신 Authorization 헤더 사용

👉 **쿠키 자동 전송 문제 제거**

```html
Authorization: Bearer token
```

✔ **특징**

- 브라우저 자동 전송 ❌
- CSRF 공격이 어려워짐

---

### 5️⃣ 사용자 추가 인증

- 비밀번호 재입력
- 2FA
- 이메일 인증

# 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건은 무엇인가요? 🍠

## 1️⃣ **도메인 일치**

- 요청 도메인이 쿠키 Domain과 일치해야 한다.

```html
example.com → example.com ✔ sub.example.com → example.com ✔
```

## 2️⃣ **경로 일치**

```html
Path=/admin → /admin ✔ → /admin/users ✔ → /home ❌
```

## 3️⃣ **Secure 조건**

- `Secure` 쿠키 → HTTPS에서만 전송

## 4️⃣ **SameSite 정책**

| 옵션   | 전송 여부                        |
| ------ | -------------------------------- |
| Strict | 같은 사이트만                    |
| Lax    | 일부 허용                        |
| None   | 크로스 사이트 허용 (Secure 필수) |

## 5️⃣ **유효기간**

- 만료되지 않은 쿠키만 전송한다.

# 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까요? 🍠

## **서버 설정**

```html
Access-Control-Allow-Credentials: true Access-Control-Allow-Origin:
https://client.com
```

- 주의할 점 ⚠️: `*` 사용하면 안된다. (credentials랑 같이 못씀!)

## **클라이언트 설정**

- `fetch`
  ```jsx
  fetch(url, {
    credentials: "include",
  });
  ```
- `axios`
  ```jsx
  axios.get(url, {
    withCredentials: true,
  });
  ```

## **쿠키 설정**

```html
Set-Cookie: token=abc; SameSite=None; Secure
```

# 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)을 정리해주세요. 🍠

- API 중심 구조에 적합

# Single Page Application, Mobile Application, Server Side Rendering Application 에서 여러분들이 생각하는 적합한 인증 전략은 무엇이라고 생각하시나요? 🍠

## **SPA**

> JWT 기반 인증 + Refresh Token 관리

SPA에서는 API 중심 구조와 잘 맞는 JWT 기반 인증이 적합하며, Access Token은 짧게, Refresh Token은 서버 제어 가능한 방식으로 함께 운영하는 전략이 적합하다.

## **Mobile Application**

> JWT 기반 인증 (Access Token + Refresh Token)

모바일 앱은 브라우저 쿠키보다 헤더 기반 토큰 인증이 자연스럽기 때문에, JWT 기반 인증과 운영체제의 보안 저장소를 함께 사용하는 전략이 적합하다.

## **SSR Application**

> 세션 기반 인증 또는 HttpOnly Cookie 기반 인증

SSR 애플리케이션은 브라우저 쿠키와 서버 렌더링 구조가 자연스럽게 맞물리기 때문에, 세션 기반 인증 또는 HttpOnly Cookie 중심 인증 전략이 적합하다.
