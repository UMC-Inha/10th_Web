- 세션 방식과 토큰 방식의 가장 큰 차이는 무엇인가요? 🍠
    
    세션과 토큰의 가장 큰 차이는 사용자의 상태 정보를 어디에 저장하냐의 차이이며
    세션은 사용자 정보를 서버에 저장하고 클라이언트에는 정보를 찾을 수 있는 세션 id만 전달
    토큰은 사용자 정보를 토큰 자체에 포함시키는 self-contained 방식으로 저장함
    (토큰에 사용자 정보가 포함됨)
    
    세션 저장 방식은 크게 메모리 저장, 인메모리 데이터 베이스, 관계형 데이터 베이스가 있음
    
    - 메모리: 가장 빠르다는 장점이 있지만 서버 재 시작시 모든 세션이 사라짐
    - 인메모리 DB(분산 환경): 속도와 영속성을 모두 얻을 수 있습니다. 분산 환경에서 여러 서버가 세션을 공유할 수 있고, 서버 재시작에도 세션이 유지됨
    - 관계형 DB: 가장 안정적이지만 다른 방식에 비해 성능은 떨어짐
- 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징을 정리해주세요. 🍠
    - Sticky Session: 같은 사용자 요청을 항상 같은 서버로 보내는 방식으로 구현이 간단하다는 장점이 있지만 서버 장애시 해당 서버에 모든 세션이 사라질 수 있음 → 서버 다운시 다른 서버로 요청을 보내면 해당 서버에는 정보가 저장되지 않았기에 문제 발생
    - Session Replication: 모든 서버가 세션 정보를 공유 하는 방식으로 서버간 네트워크 통신이 많아져 성능 저하 이슈가 있을 수 있음 → 사용자 로그인시 세션을 다른 서버로 복사하여 전달
    - Centralized Session Store: 별도 저장소에 모든 세션을 보관하여 서버에서 로드하여 사용하는 방식(가장 널리 사용)
- JWT(Json Web Token)의 장점과 단점을 각각 설명해주세요. 🍠
    
    JWT의 장점:
    토큰은 사용자 정보를 토큰 자체에 저장 하고 서버 비밀 키로 서명하여 비밀 키로 검증하면 유효성을 확인 할 수 있기에 서버가 많아도 독립적으로 검증이 가능하여 확장성이 좋다
    
    단점:
    한번 발급한 토큰이 보안 위험이 생겼을때 즉시 대처하기가 어렵다 → 만료시간을 짧게두거나 하는등의 추가 조치 필요
    
- JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법은 무엇인가요? 🍠
    
    토큰은 만료 이전까지 비밀키로 검증 하는 방식으로 작동하지만 토큰에 문제가 생겼을때 해당 토큰을 즉시 만료시킬 방법이 없기 때문에 즉시 무효화 문제가 생기며 해당 문제를 해결하기 위해 JWT를 사용하며 저장소에 메타데이터를 저장해 필요시 해당 토큰을 무효화하는 방식을 사용해 해결함
    
- 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리를 간단히 설명해주세요. 🍠
    
    먼저 사용자가 로그인을 하면 사용자의 토큰의 메타데이터(ex:id)를 서버에 저장한다 → 
    사용자가 서버에 특정 요청을 하면 해당 토큰에 해당하는 메타데이터를 서버에서 찾아 검증한다 → 사용자가 로그아웃하면 저장된 메타데이터를 블랙리스트(사용 안함)으로 등록한다 → 이 상태에서 새로운 요청이 들어오면 블랙리스트에 메타데이터가 있기에 검증시 탈취된것, 아닌것을 검증할 수 있다

- HTTP는 왜 무상태(Stateless)로 설계되었나요? 🍠
    
    HTTP가 처음으로 설계되었을때 사용자들의 요청을 메모리에 저장하는 상태 기반이면 메모리가 부족할 것이기에 확장성을 위해서, 추가적으로 클라이언트 별 상태를 관리하지 않아도 되는 단순성을 위하여 무상태를 기반으로 설계되었음
    
- HTTP의 무상태성이 주는 장점과 단점을 각각 정리해주세요. 🍠
    
    장점: 서버가 클라이언트 별 상태를 관리하지 않으므로 서버를 무한대로 확장하기 용의함
    
    단점: 클라이언트의 상태를 기억하지 않으므로 자기 정보를 계속해서 서버로 보내야함
    
- 쿠키의 Domain 디렉티브에 대해 정리해주세요. 🍠
    
    쿠키가 사용될 도메인의 범위를 지정하는것 
    
    ex: naver.com으로 도메인의 범위를 지정한다면 .mail.naver.com 같은 서브 도메인에서도 이용가능하게 함
    
- 쿠키의 Path 디렉티브에 대해 정리해주세요. 🍠
    
    쿠키가 전송될 경로를 지정함 
    
    ex: “/user”이라는 경로에 쿠키를 보내도록 지정했다면 “/items”같은 불필요한 경로로 쿠키가 가는것을 막아
    
- 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠
    
    세션 쿠키: 만료 시간을 지정하지 않아 해당 브라우저를 종료 하게 되면 삭제되는 쿠키(은행 로그인 등 한번 창을 닫으면 다시 시도 해야 하는 것)
    
    영속 쿠키: 만료 시간을 지정하여 브라우저를 종료해도 설정한 시간 동안은 삭제되지 않고 존재하는 쿠키 (자동 로그인 등 한번 창을 닫아도 일정 시간 유지되는 것)
    
- 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나요? 🍠
    
    HttpOnly: JS가 쿠키에 접근하는것을 막아 해커가 악성 스크립트를 삽입해도 쿠키를 탈취하지 못하게 함
    
    Secure: Https일 때만 쿠키를 전송하여 공용 wifi같은 안전하지 않은 상황에서 http 통신을 가로채 쿠키를 도청하는 일을 방지함 
    
    SameSite: 다른 사이트에서 내 사이트로 오는 요청에 쿠키를 보낼지 여부를 설정함 
    strict: 같은 사이트에서만 쿠키전송( 현재 사이트 내부에서만)
    lax: get 같은 안전한 메서드로 요청이 들어왔을 때 만 쿠키를 전송
    None: 아무대나 전송(Secure 필수)
    
- 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)을 정리해주세요. 🍠
    
    용량: 크기가 4KB이며 브라우저 마다 쿠키의 갯수도 한정되어 많은 정보 저장엔 부적합
    
    보안: 쿠키는 클라이언트에 저장되므로 사용자가 조작이 가능함 
    
    네트워크: 도메인의 요청에 자동적으로 포함되므로 쿠키 정보가 필요 없는 요청에도 포함되어 오버헤드 발생 가능
    
    도메인: 쿠키는 설정된 도메인에서만 공유가 접근이 가능하며 다른 사이트에서는 접근 할 수 없어 다양한 도메인에 걸쳐 서비스가 분산되어 있을 경우 번거로움(계속 전송 받아야함)
    
- 쿠키만으로 상태 관리를 해결할 수 없는 이유는 무엇인가요? 🍠
    
    사용자가 쿠키를 조작하기 쉽다는 치명적인 보안 위험이 있고 
    
    용량이 작아 정보 저장에 한계가 있기 때문에

- XSS 공격은 무엇인가요?
    - **Stored XSS (저장형):** 게시판 댓글이나 프로필 정보같은 유저가 무언가를 입력할 수 있는 영역에 스크립트 입력하여 저장해두는 방식입니다. 이렇게 하면 해당 공격을 당한 사이트의 **모든 사용자**가 공격 대상이 됩니다.
    - **Reflected XSS (반사형):** 검색어나 URL 파라미터에 스크립트를 포함시켜 링크를 클릭하게 유도하는 방식입니다.(검색 결과를 스크립트로 변경) 링크를 클릭한 **특정 사용자**에게 스크립트가 실행됩니다. (서버를 거쳐 다른 사용자에게 그대로 노출)
    - **DOM-based XSS:** 서버를 거치지 않고 브라우저의 자바스크립트가 URL의 해시(#) 등을 잘못 처리할 때 발생합니다(해커가 #뒤에 스크립트를 삽입한 링크를 만들어 올려둔것을 클릭하면 해킹).
- XSS 방어 전략에 대해 정리해주세요.
    
    출력 이스케이핑: 사용자가 입력 중 HTML로 해석될 수 있는 특수문자를 단순 문자로 바꿔버리는 것 → 브라우저는 `<script>`를 실행하지 않고 화면에 문자 그대로 출력만 합니다.
    
    csp(콘텐츠 보안 정책) :허용된, 지정된 출처에서 오는 스크립트만 시행하게함
    
    새니타이즈: 허용된 태그를 제외한 나머지를 제거하여 스크립트 실행을 방지

- 세션 하이재킹은 무엇인가요? 🍠
    
    쿠키에 담긴 세션 id를 탈취하여 해커가 자신의 브라우저에 넣고 서버에 요청을 하면 탈취당한 계정을 자기 계정 인양 사용이 가능
    
- XSS 공격을 차단하는 방법은 무엇인가요? 🍠
    - **새니타이즈 & 이스케이핑:** 사용자가 입력한 `<script>` 같은 태그를 문자열(`&lt;script&gt;`)로 치환하여 실행 방지
    - **HttpOnly 쿠키 설정:** 자바스크립트가 쿠키를 읽지 못하게 하여, XSS가 발생하더라도 세션 ID를 탈취당하지 않게 방어
    - **CSP (콘텐츠 보안 정책):** 웹사이트에서 실행 가능한 스크립트의 출처를 제한하여, 해커가 외부에서 심은 스크립트 실행을 방지
    - **입력 값 검증:** 서버단에서 정해진 형식(예: 숫자만 입력 가능)이 아닌 데이터는 아예 받지 않음.
- 쿠키의 다양한 옵션들에 대해서 정리해주세요. 🍠

    | **HttpOnly** | 자바스크립트(`document.cookie`) 접근 금지 | **XSS**를 통한 쿠키 탈취 방어 |
    | **Secure** | **HTTPS** 통신일 때만 쿠키를 전송함 | 네트워크 구간에서의 **스니핑(도청)** 방어 |
    | **SameSite** | 다른 사이트에서 우리 사이트로 오는 요청에 쿠키를 실을지 결정 | **CSRF** 공격 방어 (`Strict`, `Lax`, `None`) |
    | **Domain** | 쿠키가 유효한 도메인 범위 설정 | 서브 도메인 간의 쿠키 공유 및 격리 제어 |
    | **Path** | 특정 경로(URL)에서만 쿠키가 전송되도록 제한 | 불필요한 경로로의 쿠키 유출 방지 |
    | **Max-Age / Expires** | 쿠키의 유효 기간(수명)을 설정 | 기간이 지나면 자동 삭제 (영속 쿠키, 세션 쿠키 여부 결정) |

- CSRF는 무엇인가요? 🍠
    
    사용자가 이미 로그인된 상태를 이용해 악성 사이트를 통해 인증된 사용자 쿠키를 사용하여 해커가 원하는 동작(비밀번호 변경, 결제, 글쓰기 등)을 특정 서버에 요청하게 만드는 공격
    
- CSRF 방어 전략에 대해 정리해주세요. 🍠
    
    **SameSite 쿠키 속성:**  쿠키가 다른 사이트에서 전송되는 것을 막음
    
    **Referer / Origin 검증:** 요청이 어디서 왔는지(어느 사이트에서 보냈는지) 확인하여, 허용된 도메인이 아니면 거절
    
    **CSRF 토큰:**  서버가 발행한 특별한 토큰을 HTML내부에 삽입하고 해당 토큰을 검증하는 방법
    
- CSRF 토큰의 장점과 단점에 대해 정리해주세요. 🍠
    - **장점:**
        - **강력한 보안:** 해커는 사용자의 쿠키는 브라우저가 자동으로 보내주니 이용할 수 있지만, 서버가 발행한 고유한 **토큰 값**은 알 수 없으므로 위조가 불가능
        - **정교한 제어:** 특정 동작(결제 등)에 대해서만 선택적으로 강력한 보안을 적용 가능
    - **단점:**
        - **서버 부하:** 서버가 매번 토큰을 생성하고 저장/검증해야 하므로 자원 사용
        - **상태 관리의 복잡성:** 페이지가 바뀔 때마다 토큰을 갱신, 관리 해야 하므로 관리가 까다로움
- CAPTCHA는 무엇인가요? 🍠
    
    사람과 컴퓨터(봇)를 구별하기 위한 자동 테스트
    
    - **역할:** 찌그러진 글자 읽기, 신호등 찾기 등을 통해 현재 요청을 보내는 주체가 '진짜 사람'인지 확인
    - **CSRF 방어에서의 활용:** 중요한 요청(비밀번호 변경 등) 직전에 CAPTCHA를 넣으면, 해커가 몰래 스크립트로 요청을 보내더라도 CAPTCHA를 통과하지 못해 공격이 차단됨

- 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건은 무엇인가요? 🍠
    - **도메인 일치:** 요청을 보내는 목적지 도메인이 쿠키의 `Domain` 설정과 일치
    - **경로 일치:** 요청 URL이 쿠키의 `Path` 설정 범위 안에 있어야 함
    - **보안 및 정책:** `Secure` 설정 시 HTTPS여야 하며, `SameSite` 설정에 따라 제삼자 사이트에서 보내는 요청인지 확인.
- 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까요? 🍠
    - **클라이언트(Frontend):** 요청을 보낼 때 `withCredentials: true` 옵션을 켜야 함. (Axios나 Fetch API 등)
    - **서버(Backend):** 1. 응답 헤더에 `Access-Control-Allow-Credentials: true`를 포함해야 함
    2. `Access-Control-Allow-Origin`에 와일드카드()가 아닌 **정확한 클라이언트 도메인**을 명시해야 함
    3. 쿠키 설정 시 `SameSite=None`과 `Secure`를 반드시 적용해야 함.
- 쿠키 기반 인증에서 CSRF 공격이 발생할 수 있는 원리는 무엇인가요? 🍠
    - 사용자가 사이트 로그인하여 브라우저에 인증 쿠키가 남음
    - 해커가 만든 악성 사이트에 접속함
    - 사이트의 코드가 특정 사이트로 요청을 자동으로 보냅니다.
    - 브라우저는 인증된 쿠키를 ****실어 보냄
    - 해당 서버는 쿠키가 있으니 정상적인 요청 판독하고 요청을 처리
- 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)을 정리해주세요. 🍠
    - **CSRF 방어:** 브라우저가 자동으로 실어 보내지 않으므로, 해커 사이트에서 요청을 보내도 헤더에 토큰을 담을 방법이 없어 공격이 불가능.
    - **선택적 전송:** 쿠키처럼 무조건 가는 게 아니라, 인증이 필요한 API 요청에만 골라서 보낼 수 있어 효율적
    - **CORS 단순화:** 쿠키를 사용하지 않으면 `Allow-Credentials` 설정 등 복잡한 제약에서 비교적 자유로움.
- 토큰을 클라이언트에 저장할 때 LocalStorage, SessionStorage, 메모리 저장의 장단점을 비교해주세요. 🍠
  
    LocalStorage	장점:브라우저를 꺼도 유지됨, 구현이 매우 쉬움	단점:XSS 공격에 취약 (JS로 읽기 가능)
    SessionStorage	장점:탭을 닫으면 자동 삭제되어 보안성 증가	    단점:탭마다 따로 로그인해야 해서 불편함
    변수(메모리)	장점:XSS로부터 가장 안전 (JS 변수에 저장)	    단점:새로고침하면 로그인 풀림, 사용성 최악

    ### Single Page Application
    
    - **JWT (Header) + Refresh Token (HttpOnly Cookie)**
    - **이유:** CSRF를 차단하면서도, 새로고침 시 쿠키를 통해 토큰을 안전하게 재발급 가.
    
    ### Mobile Application
    
    - **JWT (Header) 전용**
    - **이유:** 모바일 앱은 브라우저처럼 쿠키를 자동으로 관리하는 기능이 약하므로, 앱의 안전한 저장소(Secure Storage)에 토큰을 보관하고 헤더에 직접 실어 보내는 것이 효과적
    
    ### Server Side Rendering
    
    - **전통적인 세션 쿠키** 또는 **HttpOnly 쿠키 기반 토큰**
    - **이유:** 서버에서 페이지를 그릴 때 브라우저의 쿠키를 바로 읽어 인증 상태를 확인할 수 있어 구현이 가장 매끄럽고 빠름

    Zod는 데이터를 받아서 "검문소" 역할을 수행하며 데이터가 들어오면 스키마라는 기준에 맞춰 검사하고, 결과(성공/실패)를 반환함.
    
    TypeScript는 컴파일 타임(코드 작성 중)에만 타입을 체크 하지만 서버에서 받아온 데이터나 사용자가 입력한 값은 런타임(프로그램 실행 중)에 들어오게됨
    
    Zod는 이 간극을 메워 실행 시점에 데이터를 검사하고, 통과하면 그 데이터를 안전하게 사용할 수 있게 도와줌.
    
    ---
    
    ### 핵심 기능
    
    ### **스키마 정의 (Blueprint)**
    
    데이터의 모든 성질과 제약 조건(string이여야 한다 함수여야한다 등)을 선언적으로 기술
    
    TypeScript
    
    `import { z } from "zod";
    
    const UserSchema = z.object({
      username: z.string().min(3),
      age: z.number().optional(),
      email: z.string().email(),
    });`
    
    ### **타입 추론 (Type Inference)**
    
    스키마를 만들면 따로 `interface`를 정의할 필요 없이 타입을 뽑아낼 수 있으며. 스키마가 수정되면 타입도 자동으로 수정됨.
    
    TypeScript
    
    `type User = z.infer<typeof UserSchema>;
    // 자동으로 { username: string; age?: number; email: string; } 타입이 생성됨`
    
    ### **유효성 검사 및 정제 (Validation & Refine)**
    
    단순히 타입만 체크하는 게 아니라, 복잡한 조건(비밀번호 일치 여부 등)을 검사할 수 있습니다.
    
    TypeScript
    
    `const loginSchema = z.string().refine((val) => val !== "admin", {
      message: "admin은 아이디로 사용할 수 없습니다.",
    });`
