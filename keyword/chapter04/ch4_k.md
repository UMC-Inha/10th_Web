- 세션 방식과 토큰 방식의 가장 큰 차이는 무엇인가요? 🍠
    - **세션 방식 (Stateful):** 서버가 기억합니다. 서버 메모리나 DB에 세션 정보를 저장하고, 클라이언트에게는 그 정보의 주소인 '세션 ID'만 줍니다.
    - **토큰 방식 (Stateless):** 클라이언트가 기억합니다. 서버는 사용자의 정보를 꽉꽉 눌러 담은 '토큰(JWT)'을 만들어 클라이언트에게 주고, 자신은 잊어버립니다.
- 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징을 정리해주세요. 🍠
    
    서버가 여러 대일 때 세션을 어떻게 공유할지에 대한 전략입니다.
    
    **Sticky Session :** 특정 사용자를 처음 접속한 서버로만 계속 보냄 구현이 매우 쉬움 특정 서버에 부하가 쏠릴 수 있고, 서버 다운 시 세션 유실
    
    **Session Replication :** 모든 서버가 모든 세션 정보를 복사해서 나눠 가짐 어떤 서버로 가도 세션이 유지됨 서버가 늘어날수록 복사 비용(메모리, 네트워크)이 폭발함
    
    **Centralized Store :** 별도의 저장소(Redis 등)에 세션을 모아둠 **가장 권장됨.** 서버 증설이 자유롭고 데이터 일관성 완벽저장소(Redis)가 죽으면 전체 로그인이 마비됨 (별도 관리 필요)
    
- JWT(Json Web Token)의 장점과 단점을 각각 설명해주세요. 🍠
    
    ### **장점**
    
    - **확장성 갑:** 서버가 상태를 안 들고 있으니 서버를 수백 대 늘려도 상관없습니다.
    - **비용 절감:** DB를 조회해서 "얘 로그인했나?" 물어볼 필요가 없습니다. 토큰 자체만 검증하면 끝납니다.
    - **멀티 서비스:** 한 번의 로그인으로 여러 서비스(도메인)를 이용하기 좋습니다.
    
    ### **단점**
    
    - **한 번 내뱉으면 끝:** 서버가 제어권을 잃습니다. 토큰을 뺏겨도 강제로 로그아웃시키기가 매우 어렵습니다.
    - **데이터 크기:** 세션 ID는 아주 짧지만, JWT는 안에 정보를 담고 있어 데이터가 커집니다. 네트워크 비용이 증가하죠.
    - **보안:** Payload는 누구나 열어볼 수 있는 Base64 인코딩입니다. 민감한 정보(비밀번호 등)를 절대 넣으면 안 됩니다.
- JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법은 무엇인가요? 🍠
    
    **문제가 생기는 이유는 서버가 자기 기억력을 포기한 대가로 제어권을 잃어버린 것입니다.**
    
    ### **블랙리스트**
    
    - 로그아웃하거나 뺏긴 토큰의 ID를 **Redis(메모리 DB)** 같은 곳에 '블랙리스트'로 올려둡니다.  결국 서버가 뭔가를 '저장'하고 '확인'해야 하니, JWT의 최대 장점인 "아무것도 안 봐도 됨"이 살짝 깨집니다.
    
    ### **RT/AT 전략 (짧게 쓰고 자주 바꾸기)**
    
    짧은 입장권(Access Token)과 비교적 긴 신분증(Refresh Token)을 같이 주는 거예요.
    
    - 입장권이 15분 만에 끝나버리니까, 도둑이 훔쳐가도 15분 뒤면 쓰레기가 됩니다.
    - 진짜 사용자는 15분마다 '신분증'을 보여주고 새 입장권을 받으면 돼요.
    - 만약 사고가 터지면? 서버가 들고 있는 그 사용자의 **'신분증'만 파기**해버립니다. 그럼 다음번에 입장권을 새로 못 받게 되니 자연스럽게 차단됩니다.
- 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리를 간단히 설명해주세요. 🍠
    
    "토큰의 편리함"과 "세션의 통제권"을 모두 갖고 싶을 때 쓰는 방식입니다.
    
    - **동작 원리:**
        1. 클라이언트에게는 **JWT**를 줍니다. (서버는 Stateless 유지)
        2. 동시에 서버 저장소(Redis)에 **해당 유저의 상태 메타데이터**를 아주 가볍게 저장합니다. (예: `user_id: 123, status: active, last_login_device: MacBook`)
        3. 요청이 오면 1차로 JWT 서명을 검증하고, 2차로 Redis를 '살짝' 조회해서 이 사용자가 현재 차단 상태인지, 혹은 중복 로그인으로 인해 쫓겨났는지 확인합니다.
    - **특징:** 순수 JWT보다는 DB 조회 비용이 발생하지만, 세션 방식처럼 무거운 데이터를 다루지 않으면서도 **실시간 제어권**을 가질 수 있다는 게 장점입니다.

    - HTTP는 왜 무상태(Stateless)로 설계되었나요? 🍠
    
    HTTP가 무상태로 설계된 근본적인 이유는 서버의 확장성(Scalability)과 **단순성**을 확보하기 위함이다.
    
    - **서버 자원 효율화:** 서버가 클라이언트의 이전 상태를 기억하지 않으므로, 연결을 유지하기 위한 메모리나 연산 자원을 소모할 필요가 없다.
    - **서버 확장성(Scale-out):** 클라이언트의 상태 정보가 서버에 남아있지 않으므로, 요청이 들어올 때마다 서로 다른 서버가 응답해도 일관된 처리가 가능하다. 이는 수만 명의 사용자가 접속하는 대규모 분산 환경에서 필수적인 특성이다.
    
- HTTP의 무상태성이 주는 장점과 단점을 각각 정리해주세요. 🍠
    
    **장점:** 서버가 클라이언트의 상태를 보관하지 않으므로 서버 응답의 **가용성**이 높고, 서버 증설이 자유롭다.
    
    **단점:** 매 요청마다 필요한 모든 정보를 포함해서 보내야 하므로 **데이터 전송량**이 증가하며, 로그인과 같은 상태 유지가 필요한 기능 구현 시 추가적인 메커니즘이 요구된다.
    
- 쿠키의 Domain 디렉티브에 대해 정리해주세요. 🍠
    
    쿠키가 전송될 호스트를 명시한다.
    
    - **명시하지 않을 경우:** 현재 문서의 호스트로 기본 설정되며 서브 도메인은 포함되지 않는다.
    - **명시할 경우:** 지정한 도메인뿐만 아니라 모든 서브 도메인(예: `dev.example.com`)에서도 해당 쿠키가 전송된다.
- 쿠키의 Path 디렉티브에 대해 정리해주세요. 🍠
    
    쿠키가 전송될 특정 URL 경로를 제한한다.
    
    - 예를 들어 `Path=/docs`로 설정하면, `/docs`, `/docs/Web/` 등의 하위 경로로 요청을 보낼 때만 쿠키가 포함된다.
    - 경로가 일치하지 않는 다른 API 요청에는 쿠키가 노출되지 않으므로 최소한의 보안 범위를 설정할 때 사용된다.
- 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠
    - **세션 쿠키(Session Cookie):** `Expires`나 `Max-Age` 속성을 명시하지 않은 쿠키다. 브라우저 메모리에 저장되며, 브라우저 종료 시(세션 종료 시) 즉시 삭제된다.
    - **영속 쿠키(Persistent Cookie):** 특정 만료 날짜(`Expires`)나 유효 기간(`Max-Age`)이 설정된 쿠키다. 브라우저 종료 여부와 상관없이 하드디스크 등 물리적 저장소에 보관되며, 설정된 기간이 지나야 삭제된다.
- 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나요? 🍠
    
    쿠키의 보안 속성은 웹 취약점을 방어하기 위한 필수 설정이다.
    
    1. **HttpOnly:** 자바스크립트의 `document.cookie`를 통한 접근을 차단한다. 이를 통해 **XSS(Cross-Site Scripting)** 공격으로 인한 세션 탈취를 방어한다.
    2. **Secure:** HTTPS 프로토콜을 통해서만 쿠키를 전송하도록 제한한다. 평문 네트워크를 가로채는 중간자 공격(MITM)으로부터 쿠키 유출을 방지한다.
    3. **SameSite:** 크로스 사이트 요청 시 쿠키 전송 여부를 결정한다. 이를 통해 타 사이트에서 본인의 권한으로 악의적인 요청을 보내는 **CSRF(Cross-Site Request Forgery)** 공격을 효과적으로 방어한다. (`Strict`, `Lax`, `None` 설정 가능)
- 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)을 정리해주세요. 🍠
    - **용량 제한:** 도메인당 약 4KB 내외로 저장 용량이 매우 작다.
    - **보안 취약성:** 기본적으로 평문으로 저장되며 사용자 기기에 남기 때문에 변조 및 탈취 위험이 크다.
    - **네트워크 부하:** 설정된 범위 내의 모든 요청에 자동으로 포함되므로, 불필요한 네트워크 트래픽(오버헤드)을 발생시킨다.
    - **도메인 제약:** 동일 출처 정책(SOP)과 도메인 설정에 묶여 있어 서로 다른 도메인 간의 상태 공유가 까다롭다.
- 쿠키만으로 상태 관리를 해결할 수 없는 이유는 무엇인가요? 🍠
    
    쿠키는 순수하게 클라이언트 측 저장소에 의존하는 방식이다. 따라서 **서버 측에서의 엄격한 검증**이 불가능하며, 민감한 개인정보나 중요한 비즈니스 로직에 필요한 상태를 안전하게 보관하기에는 신뢰성이 부족하다. 이를 보완하기 위해 서버의 세션(Session)이나 토큰(JWT) 기반 인증 시스템이 결합된 형태를 사용하게 된다.


- XSS 공격은 무엇인가요?
    
    XSS는 공격자가 웹 사이트에 악성 스크립트를 삽입하여, 해당 사이트를 방문하는 사용자의 브라우저에서 스크립트가 실행되게 하는 공격 방식이다. 주된 목적은 사용자의 세션 쿠키 탈취, 개인정보 유출, 페이지 변조, 악성 사이트로의 리다이렉션 등이다.
    
    **XSS 공격의 주요 유형**
    
    | **유형** | **특징** | **공격 경로** |
    | --- | --- | --- |
    | **Stored XSS** | 악성 스크립트가 서버의 데이터베이스 등에 **영구적으로 저장**된다. | 게시판 댓글, 사용자 프로필 정보 등 |
    | **Reflected XSS** | 사용자의 요청(URL 파라미터 등)에 포함된 스크립트가 응답 페이지에 **즉시 반사**되어 실행된다. | 검색 결과 페이지, 이메일 내 악성 링크 등 |
    | **DOM-based XSS** | 서버와의 통신 없이 클라이언트 측 스크립트가 **브라우저의 DOM**을 조작하며 발생한다. | 브라우저 내 자바스크립트의 `location.hash` 처리 등 |
- XSS 방어 전략에 대해 정리해주세요.
    
    XSS 방어의 핵심은 사용자의 입력을 신뢰하지 않고, 브라우저가 이를 데이터가 아닌 실행 가능한 코드로 인식하지 못하게 하는 데 있다.
    
    ### 1) 출력 인코딩 (Output Encoding)
    
    가장 근본적인 방어책으로, 사용자가 입력한 데이터를 HTML에 출력할 때 특수 문자를 안전한 형태로 변환하는 방식이다.
    
    - `<` → `&lt;`
    - `>` → `&gt;`
    - `&` → `&amp;`
    - `"` → `&quot;`
    - `'` → `&#x27;`
    
    ### 2) 입력 검증 (Input Validation)
    
    사용자가 입력한 데이터가 기대하는 형식(예: 숫자만 입력, 특정 길이 제한 등)인지 서버 측에서 검증한다. 다만, 입력 검증만으로는 모든 XSS를 막을 수 없으므로 출력 인코딩과 병행해야 한다.
    
    ### 3) CSP (Content Security Policy) 설정
    
    웹 페이지에 로드할 수 있는 스크립트의 출처(도메인)를 제한하는 보안 정책이다.
    
    - **기능:** 허용되지 않은 외부 스크립트의 실행을 차단하고, 인라인 스크립트(HTML 태그 내 직접 작성된 스크립트) 사용을 금지한다.
    - **방법:** HTTP 응답 헤더에 `Content-Security-Policy`를 설정한다.
    
    ### 4) HttpOnly 쿠키 속성 활용
    
    쿠키 설정 시 `HttpOnly` 속성을 부여하면 자바스크립트(`document.cookie`)를 통해 쿠키에 접근할 수 없다. 이를 통해 공격자가 XSS 성공 후 사용자의 세션 토큰을 탈취하는 행위를 방어한다.
    
    ### 5) 최신 프레임워크 사용 및 데이터 정제
    
    - **React/Vue:** 최신 프레임워크는 렌더링 시 자동으로 데이터를 이스케이프(Escape) 처리하여 XSS를 방지한다. 다만, `dangerouslySetInnerHTML`과 같은 속성을 사용할 때는 각별한 주의가 필요하다.
    - **Sanitize 라이브러리:** HTML 태그 입력이 반드시 필요한 경우(예: 위지윅 에디터), 허용된 태그만 남기고 악성 태그를 제거하는 `DOMPurify` 같은 라이브러리를 사용한다.


    - 세션 하이재킹은 무엇인가요? 🍠
    
    세션 하이재킹이란 공격자가 정당한 사용자의 세션 ID(Session ID)를 탈취하여, 해당 사용자인 것처럼 시스템에 접근하는 공격 방식을 의미한다.
    
    ### 주요 공격 경로
    
    - **스니핑(Sniffing):** 암호화되지 않은 네트워크(HTTP)에서 패킷을 가로채 쿠키에 담긴 세션 ID를 추출한다.
    - **XSS(Cross-Site Scripting):** 스크립트를 삽입하여 사용자의 `document.cookie`를 공격자의 서버로 전송하게 만든다.
    - **세션 고정(Session Fixation):** 공격자가 미리 발급받은 세션 ID를 사용자에게 강제로 심어준 뒤, 사용자가 해당 ID로 로그인하면 그 세션을 가로챈다.
    - **무작위 대입(Brute Force):** 세션 ID의 규칙성을 찾아내어 유효한 ID를 추측한다.
- XSS 공격을 차단하는 방법은 무엇인가요? 🍠
    
    XSS 방어의 핵심은 사용자가 입력한 데이터를 실행 가능한 코드가 아닌 **단순 텍스트**로 처리하는 데 있다.
    
    ### 방어 방법
    
    1. **출력 인코딩 (Output Encoding):** HTML 태그로 인식될 수 있는 특수 문자(`<`, `>`, `&`, `"`, `'`)를 HTML 엔티티(Entity)로 변환하여 출력한다.
    2. **입력 검증 (Input Validation):** 허용된 형식의 데이터만 입력받도록 서버 측에서 필터링한다. (예: 이메일 형식, 숫자 전용 등)
    3. **CSP (Content Security Policy) 도입:** 브라우저에 신뢰할 수 있는 스크립트 출처를 명시하여, 허용되지 않은 인라인 스크립트나 외부 스크립트의 실행을 차단한다.
    4. **HttpOnly 쿠키 설정:** 자바스크립트를 통한 세션 쿠키 접근을 원천 차단하여 탈취 위험을 최소화한다.
- 쿠키의 다양한 옵션들에 대해서 정리해주세요. 🍠
    
    
    | **옵션명** | **설명** |
    | --- | --- |
    | **Domain** | 쿠키가 전송될 호스트를 지정한다. 명시할 경우 해당 도메인과 모든 서브 도메인에 쿠키가 전송된다. |
    | **Path** | 쿠키가 전송될 특정 URL 경로를 제한한다. 지정한 경로와 그 하위 경로로 요청 시에만 쿠키가 포함된다. |
    | **Expires** | 쿠키의 절대적인 만료 날짜와 시간을 지정한다. 해당 시간이 지나면 브라우저에서 자동 삭제된다. |
    | **Max-Age** | 쿠키의 수명을 초 단위로 지정한다. `Expires`보다 우선순위가 높으며, 유연한 만료 관리가 가능하다. |
    | **HttpOnly** | 자바스크립트(`document.cookie`)를 통한 접근을 금지한다. XSS 공격으로 인한 쿠키 탈취를 방지하는 핵심 보안 옵션이다. |
    | **Secure** | HTTPS 프로토콜을 통해서만 쿠키를 전송하도록 제한한다. 네트워크 상의 가로채기를 방지한다. |
    | **SameSite** | 크로스 사이트 간 쿠키 전송 여부를 제어하여 CSRF 공격을 방어한다. (`Strict`, `Lax`, `None` 중 선택) |


    - CSRF는 무엇인가요? 🍠
    
    CSRF는 사용자가 자신의 의지와 무관하게 공격자가 의도한 행위(수정, 삭제, 등록 등)를 특정 웹사이트에 요청하게 만드는 공격 방식이다.
    
    - **성공 조건:** 사용자가 타겟 사이트에 로그인되어 있어 브라우저에 유효한 세션 쿠키가 존재해야 한다.
    - **특징:** 공격자는 사용자의 데이터를 직접 훔치는 것이 아니라, 사용자의 권한을 도용하여 서버의 상태를 변경시킨다.
- CSRF 방어 전략에 대해 정리해주세요. 🍠
    - **CSRF 토큰(Token):** 서버가 사용자에게 임의의 난수(Token)를 발급하고, 요청 시 이 토큰이 포함되었는지 확인한다. 공격자는 타겟 사이트의 토큰 값을 알 수 없으므로 위조 요청이 불가능하다.
    - **SameSite 쿠키 속성:** 쿠키의 `SameSite` 속성을 `Lax` 또는 `Strict`로 설정하여 타 사이트에서 발생하는 요청에 쿠키가 포함되지 않도록 제한한다.
    - **Referer / Origin 검증:** 서버 측에서 요청이 들어온 도메인이 신뢰할 수 있는 도메인인지 확인한다.
    - **사용자 재인증:** 중요한 작업(비밀번호 변경, 결제 등) 수행 시 비밀번호 재입력이나 CAPTCHA를 통해 사용자의 의사를 재확인한다.
- CSRF 토큰의 장점과 단점에 대해 정리해주세요. 🍠
    
    
    | **구분** | **내용** |
    | --- | --- |
    | **장점** | 쿠키의 자동 전송 특성을 이용한 공격을 근본적으로 차단할 수 있으며, 도메인에 구애받지 않는 강력한 보안을 제공한다. |
    | **단점** | 서버 측에서 토큰을 생성 및 관리(저장)해야 하므로 추가 자원이 소모되며, Stateless한 REST API 설계 원칙에 반할 수 있다. |
- CAPTCHA는 무엇인가요? 🍠
    
    **CAPTCHA**는 'Completely Automated Public Turing test to tell Computers and Humans Apart'의 약자로, 기계는 풀기 어렵고 사람은 쉽게 풀 수 있는 문제를 제시하여 요청 주체가 인간인지 컴퓨터인지 판별하는 기술이다. 봇(Bot)에 의한 자동화된 공격(무작위 대입, 스팸 등록)을 방어하는 데 사용된다.


    - 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건은 무엇인가요? 🍠
    
    브라우저가 서버로 요청을 보낼 때 쿠키를 자동으로 포함하기 위한 조건은 다음과 같다.
    
    - 요청하려는 URL의 **도메인**과 쿠키의 `Domain` 설정이 일치해야 한다.
    - 요청 경로가 쿠키의 `Path` 설정 범위 내에 있어야 한다.
    - 쿠키가 만료(`Expires`/`Max-Age`)되지 않아야 한다.
    - `Secure` 속성 설정 시 HTTPS 환경이어야 하며, `SameSite` 속성에 따라 크로스 사이트 허용 여부가 결정된다.
- 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까요? 🍠
    - **클라이언트(Client):** `fetch`나 `axios` 요청 시 `withCredentials: true` 또는 `credentials: 'include'` 옵션을 설정해야 한다.
    - **서버(Server):**
        - 응답 헤더에 `Access-Control-Allow-Credentials: true`를 포함해야 한다.
        - `Access-Control-Allow-Origin`은 와일드카드()가 아닌 특정 도메인으로 명시되어야 한다.
- 쿠키 기반 인증에서 CSRF 공격이 발생할 수 있는 원리는 무엇인가요? 🍠
    
    브라우저의 메커니즘상 특정 도메인으로 요청을 보낼 때 해당 도메인의 쿠키를 **자동으로 첨부**하기 때문이다. 사용자가 악성 사이트 내의 버튼을 클릭하거나 이미지를 로드하는 것만으로도, 브라우저는 타겟 사이트의 인증 쿠키를 함께 실어 보내므로 서버는 이를 정당한 사용자의 요청으로 오인하게 된다.
    
- 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)을 정리해주세요. 🍠
    - **CSRF 방어:** 헤더(예: `Authorization`)는 쿠키와 달리 브라우저가 자동으로 포함하지 않는다. 자바스크립트를 통해 명시적으로 코드를 작성해야 하므로 CSRF 공격으로부터 안전하다.
    - **선택적 전송:** 모든 요청이 아닌, 필요한 요청에만 선택적으로 인증 정보를 포함할 수 있어 네트워크 오버헤드를 줄인다.
    - **CORS 단순화:** 표준 헤더를 사용할 경우 CORS 설정이 상대적으로 직관적이며, 모바일 앱 등 쿠키 관리가 까다로운 환경에서 이점을 가진다.
- 토큰을 클라이언트에 저장할 때 LocalStorage, SessionStorage, 메모리 저장의 장단점을 비교해주세요. 🍠
    
    
    | **저장소** | **장점** | **단점** |
    | --- | --- | --- |
    | **LocalStorage** | 브라우저 종료 후에도 유지되어 편리함. | **XSS 공격**에 매우 취약함 (자바스크립트로 탈취 가능). |
    | **SessionStorage** | 탭 종료 시 삭제되므로 상대적으로 안전함. | 탭을 새로 열 때마다 다시 로그인해야 하는 불편함. |
    | **메모리(변수)** | XSS로부터 가장 안전함. | 새로고침 시 정보가 소멸되어 Refresh Token 로직이 필수임. |
- Single Page Application, Mobile Application, Server Side Rendering Application 에서 여러분들이 생각하는 적합한 인증 전략은 무엇이라고 생각하시나요? 🍠
    - **Single Page Application (SPA)**
        - **전략:** **JWT + Header 방식**을 권장한다.
        - **이유:** CSRF 방어가 용이하며, 클라이언트 측 라우팅과 API 통신에 최적화되어 있다. 보안 강화를 위해 Access Token은 메모리에, Refresh Token은 `HttpOnly` 쿠키에 저장하는 하이브리드 방식이 선호된다.
    - **Mobile Application**
        - **전략:** **JWT + Header 방식** 또는 **OAuth 2.0 / OIDC**를 권장한다.
        - **이유:** 모바일 환경은 브라우저 쿠키 관리 메커니즘이 웹과 다르며, 생체 인증 연동 및 타 서비스 연동 확장성이 중요하다.
    - **Server Side Rendering Application (SSR)**
        - **전략:** **전통적인 세션-쿠키 방식**을 권장한다.
        - **이유:** 서버에서 페이지를 생성하므로 클라이언트 측 자바스크립트의 개입 없이 인증을 처리할 수 있으며, `HttpOnly`, `Secure` 쿠키를 통해 고전적이고 검증된 보안을 제공할 수 있다.

### 1. Refresh Token Rotation (RTR) - "가장 강력한 수단"

이게 요즘 인증 보안의 대세입니다. 핵심은 Refresh Token을 일회용으로 만드는 것"입니다.

- **동작 원리:** 사용자가 Refresh Token을 써서 새로운 Access Token을 받을 때, 서버는 **Refresh Token도 새것으로 갈아치웁니다.**
- **해커가 털어갔을 때:** 해커가 훔친 토큰을 먼저 써버리면, 실제 사용자가 나중에 토큰을 썼을 때 서버는 "어? 이거 이미 누가 썼던 건데? 복제 토큰이다!"라고 인지하게 됩니다.
- **결과:** 서버는 즉시 해당 유저에게 발급된 **모든 토큰(신구 불문)을 무효화** 처리합니다. 해커와 사용자 모두를 강제 로그아웃시키고 "다시 로그인해서 본인 인증해!"라고 요구하는 거죠.

---

### 2. 기기/IP 기반 검증 (Contextual Validation)

토큰만 보는 게 아니라 **"누가, 어디서"** 보냈는지를 따져보는 방식입니다.

- **동작:** 사용자가 처음 로그인했을 때의 IP 주소나 기기 정보(User-Agent, Fingerprint)를 Refresh Token과 함께 서버에 저장해둡니다.
- **대응:** 만약 서울에서 로그인했던 사용자의 토큰이 갑자기 러시아 IP에서 날아온다면? 서버는 토큰이 유효하더라도 "수상한 활동"으로 간주하고 추가 인증(2FA)을 요구하거나 차단합니다.

---

### 3. 실시간 무효화 (Token Revocation)

JWT의 단점이 "내뱉으면 끝"이라는 거였죠? 이를 보완하기 위해 서버에 '정지 명단'을 둡니다.

- **블랙리스트(Blacklist):** 사용자가 "내 폰 잃어버렸어요!"라고 신고하거나 해킹이 의심될 때, 해당 Refresh Token의 ID(jti)를 Redis 같은 고속 저장소에 등록해둡니다.
- **화이트리스트(Whitelist):** 아예 서버가 발급해준 유효한 토큰 목록을 직접 들고 있는 방식입니다. 목록에 없는 토큰은 아무리 형식이 완벽해도 문전박대합니다.

### 4. 침입 탐지 및 알림 (Kill Switch)

- **비정상 패턴 감지:** 1초에 수백 번 토큰 갱신을 요청하는 등 기계적인 움직임이 보이면 서버가 자동으로 해당 계정의 세션을 끊어버립니다.
- **사용자 알림:** "새로운 기기에서 로그인되었습니다"라는 메일을 보내서 사용자가 직접 "내가 아닌데?" 버튼을 눌러 모든 세션을 종료(Kill Switch)할 수 있게 합니다.

나의 답변: 그 토큰은 가치 없는것 이된다. 그러면 초기 로그인 화면으로 보낼 거 같다.

이유: 유효기간이 지났기때문에 누군지 알 수 없지 않을까요?


React-hook-form 학습 내용 정리
React-hook-form은 **"리액트에서 폼(Form)을 성능 최적화하여 관리하게 해주는 라이브러리"**입니다.

주요 개념
비제어 컴포넌트 방식: 기존 useState를 써서 모든 타이핑마다 리렌더링을 일으키는 방식 대신, 필요한 경우에만 렌더링을 일으켜 성능이 매우 뛰어납니다.

register: 입력 요소(input)를 라이브러리에 등록하여 값을 추적하고 유효성 규칙을 적용합니다.

handleSubmit: 모든 검증이 통과되었을 때만 최종 데이터를 처리하는 콜백 함수를 실행합니다.

watch/errors: 특정 필드의 값을 실시간으로 관찰하거나, 발생한 에러 메시지를 쉽게 가져올 수 있습니다.

왜 사용하는가?
성능: 폼이 커져도 입력 속도가 느려지지 않습니다.

간결한 코드: useState와 핸들러 함수를 여러 개 만들 필요 없이 register 하나로 해결됩니다.



Zod 학습 내용 정리
Zod는 **"TypeScript-first 스키마 선언 및 유효성 검사 라이브러리"**입니다.

주요 개념
스키마 정의: 데이터의 모양(타입)과 규칙을 미리 정의합니다.

예: z.string().email("이메일 형식이 아닙니다.")

타입 추론 (Inference): 정의한 스키마로부터 타입스크립트 타입을 자동으로 추출할 수 있어, 중복 코드를 줄여줍니다.

type User = z.infer<typeof userSchema>

유효성 검사: parse 또는 safeParse를 통해 데이터가 규칙에 맞는지 확인하고, 틀리다면 구체적인 에러 메시지를 제공합니다.

왜 사용하는가?
런타임 안전성: 타입스크립트는 컴파일 타임에만 작동하지만, Zod는 실제 프로그램이 돌아갈 때 들어오는 데이터(API 응답, 유저 입력 등)를 검증합니다.

개발자 경험(DX): 에러 메시지를 커스텀하기 쉽고, 타입스크립트와의 연동이 완벽합니다.
