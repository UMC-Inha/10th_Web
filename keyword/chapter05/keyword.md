## 📚블로그 읽고 Content Security Policy(CSP) 정리해보기 🍠
    - 서버에서 HTML 문서를 응답할 때 CSP를 적용하려면 어떤 HTTP 응답 헤더를 설정해야 하나요? 블로그에 나온 Express.js 코드 예시를 기반으로 설명해보세요.
        
        CSP 적용을 위해서는 Contents-Security-Policy라는 HTTP 응답 헤더를 설정해야한다
        
        이때 정책 적용시 헤더의 속성을 알아야하는데 
        
        블로그에 나온 코드를 예시로 설명하자면
        
        `Content-Security-Policy: default-src 'self’` 
        
        해당 코드는 모든 콘텐츠를 자체 사이트에서만 오도록 설정한것이다
        
        `Content-Security-Policy: default-src 'self' [example.com](http://example.com/) *.example.com`
        
        해당 코드는 자체 출처를 포함하고 신뢰할 수 있는 사이트를 명시하여 해당 사이트에서 오는 콘텐츠는 허용한다는 뜻이다
        
        `Content-Security-Policy: default-src 'self';
        img-src *;
        media-src [example.org](http://example.org/) [example.net](http://example.net/);
        script-src [userscripts.example.com](http://userscripts.example.com/)`
        
        해당 코드는 자체 출처를 포함하며 리소스 별로 콘텐츠 허용 출처를 명시한 구조이다
        
        - [**`img-src`](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/Content-Security-Policy/img-src) ***: 이미지는 모든 출처에서 허용
        - [**`media-src`**](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/media-src): 오디오/비디오는 **`example.org`**와 **`example.net`**에서만 허용
        - [**`script-src`**](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/Content-Security-Policy/script-src): 자바스크립트는 **`userscripts.example.com`**에서만 허용
        
    - `default-src 'self'` 설정은 브라우저에게 어떤 보안 정책을 의미하나요? 또한 `'self'` 값은 어떤 출처를 포함하거나 제외하나요?
        
        브라우저에게 현재 출처에서 제공하는 데이터만 활용하라는 정책을 의미하며 self 값은 현재 도메인과 포트, 프로토콜이 일치하는 출처에서 즉 자신의 브라우저에서 제공되는 데이터만 포함하며 외부 사이트는 물론이며 개발자 도구의 inline 스타일 수정, 사용자 입력 영역으로 부터 주입된 스크립트 데이터 제외한다 
        
    - 블로그에 나온 악성 스크립트(`<script>fetch(...)</script>`)를 주입했을 때 CSP가 어떻게 동작하는지 네트워크 탭과 콘솔 메시지 측면에서 설명해보세요.
        
        콘솔 메시지에는 Content Security Policy 경고가 표시되며 해당 정책을 위반했기에 스크립트를 차단했다는 메시지가 출력된다 
        반면 네트워크 탭에서는 CSP가 스크립트 실행을 차단했으므로 해킹 사이트로의 요청이 실행되지 않은것을 볼 수 있다
        
    - 기본 CSP 설정에서 인라인 스타일이 차단된다고 했습니다. 블로그 예시 중 `width:600px`이 적용되지 않는 이유를 설명하세요.
        
        default-src: “self”는 xss 공격 혹은 사이트 환경에서의 스타일 무결성을 지키기 위해서 인라인 스타일을 차단하고 있기 떄문에 width: 600px같은 인라인 스타일을 주입해도 작동하지 않는다
        
    - 구글 애널리틱스, 카카오맵, 외부 API 등이 CSP 때문에 차단될 수 있다고 했습니다. 이러한 현상을 "건물 보안을 강화한다"는 비유와 연결해 설명해보세요.
        
        서비스를 구상하다보면 다양한 외부 API들을 사용하게 되는데 CSP를 너무 강하게 설정하여 예를들면 자기출처만 허용한다면 API가 CSP 때문에 차단 될 수 있는데 이는 건물 보안을 강화해서 외부인 출입을 막겠다고 건물의 모든 출입구를 막아 출입해야하는 직원들도 출입하지 못하는 상황과 비슷한  상황이라고 할 수 있다
        
    - Report-Only 모드에서는 실제 리소스 실행이 차단되지 않습니다. 그 대신 브라우저와 서버에서 각각 어떤 동작을 수행하나요?
        
        브라우저에서는 실제 리소스가 실행되지만 해당 보안 정책 위반 사항이 서버에는 리포트 형식으로 전달되어 현재 사이트에서 어떤 보안 위반이 일어 났는지 모니터링 할 수 있게 한다
        
    - CSP만으로는 CSRF를 막을 수 없다고 했습니다. 블로그에 정리된 다른 보안 조치들(SameSite 쿠키, X-Frame-Options 등) 중 2가지를 설명하세요.
        
        ### 1.쿠키 보안 설정
        
        - [**`SameSite`](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/Set-Cookie/SameSite) 속성**: CSRF 공격 방어
        - [**`HttpOnly`](https://developer.mozilla.org/ko/docs/Web/HTTP/Cookies#httponly_%EC%BF%A0%ED%82%A4) 속성**: 클라이언트 측 스크립트의 쿠키 접근 차단
        - [**`Secure`](https://developer.mozilla.org/ko/docs/Web/HTTP/Cookies#secure_%EC%BF%A0%ED%82%A4) 속성**: HTTPS 연결에서만 쿠키 전송
        
        ### 2. 입력 데이터 검증
        
        - **Sanitize**: 악성 코드 제거
        - **Escape**: 특수 문자 이스케이프 처리
        - **Validation**: 데이터 형식 및 범위 검증
        
        ### 3. 추가 보안 헤더
        
        - [**`X-Frame-Options`**](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/X-Frame-Options): 클릭재킹 방어
        - [**`X-Content-Type-Options`**](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/X-Content-Type-Options): MIME 타입 스니핑 방지
        - [**`Strict-Transport-Security`**](https://developer.mozilla.org/ko/docs/Web/HTTP/Headers/Strict-Transport-Security): HTTPS 강제

## 📚 블로그 읽고 동일 출처 정책(Same Origin Policy) 정리해보기 🍠
    - 출처(Origin)는 어떤 세 요소의 조합으로 결정되나요?
        
        출처는 프로토콜(Protocol), 호스트(Host), 포트(Port)의 조합으로 결정됩니다. 이 중 하나라도 다르면 다른 출처로 간주한다
        
    - 출처의 요소가 다른 경우(예: 프로토콜만, 포트만 다른 경우)에 같은 출처인지 아닌지를 예시 3개(같은 출처 1개, 다른 출처 2개)로 설명하세요.
        
        기준:
        
        https://example.com:80
        
        1. [https://example.com:80/home](https://example.com:80/home(같은) (같은출처)
        
        프로토콜, 호스트, 포트가 모두 일치함 (경로는 상관없음)
        
        1. http://example.com:80 (다른출처)
        
        프로토콜이 다름
        
        1. https://example.com:443 (다른출처)
        
        포트가 다름
        
    - 블로그에 나온 `fetch` 기반 악성 스크립트를 다른 출처로 실행했을 때 브라우저에서 어떤 일이 발생하나요? 네트워크 전송 여부, 응답 사용 가능성, 브라우저 콘솔 메시지 측면에서 서술하세요.
        
        사용자의 브라우저에서 공격자의 서버로 `fetch` 요청을 보내는 악성 스크립트가 실행될 경우, 브라우저는 다음과 같이 반응한다.
        
        - **네트워크 전송 여부:** 요청 자체는 서버로 **전송됨**. 브라우저는 요청을 보내고 서버로부터 응답까지 받게된다.
        - **응답 사용 가능성:** 브라우저가 받은 응답 데이터에 스크립트가 **접근할 수 없다**. SOP에 의해 데이터 읽기가 차단되기 때문.
        - **브라우저 콘솔 메시지:** "Access to fetch at '...' from origin '...' has been blocked by CORS policy"와 같은 **교차 출처 리소스 공유(CORS) 관련 에러 메시지**가 출력.
    - SOP가 어떻게 Session Hijacking(세션 하이재킹) 시도를 방지하는지 구체적으로 설명하세요. SOP가 차단하는 것과 허용되는 것(예: 네트워크 요청은 나가지만 응답 데이터에 접근 불가)을 포함하세요.
        - **공격 방식:** 공격자의 사이트(`attacker.com`)에 접속한 사용자의 브라우저가 사용자의 은행 사이트(`bank.com`)로 요청을 보낼 때, 브라우저는 자동으로 쿠키를 포함해 보낼 수 있다(네트워크 요청 허용).
        - **차단 핵심:** 하지만 SOP의 데이터 읽기 차단으로 공격자의 스크립트는 은행 서버에서 돌아온 **응답 데이터(개인정보, 잔액 등)를 읽을 수 없게된다**.
        - **방지 효과:** 응답 내용을 볼 수 없으므로, 공격자는 세션 ID를 탈취하거나 사용자의 계정 정보를 훔쳐보는 하이재킹 시도를 성공시킬 수 없다.
    - 블로그에서 명시한 대로 SOP가 반드시 동일 출처에서만 접근하도록 하는 주요 브라우저 API/리소스 3가지를 쓰고, 각각에 대해 간단한 설명(왜 제한되는지)을 덧붙이세요.
        - **XMLHttpRequest / fetch API:** 다른 출처의 데이터를 읽어오는 것을 제한합니다. 민감한 API 응답 데이터 유출을 막기 위함.
        - **DOM (Document Object Model):** `<iframe>` 등을 통해 로드된 다른 출처 페이지의 DOM 구조에 접근하거나 조작하는 것을 막음. (예: 로그인창의 입력값 탈취 방지)
        - **Cookie / LocalStorage:** 특정 출처에서 생성한 저장소 데이터를 다른 출처의 스크립트가 읽거나 수정하지 못하게 하여 세션 정보를 보호.
    - SOP와 CSP의 차이를 블로그 내용에 따라 요점 4개(각 항목 1문장)로 정리하세요. (예: 누가 적용하는가, 제어 주체, 설정 가능 여부 등)
        - **적용 대상:** SOP는 브라우저에 내장된 기본 보안 메커니즘이며, CSP는 웹 서버가 브라우저에게 전달하는 보안 지침.
        - **제어 주체:** SOP는 브라우저가 주체가 되어 다른 출처와의 상호작용을 제어하고, CSP는 사이트 관리자가 허용할 자원의 목록을 직접 제어.
        - **설정 유연성:** SOP는 사용자가 임의로 끄거나 켤 수 없는 고정된 규칙인 반면, CSP는 HTTP 헤더를 통해 정책을 세밀하게 설정가능.
        - **방어 목적:** SOP는 주로 다른 출처 간의 데이터 유출을 방지하며, CSP는 신뢰할 수 없는 스크립트 실행(XSS) 자체를 차단하는 데 중점.
    - 브라우저에서 SOP 관련 차단 오류를 발견했을 때(예: 콘솔에 “동일 출처 정책으로 인해 ... 차단했습니다” 메시지) 문제 원인 파악을 위한 체크리스트(최소 3항목)를 작성하고, 임시·영구 대응 방안(각 1~2줄)도 제시하세요.
        
        체크리스트
        
        - **프로토콜/호스트/포트 일치 여부:** 요청을 보내는 곳과 받는 곳의 출처 3요소가 정확히 일치하는가?
        - **CORS 헤더 설정 여부:** 서버 응답 헤더에 `Access-Control-Allow-Origin`이 적절히 포함되어 있는가?
        - **인증 정보(Credentials) 포함 여부:** 쿠키를 포함한 요청일 경우 서버에서 `Access-Control-Allow-Credentials: true`를 설정했는가?
        
        대응방안
        
        - **임시 방안:** 개발 단계에서만 브라우저의 보안 기능을 끄고 실행하거나(크롬 실행 옵션 변경), 로컬 프록시 서버를 사용하여 같은 출처인 것처럼 만들어 테스트.
        - **영구 방안:** 대상 서버(API 서버)에서 **CORS(Cross-Origin Resource Sharing) 헤더**를 올바르게 설정하여 특정 도메인에 대해서 명시적으로 접근을 허용. 

## 📚 블로그 읽고 교차 출처 리소스 공유(CORS) 정리해보기 🍠
    - 브라우저에서 `http://localhost:8080` 애플리케이션이 `http://localhost:8081/resource.json`을 요청했을 때, 네트워크 요청과 응답은 어떻게 처리되며, 브라우저가 응답 본문을 사용하지 못하는 이유는 무엇인가요?
        - **네트워크 처리:** 브라우저는 출처가 다르더라도 네트워크 요청을 **정상적으로 서버에 전송**하며, 서버 또한 요청을 처리하고 **응답(데이터)을 브라우저에 보냄.**
        - **사용 불가 이유:** 브라우저는 응답을 받은 후 SOP를 확인하게됨. 두 주소는 포트 번호가 다르기 때문에 다른 출처로 판별되며 서버 응답에 다른 출처의 접근을 허용 보안 설정(CORS)이 없다면, 브라우저는 보안상의 이유로 응답 본문을 스크립트가 읽지 못하도록 **차단함**.
    - 서버가 다른 출처(`http://localhost:8080`)에서 자원을 사용할 수 있게 하려면 어떤 응답 헤더를 어떻게 설정해야 하나요? 글의 예시 코드를 참고해 헤더 이름과 값까지 구체적으로 쓰세요.
        - **헤더 이름:** `Access-Control-Allow-Origin`
        - **값:** `http://localhost:8080` (또는 모든 출처를 허용할 경우 `*`)
    - 단순 요청으로 분류되기 위해서는 어떤 두 가지 조건을 만족해야 하나요? 또한 `GET /resource.json` 요청이 단순 요청에 해당하는 이유를 설명하세요.
        1. **메서드 제한:** `GET`, `HEAD`, `POST` 중 하나. 
        2. **헤더 제한:** `Accept`, `Accept-Language`, `Content-Language`, `Content-Type`(특정 값만 허용) 등 표준 헤더만 사용해야 함. 
        - **GET /resource.json이 단순 요청인 이유:** 위 조건 중 `GET` 메서드를 사용하며, 별도의 커스텀 헤더 없이 표준 요청을 보내기 때문에 사전 검사 단계 없이 즉시 전송될 수 있는 단순 요청으로 분류됨.
    - 브라우저에서 `X-Goguma`라는 커스텀 헤더를 추가했을 때 왜 차단이 발생하나요? 이 문제를 해결하기 위해 서버에서 추가해야 하는 응답 헤더와 값은 무엇인가요?
        - **차단 이유:** `X-Goguma`와 같은 커스텀 헤더는 브라우저가 정한 **표준 헤더가 아님.** 브라우저는 보안을 위해 표준이 아닌 헤더가 포함된 요청을 보낼 때, 서버가 해당 헤더를 수용할 준비가 되었는지 먼저 확인하려 하지만 서버 응답에 허용 설정이 없으면 차단됨.
        - **서버 해결 방법:** 응답 헤더에 해당 커스텀 헤더를 허용한다는 설정을 추가해야 함.
            - **헤더:** `Access-Control-Allow-Headers`
            - **값:** `X-Goguma`
    - `PUT` 요청을 보낼 때 브라우저는 왜 먼저 `OPTIONS` 요청을 보내나요? 이때 브라우저가 보내는 헤더와 서버가 응답해야 하는 헤더를 각각 쓰고, 사전 요청과 실제 요청이 어떻게 이어지는지 간단히 서술하세요.
        
        `PUT` 메서드는 데이터 수정 등 서버에 영향을 줄 수 있는 위험이 있어 브라우저가 먼저 사전 요청(OPTIONS)을 보냄. 
        
        - **브라우저가 보내는 사전 요청 헤더:**
            - `Access-Control-Request-Method: PUT` (실제 보낼 메서드 확인)
            - `Origin: http://localhost:8080` (요청 출처 알림)
        - **서버가 응답해야 하는 헤더:**
            - `Access-Control-Allow-Methods: PUT` (해당 메서드 허용 알림)
            - `Access-Control-Allow-Origin: http://localhost:8080`
        - **진행 과정:** 브라우저가 `OPTIONS` 요청을 먼저 보내 서버의 허용 여부를 확인(사전 요청)한 뒤, 서버로부터 "허용한다"는 응답을 받으면 실제 `PUT` 요청 전송.

##  📚블로그 읽고 ABAC 정리해보기 🍠
    - **RBAC**의 한계에 대해 설명해주세요.
        
        RBAC은 관리가 편하지만, 현대의 복잡한 비즈니스 로직 에서는 경직성이 있음.
        
        - **역할 폭발:** 세밀한 권한 제어가 필요할수록 "A팀_낮시간_서울지역_열람권한" 같은 기괴하고 파편화된 역할이 무한 생성.
        - **상황 인지 불능:** "근무 시간 외 접근 차단"이나 "특정 IP 대역 외 접속 제한" 같은 **동적인 조건**을 역할만으로는 처리 불가.
        - **데이터 수준의 통제 취약:** '매니저' 역할이라도 '자기가 담당하는 팀원'의 정보만 보게 하려면 RBAC 내부에서 추가적인 코드 로직이 필요해 시스템이 지저분해 짐.
    - **ABAC**으로의 전환, 어떤 '기준'이 적절할까요?
        
        ABAC(속성 기반 제어)으로의 전환은 
        
        - **다차원적 조건(Multidimensionality):** 권한 판단에 누구 뿐만 아니라 언제, 어디서, 어떤 기기로 등 3개 이상의 변수가 복합적으로 작용할 때.
        - **비즈니스 규제 대응:** 금융이나 의료 데이터처럼 법적으로 "특정 상황"에서의 접근을 엄격히 규제해야 하는 도메인일 때.
        - **확장성(Scalability):** 사용자와 리소스의 숫자가 기하급수적으로 늘어나, 일일이 역할을 매핑하는 것이  큰 운영 비용을 발생시킬 때.
    - 어떤 서비스 영역에 **RBAC**을 남겨두고, **ABAC**을 도입하시겠어요?
        - **RBAC 유지 영역 (정적/내부 시스템):**
            - **사내 ERP 기초 메뉴:** 인사, 총무 등 직무가 명확히 나뉘고 변동성이 적은 영역.
            - **인프라 관리 도구:** 'DB 관리자', '네트워크 관리자' 등 전문 기술 영역은 역할 기반이 여전히 직관적이고 안전합니다.
        - **ABAC 도입 영역 (동적/민감 시스템):**
            - **고객 개인정보 열람:** "조회자의 직급 + 조회 대상과의 관계 + 현재 접속 위치 + 업무 시간 여부"를 체크해야 하는 영역.
            - **콘텐츠/자산 관리:** 유료 멤버십 등급, 구매 이력, 지역 제한 등 속성에 따라 권한이 실시간으로 변하는 서비스 최전방.
    - 여러분들은 다른 부서에서 요청을 받았을 때 어떤식으로 행동하실껀가요?
        
        다른 부서에서 요청을 받았을때 해당 요청이 단순 역할로 정의할 수 있는 영역이라면 RBAC를 고려하여 개발을 진행 할 것 같고 
        
        사용자들과 직접 연결되는 콘텐츠관련 요청은 ABAC를 고려하여 개발을 진행 할 것 같다
