# RBAC과 ABAC

## 왜 나누어 보나

- 시스템과 데이터에 **누가**, **무엇을** 할 수 있는지 결정하는 방식을 **접근 제어(Access Control) 모델**이라 함
- **RBAC**은 사용자의 **역할(Role)** 이라는 신분증을 보고 권한을 결정하고, **ABAC**은 사용자·리소스·환경 등 **다양한 속성(Attribute)** 을 조합하여 동적으로 권한을 결정함
- 두 모델은 상호 배타적이지 않으며, 서비스 복잡도에 따라 혼합 적용하는 것이 일반적임

공식: [NIST SP 800-162 — Guide to Attribute Based Access Control (ABAC) Definition and Considerations](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-162.pdf)  
공식: [NIST SP 800-207 — Zero Trust Architecture](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-207.pdf)

---

## RBAC (Role-Based Access Control) — 역할 기반 접근 제어

### 개요

사용자의 **역할(Role)** 을 기준으로 권한을 할당하는 방식. 가장 널리 사용되며, 관리 효율이 높은 정적 모델임.

NIST에서 RBAC의 표준 모델을 정의함 — [NIST SP 800-207 Section 2.4](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-207.pdf), [ANSI INCITS 359-2004 — RBAC](https://profsandhu.com/journals/tissec/ANSI+INCITS+359-2004.pdf)

### 핵심 구성 요소

| 용어 | 설명 | 예시 |
| --- | --- | --- |
| **사용자 (User)** | 시스템을 이용하는 사람 또는 계정 | 개발팀 김철수 사원 |
| **역할 (Role)** | 직무·기능에 따라 권한들을 묶어 놓은 집합 | `재무팀 관리자`, `일반 사용자`, `서버 엔지니어` |
| **권한 (Permission)** | 리소스에 대해 구체적으로 허용된 작업 | `파일 읽기`, `사용자 추가`, `결제 승인` |

### 작동 방식

1. **권한을 역할에 할당** — `재무팀 관리자` 역할에 `결제 승인` 권한을 부여함
2. **사용자에게 역할을 부여** — 김철수 사원에게 `재무팀 관리자` 역할을 할당함
3. **결과** — 김철수 사원은 해당 역할에 묶인 모든 권한을 자동으로 갖게 됨

### 장점

- **관리 효율성** — 수백 명의 사용자에게 일일이 권한을 부여할 필요 없이 역할만 할당하면 됨. 확장성이 좋음
- **단순함** — 구조가 명확하여 대부분의 조직 환경에 적용하기 쉬움

### 단점과 한계

1. **유연성 부족** — "원장이면 무조건 삭제 가능"과 같은 단순한 규칙만 적용할 수 있음. "특정 학원의 삭제 권한을 가진 원장만" 또는 "본인이 작성한 글만"처럼 세부 예외 조건이 생기면 대응하기 어려움
2. **역할 폭발 (Role Explosion)** — 예외 상황을 해결하기 위해 역할을 무한정 늘리다 보면 관리가 불가능해짐. `삭제 가능한 원장`, `조회만 가능한 원장` 등의 역할이 폭발적으로 증가함
3. **코드 복잡도 증가** — UI 코드 곳곳에 `if (role === 'ADMIN' || (role === 'PRINCIPAL' && ...))`와 같은 복잡한 조건문이 흩어져 유지보수가 어려워짐

## ABAC (Attribute-Based Access Control) — 속성 기반 접근 제어

### 개요

**속성(Attribute)** 과 **정책(Policy)** 을 기반으로 권한을 **동적으로** 결정하는 방식. RBAC보다 훨씬 세밀하고 유연한 통제가 필요할 때 사용함.

공식: [NIST SP 800-162 — Guide to Attribute Based Access Control (ABAC)](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-162.pdf)

### 핵심 구성 요소 — 네 가지 속성 유형

| 속성 유형 | 설명 | 예시 |
| --- | --- | --- |
| **주체(사용자) 속성** | 접근을 요청하는 사용자의 특성 | 부서: `마케팅팀`, 직위: `팀장`, 보안등급: `1등급` |
| **객체(리소스) 속성** | 접근 대상(파일, 데이터)의 특성 | 민감도: `극비`, 소유자: `John`, 위치: `US-East` |
| **작업 속성** | 사용자가 수행하려는 행동 | `읽기(Read)`, `쓰기(Write)`, `다운로드(Download)` |
| **환경 속성** | 접근이 요청되는 외부 상황(컨텍스트) | 시간: `09:00~18:00`, 네트워크: `내부망`, 장치: `회사 지급 노트북` |

### 작동 방식

1. **정책(Policy)을 정의** — 속성들을 조합한 "만약/그러면" 형태의 규칙을 작성함
2. **정책 예시:**
    > "사용자 속성(부서=재무팀) **이고**, 리소스 속성(민감도=높음) **이며**, 환경 속성(시간=업무 시간)일 **때만**, 작업 속성(읽기)을 **허용**한다."
3. **결과** — 접근 요청이 들어올 때마다 해당 시점의 속성을 실시간으로 평가하여 권한을 **동적으로** 결정함

### 장점

- **세분화된 제어** — "재무팀 김철수 사원이 평일 오전 9시~오후 6시에만 극비 파일을 읽을 수 있다"와 같이 매우 정교한 제어가 가능함
- **동적(Dynamic) 모델** — 시간, 위치 등 상황이 바뀌면 실시간으로 접근 권한도 바뀜
- **유연성** — 새로운 리소스나 사용자가 추가되어도 정책의 속성만 잘 정의하면 기존 정책을 그대로 활용할 수 있음

### 단점

- **정책 정의와 관리가 복잡** — 속성과 조건 조합이 많아질수록 정책을 설계·관리하는 난이도가 높아짐
- **초기 구축 비용** — RBAC보다 설계·구현에 더 많은 시간과 비용이 필요함

## RBAC vs ABAC 비교

| 구분 | RBAC (역할 기반) | ABAC (속성 기반) |
| --- | --- | --- |
| **핵심 요소** | 역할 (Role) | 속성 (Attribute) + 정책 (Policy) |
| **접근 결정** | 할당된 역할만 보고 결정 | 다양한 속성과 조건을 모두 따져 동적으로 결정 |
| **유연성** | 상대적으로 낮음 | 상대적으로 매우 높음 |
| **복잡성** | 단순하고 관리하기 쉬움 | 정책 정의와 관리가 복잡할 수 있음 |
| **적합한 상황** | 권한 구조가 고정적이고 명확한 일반 기업 시스템 | 클라우드 환경, 민감한 데이터 제어처럼 세밀하고 동적인 제어가 필요할 때 |

## RBAC의 한계와 ABAC 도입 기준

### RBAC의 한계

서비스가 성장하면서 단순 역할 기반 권한 모델이 다음 한계에 부딪히게 됨:

- **예외 조건 처리 불가** — "본인이 쓴 글만 삭제 가능", "소속 학원의 데이터만 열람 가능"처럼 리소스와 사용자 간의 **관계**를 따져야 하는 조건은 역할만으로 표현할 수 없음
- **역할 폭발** — 예외를 역할로 해결하려 하면 역할 수가 폭발적으로 증가하여 관리가 불가능해짐
- **코드 분산** — 권한 로직이 UI 여기저기에 흩어져 변경 사항이 생길 때 모든 곳을 찾아 수정해야 함

### ABAC 도입이 적절한 기준

아래 중 하나라도 해당하면 ABAC 도입을 검토할 수 있음:

1. **리소스 속성이 권한 판단에 필요한 경우** — 리소스의 소유자(`ownerId`), 소속 그룹(`academyId`) 등 대상의 정보가 접근 여부 결정에 필요할 때
2. **사용자의 세부 속성이 필요한 경우** — 단순 역할 외에 차단 여부, `canDeleteChallenge` 같은 **세부 플래그(Flag)** 를 확인해야 할 때
3. **동적인 관계 조건이 필요한 경우** — "본인이 작성한 글인가?", "자신이 소속된 학원의 데이터인가?"처럼 사용자와 리소스 간의 **관계**를 따져야 할 때

### 혼합 적용 전략

모든 권한을 ABAC으로 전환할 필요는 없음. 영역별로 나누어 적용하는 것이 효율적임:

- **RBAC 유지 영역** — 시스템 전체 관리자(`ADMIN`)처럼 역할만으로 충분히 설명 가능한 전역 권한
- **ABAC 도입 영역** — 학원별 권한, 챌린지 삭제처럼 리소스 소유 관계나 세부 플래그가 필요한 복잡한 비즈니스 로직 영역

## 실무 설계 원칙

### 중앙 집중식 권한 관리

권한 로직이 UI 코드 곳곳에 흩어지지 않도록 `hasPermission`과 같은 **공통 유틸 함수**를 두고, 권한 정책 테이블(`APP_ROLES`, `ACADEMY_ROLES`)만 수정하는 방식으로 관리함.

```ts
// 분산된 조건문 대신
// if (role === 'ADMIN' || (role === 'PRINCIPAL' && canDelete)) { ... }

// 중앙 집중식 권한 함수 사용
if (hasPermission(user, 'challenge:delete', challenge)) { ... }
```

- 새로운 권한 요청이 들어올 때마다 UI 코드를 수정하는 대신, 정책 테이블만 변경하면 됨
- 기획자에게 테스트 코드를 통해 "이러한 케이스들이 정상적으로 작동한다"는 것을 명확히 전달할 수 있음

### 미래 확장성 고려

권한 요청이 들어왔을 때 "나중에 이 역할 외에 다른 사람이 이 작업을 할 일이 생길까?"를 미리 물어보며 기획 의도를 파악하는 것이 중요함. 처음에 단순해 보이는 요구사항도 확장 가능한 구조로 설계해두면 추후 역할 폭발을 방지할 수 있음.

## 최소 권한 원칙 (Principle of Least Privilege)

RBAC과 ABAC 모두 **최소 권한 원칙**과 함께 설계되어야 함.

> 사용자에게는 업무 수행에 필요한 최소한의 권한만 부여해야 함.

공식: [NIST SP 800-207 Section 2.4 — Zero Trust Architecture (Least Privilege)](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-207.pdf)

- 과도한 권한 부여는 내부자 위협이나 계정 탈취 시 피해를 확대시킴
- ABAC은 상황 기반으로 권한을 동적으로 좁힐 수 있어 최소 권한 원칙을 구현하기에 더 적합한 모델임
